有区别，而且区别挺大。

先一句话概括：

> **普通 socket 是传输层接口，ZeroMQ 是建立在 socket 之上的消息通信框架。**

也就是说，`socket` 更像“原始零件”，`ZMQ` 更像“已经封装好通信模式的工具箱”。

---

# 一、和普通 socket 的区别

## 1. 普通 socket 更底层

比如 Linux socket 编程，你通常要自己处理这些事：

* `socket()`
* `bind()`
* `listen()`
* `accept()`
* `connect()`
* `send()/recv()`
* 粘包/拆包
* 重连
* 多连接管理
* 心跳
* 超时
* 多路复用 `select/poll/epoll`

例如 TCP server，光“接收多个客户端”就要自己维护一堆连接。

---

## 2. ZeroMQ 更高层

ZeroMQ 帮你把很多“通信模式”直接封装了，比如：

* 请求/响应
* 发布/订阅
* 推送/拉取
* 异步路由

你只需要关心：

* 我是什么角色
* 我想发什么消息
* 对端怎么收

例如：

```cpp
zmq::socket_t s(ctx, zmq::socket_type::pub);
s.bind("tcp://*:5555");
s.send(zmq::buffer("hello"));
```

你不用手写 accept 循环，也不用自己设计消息分发机制。

---

## 3. socket 传的是字节流，ZMQ 传的是“消息”

普通 TCP 的核心特点之一是：

> **TCP 是字节流，不保消息边界**

你发两次：

```cpp
send(fd, "abc", 3, 0);
send(fd, "def", 3, 0);
```

对端可能一次 `recv()` 收到：

```text
abcdef
```

也可能分两次收到：

```text
abc
def
```

也可能：

```text
ab
cdef
```

所以你必须自己定义协议，比如：

* 固定长度
* 长度字段 + payload
* 分隔符

而 ZMQ 的发送接收是“消息”语义：

```cpp
socket.send(zmq::buffer("abc"));
socket.send(zmq::buffer("def"));
```

接收端会按两条独立消息来收，不会把两条消息粘在一起。

这个非常重要。

---

## 4. socket 关注连接，ZMQ 关注模式

普通 socket 你经常在想：

* 谁监听
* 谁 accept
* 哪个 fd 对应哪个 client
* 连接断了怎么办

ZMQ 更像在想：

* 我是 `REQ` 还是 `REP`
* 我是 `PUB` 还是 `SUB`
* 我是 `PUSH` 还是 `PULL`

它的抽象层次更高。

---

## 5. ZMQ 自带很多工程化能力

ZeroMQ 内置或半内置支持：

* 自动重连
* 异步队列
* 多 peer 管理
* 负载均衡
* 简单的多播式分发
* 多 transport 支持

支持的传输方式常见有：

* `tcp://`
* `ipc://`
* `inproc://`

其中：

* `tcp`：跨机器/跨进程
* `ipc`：同机进程间
* `inproc`：同进程线程间

普通 socket 基本没有这么统一的抽象。

---

# 二、什么时候用 socket，什么时候用 ZMQ

## 更适合普通 socket 的场景

* 你要完全控制协议细节
* 你做的是标准 TCP/UDP 服务
* 你要兼容已有网络协议
* 你对性能路径要求极致且愿意自己维护复杂度
* 你做内核/网络协议/系统级开发

## 更适合 ZMQ 的场景

* 模块之间通信
* 分布式组件解耦
* 高性能消息传递
* 快速实现请求/响应或发布/订阅
* 不想自己处理粘包拆包和连接管理

---

# 三、ZMQ 的核心概念

先把最重要的几个概念说清楚。

## 1. context

`context` 可以理解成 ZMQ 运行时环境。

```cpp
zmq::context_t ctx(1);
```

一般一个进程里创建一个全局 context 就够了。

它不是“连接”，更像是整个 ZMQ 世界的根对象。

---

## 2. socket

ZMQ 的 socket 不是传统 OS socket 的等价物，它是更高层的“消息端点”。

```cpp
zmq::socket_t sock(ctx, zmq::socket_type::req);
```

注意这个类型很关键，决定通信模式。

---

## 3. bind 和 connect

### bind

表示“我在某个地址上提供服务/等待别人连我”。

```cpp
sock.bind("tcp://*:5555");
```

### connect

表示“我主动去连接某个地址”。

```cpp
sock.connect("tcp://127.0.0.1:5555");
```

一般不是死规则，但常见习惯是：

* 服务端：`bind`
* 客户端：`connect`

---

## 4. message

ZMQ 按“消息”传递。

常见写法：

```cpp
socket.send(zmq::buffer("hello"), zmq::send_flags::none);
```

接收：

```cpp
zmq::message_t msg;
socket.recv(msg, zmq::recv_flags::none);
```

取字符串：

```cpp
std::string s = msg.to_string();
```

---

# 四、最常用的四种模式

---

## 模式 1：REQ / REP

这是最容易理解的，请求-响应模式。

### 特点

* `REQ` 发送请求
* `REP` 接收请求并回复
* 一问一答，顺序严格

### REQ 端

```cpp
#include <cppzmq/zmq.hpp>
#include <iostream>

int main() {
    zmq::context_t ctx(1);
    zmq::socket_t client(ctx, zmq::socket_type::req);

    client.connect("tcp://127.0.0.1:5555");

    client.send(zmq::buffer("hello"), zmq::send_flags::none);

    zmq::message_t reply;
    auto ret = client.recv(reply, zmq::recv_flags::none);
    if (ret) {
        std::cout << "reply: " << reply.to_string() << '\n';
    }
}
```

### REP 端

```cpp
#include <cppzmq/zmq.hpp>
#include <iostream>

int main() {
    zmq::context_t ctx(1);
    zmq::socket_t server(ctx, zmq::socket_type::rep);

    server.bind("tcp://*:5555");

    while (true) {
        zmq::message_t req;
        auto ret = server.recv(req, zmq::recv_flags::none);
        if (!ret) {
            continue;
        }

        std::cout << "recv: " << req.to_string() << '\n';
        server.send(zmq::buffer("world"), zmq::send_flags::none);
    }
}
```

### 注意事项

`REQ/REP` 最大的坑是：

> **发送和接收顺序必须匹配**

`REQ` 必须：

```text
send -> recv -> send -> recv
```

`REP` 必须：

```text
recv -> send -> recv -> send
```

你不能在 `REQ` 上连续 `send` 两次，也不能在 `REP` 上连续 `recv` 两次。

否则就会卡住或报错。

所以它适合简单同步问答，不适合复杂异步系统。

---

## 模式 2：PUB / SUB

发布订阅模式。

### 特点

* `PUB` 广播消息
* `SUB` 订阅自己关心的主题
* 一个发布者可以对应多个订阅者

### 发布端

```cpp
#include <cppzmq/zmq.hpp>
#include <chrono>
#include <thread>
#include <string>

int main() {
    zmq::context_t ctx(1);
    zmq::socket_t pub(ctx, zmq::socket_type::pub);

    pub.bind("tcp://*:5556");

    int i = 0;
    while (true) {
        std::string topic = "weather";
        std::string data = topic + " sunny " + std::to_string(i++);
        pub.send(zmq::buffer(data), zmq::send_flags::none);
        std::this_thread::sleep_for(std::chrono::seconds(1));
    }
}
```

### 订阅端

```cpp
#include <cppzmq/zmq.hpp>
#include <iostream>

int main() {
    zmq::context_t ctx(1);
    zmq::socket_t sub(ctx, zmq::socket_type::sub);

    sub.connect("tcp://127.0.0.1:5556");

    sub.set(zmq::sockopt::subscribe, "weather");

    while (true) {
        zmq::message_t msg;
        auto ret = sub.recv(msg, zmq::recv_flags::none);
        if (ret) {
            std::cout << msg.to_string() << '\n';
        }
    }
}
```

### 关键点

#### 订阅过滤

```cpp
sub.set(zmq::sockopt::subscribe, "weather");
```

表示只收以 `"weather"` 开头的消息。

订阅所有主题：

```cpp
sub.set(zmq::sockopt::subscribe, "");
```

#### 慢加入者问题

PUB/SUB 有个经典现象：

> 订阅者刚连上时，前几条消息可能收不到

因为订阅关系建立需要一点时间。
所以测试时常见做法是：publisher 启动后先 sleep 一下，或者 subscriber 先启动。

#### PUB 不关心谁收到了

PUB 更像广播，不提供强一致交付语义。
如果订阅者没连上、太慢、断开，消息可能就丢了。

---

## 模式 3：PUSH / PULL

用于任务分发、流水线处理。

### 特点

* `PUSH` 把任务推给下游
* `PULL` 从上游拉任务
* 多个 PULL 存在时，通常是负载均衡分发

### PUSH

```cpp
#include <cppzmq/zmq.hpp>
#include <string>

int main() {
    zmq::context_t ctx(1);
    zmq::socket_t push(ctx, zmq::socket_type::push);

    push.bind("tcp://*:5557");

    for (int i = 0; i < 10; ++i) {
        std::string task = "task-" + std::to_string(i);
        push.send(zmq::buffer(task), zmq::send_flags::none);
    }
}
```

### PULL

```cpp
#include <cppzmq/zmq.hpp>
#include <iostream>

int main() {
    zmq::context_t ctx(1);
    zmq::socket_t pull(ctx, zmq::socket_type::pull);

    pull.connect("tcp://127.0.0.1:5557");

    while (true) {
        zmq::message_t msg;
        auto ret = pull.recv(msg, zmq::recv_flags::none);
        if (ret) {
            std::cout << "got: " << msg.to_string() << '\n';
        }
    }
}
```

如果你同时起多个 `PULL` worker，任务会分散到不同 worker，不是每个 worker 都收到全部消息。

所以：

* `PUB/SUB` 像广播
* `PUSH/PULL` 像分发队列

---

## 模式 4：DEALER / ROUTER

这是更高级、更灵活的异步通信模式。

可以把它理解成：

* `REQ/REP` 的增强版
* 支持异步、多客户端、路由控制

它适合复杂服务端，但入门先理解：

* `DEALER`：更自由的客户端端点
* `ROUTER`：可以识别不同客户端身份并路由消息

如果你现在只是学习，先掌握前三种，`ROUTER/DEALER` 后面再学更合适。

---

# 五、ZMQ 常见用法细节

## 1. 发送字符串

```cpp
std::string s = "hello";
socket.send(zmq::buffer(s), zmq::send_flags::none);
```

`zmq::buffer` 很常用，它把字符串或内存包装成可发送缓冲区。

---

## 2. 接收消息

```cpp
zmq::message_t msg;
auto ret = socket.recv(msg, zmq::recv_flags::none);
if (ret) {
    std::string data = msg.to_string();
}
```

---

## 3. 非阻塞接收

```cpp
auto ret = socket.recv(msg, zmq::recv_flags::dontwait);
if (!ret) {
    // 当前没有消息
}
```

这类似普通 socket 的非阻塞读取。

---

## 4. 设置超时

```cpp
socket.set(zmq::sockopt::rcvtimeo, 3000); // 3秒
socket.set(zmq::sockopt::sndtimeo, 3000);
```

这样 `recv/send` 不会无限阻塞。

---

## 5. 设置高水位线 HWM

高水位线用于控制队列缓存大小。

```cpp
socket.set(zmq::sockopt::rcvhwm, 1000);
socket.set(zmq::sockopt::sndhwm, 1000);
```

当生产速度远大于消费速度时，这个配置很重要。
否则可能：

* 内存涨
* 消息堆积
* 超出预期延迟

---

## 6. linger

关闭 socket 时，是否等待未发送消息发完。

```cpp
socket.set(zmq::sockopt::linger, 0);
```

常见用法是程序退出时设成 0，避免卡在关闭阶段。

---

# 六、多部分消息 multipart

ZMQ 支持一条消息拆成多帧发送。

这很适合：

* topic + payload
* header + body
* metadata + binary data

### 发送两帧

```cpp
socket.send(zmq::buffer("topic1"), zmq::send_flags::sndmore);
socket.send(zmq::buffer("hello"), zmq::send_flags::none);
```

第一帧加 `sndmore`，表示后面还有帧。

### 接收两帧

```cpp
zmq::message_t part1, part2;
socket.recv(part1);
socket.recv(part2);
```

例如 PUB/SUB 很多时候会这么设计：

* 第一帧：topic
* 第二帧：真实数据

这样比自己拼字符串更清晰。

---

# 七、线程和 ZMQ 的关系

这是新手特别容易踩的坑。

## 1. context 可以跨线程共享

一般一个进程一个 context。

## 2. socket 不是线程安全共享对象

通常原则是：

> **一个 socket 只给一个线程用**

不要多个线程同时对同一个 `zmq::socket_t` 做 `send/recv`。

如果多线程通信，推荐：

* 每个线程自己的 socket
* 共享一个 context
* 线程间用 `inproc://` 通信

---

# 八、三种常见 transport

## 1. tcp://

跨主机或同机网络通信。

```cpp
tcp://127.0.0.1:5555
tcp://*:5555
```

---

## 2. ipc://

同一台机器不同进程之间。

```cpp
ipc:///tmp/my_socket
```

Linux 下很好用。

---

## 3. inproc://

同一进程不同线程之间。

```cpp
inproc://worker1
```

性能通常很好，但要求共享同一个 context。

# 十、一个更完整的 REQ/REP 示例

服务端：

```cpp
#include <cppzmq/zmq.hpp>
#include <iostream>
#include <string>

int main() {
    try {
        zmq::context_t ctx(1);
        zmq::socket_t server(ctx, zmq::socket_type::rep);

        server.set(zmq::sockopt::rcvtimeo, 5000);
        server.set(zmq::sockopt::sndtimeo, 5000);
        server.set(zmq::sockopt::linger, 0);

        server.bind("tcp://*:5555");
        std::cout << "server started at tcp://*:5555\n";

        while (true) {
            zmq::message_t req;
            auto ret = server.recv(req, zmq::recv_flags::none);
            if (!ret) {
                std::cout << "recv timeout\n";
                continue;
            }

            std::string input = req.to_string();
            std::cout << "recv: " << input << '\n';

            std::string output = "ack: " + input;
            server.send(zmq::buffer(output), zmq::send_flags::none);
        }
    } catch (const zmq::error_t& e) {
        std::cerr << "zmq error: " << e.what() << '\n';
        return 1;
    } catch (const std::exception& e) {
        std::cerr << "std error: " << e.what() << '\n';
        return 1;
    }
}
```

客户端：

```cpp
#include <cppzmq/zmq.hpp>
#include <iostream>
#include <string>

int main() {
    try {
        zmq::context_t ctx(1);
        zmq::socket_t client(ctx, zmq::socket_type::req);

        client.set(zmq::sockopt::rcvtimeo, 3000);
        client.set(zmq::sockopt::sndtimeo, 3000);
        client.set(zmq::sockopt::linger, 0);

        client.connect("tcp://127.0.0.1:5555");

        for (int i = 0; i < 3; ++i) {
            std::string msg = "hello-" + std::to_string(i);
            client.send(zmq::buffer(msg), zmq::send_flags::none);

            zmq::message_t reply;
            auto ret = client.recv(reply, zmq::recv_flags::none);
            if (!ret) {
                std::cout << "recv timeout\n";
                continue;
            }

            std::cout << "reply: " << reply.to_string() << '\n';
        }
    } catch (const zmq::error_t& e) {
        std::cerr << "zmq error: " << e.what() << '\n';
        return 1;
    } catch (const std::exception& e) {
        std::cerr << "std error: " << e.what() << '\n';
        return 1;
    }
}
```

# 十一、ZMQ 的典型坑

## 1. 把它当普通 socket 用

ZMQ 不是“换个 API 的 TCP”。
它有自己的模式约束，尤其 `REQ/REP`。

## 2. PUB 消息丢失

订阅刚建立时，消息不保证能收到。

## 3. 一个 socket 多线程共用

很容易出问题。

## 4. 不设置超时

程序容易永远卡住。

## 5. 不理解消息边界

虽然 ZMQ 有消息边界，但如果你发 multipart，就要按帧处理。

## 6. 退出时卡住

常见是 `linger` 没处理好。

---

# 十二、怎么理解 ZMQ 在工程里的位置

你可以把它想成三层：

## 第一层：OS socket

最底层的 TCP/UDP 能力。

## 第二层：libzmq

在 socket 上实现消息队列、重连、路由、模式。

## 第三层：cppzmq

给 C++ 程序员更舒服的 API。

所以：

```text
cppzmq/zmq.hpp
    ↓
libzmq
    ↓
TCP / IPC / inproc
    ↓
操作系统 socket
```

# 十三、给你一个学习路线

如果你想真正掌握 ZMQ，建议按这个顺序学：

1. `REQ/REP`
2. `PUB/SUB`
3. `PUSH/PULL`
4. 超时、非阻塞、HWM、linger
5. multipart message
6. `DEALER/ROUTER`
7. 线程模型和 `inproc`

# 十四、和 ROS / Cyber RT 的关系怎么理解

它们不完全是一回事，但可以类比：

* `PUB/SUB` 很像 ROS topic
* `REQ/REP` 有点像 service
* `PUSH/PULL` 像任务管道
* 更复杂的中间件会在此基础上增加：

  * 序列化
  * 服务发现
  * QoS
  * 共享内存
  * 拓扑管理

ZeroMQ 更“轻”，更偏通信库；
ROS / Cyber RT / DDS 更“重”，更像完整中间件。
