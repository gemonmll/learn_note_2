`std::shared_ptr` 本身是 RAII，不“容易”泄漏；**真正导致“内存一直不释放/看起来像泄漏”**的常见原因，基本都和“引用计数永远归零不了”或“对象生命周期被意外延长”有关。下面按高频原因讲，并给出最小代码例子（含修复方式）。

---

## 1) 循环引用（最常见、最典型）

两个对象互相用 `shared_ptr` 持有对方，引用计数永远 >0，析构永远不会被调用。

### ❌ 会泄漏

```cpp
#include <memory>
#include <iostream>

struct B;

struct A {
    std::shared_ptr<B> b;
    ~A() { std::cout << "~A\n"; }
};

struct B {
    std::shared_ptr<A> a;
    ~B() { std::cout << "~B\n"; }
};

int main() {
    auto a = std::make_shared<A>();
    auto b = std::make_shared<B>();
    a->b = b;
    b->a = a;   // 循环引用
} // ~A ~B 都不会打印
```

### ✅ 修复：把“非拥有关系”改成 `std::weak_ptr`

```cpp
#include <memory>
#include <iostream>

struct B;

struct A {
    std::shared_ptr<B> b;
    ~A() { std::cout << "~A\n"; }
};

struct B {
    std::weak_ptr<A> a; // 关键：weak_ptr 不增加引用计数
    ~B() { std::cout << "~B\n"; }
};

int main() {
    auto a = std::make_shared<A>();
    auto b = std::make_shared<B>();
    a->b = b;
    b->a = a;

    if (auto locked = b->a.lock()) {
        // 使用 locked
    }
} // 正常析构
```

## 2) `shared_ptr` 的“别名构造 / 自定义 deleter”误用导致资源没释放（看起来像泄漏）

`shared_ptr` 可以管理“不是 new 出来的指针”（比如指向内部成员、外部内存、C API），如果 deleter 写错或根本没释放真正资源，就会泄漏底层资源。

### ❌ 自定义 deleter 写错：没有释放

```cpp
#include <memory>
#include <cstdlib>

int main() {
    int* p = (int*)std::malloc(100 * sizeof(int));

    std::shared_ptr<int> sp(p, [](int*) {
        // 忘了 free，底层内存泄漏
    });
}
```

### ✅ 修复：正确释放

```cpp
#include <memory>
#include <cstdlib>

int main() {
    int* p = (int*)std::malloc(100 * sizeof(int));

    std::shared_ptr<int> sp(p, [](int* q) {
        std::free(q);
    });
}
```

> 类似地，管理 `FILE*` / socket / `pthread_mutex_t*` 等也要保证 deleter 做对。

## 3) 多个 `shared_ptr` **分别**从同一个裸指针构造（严重 Bug：双重释放/未定义行为，也可能表现为“泄漏/崩溃”）

这不是传统意义“泄漏”，但很常见：把同一裸指针交给两个不同控制块管理。

### ❌ 错误用法（未定义行为）

```cpp
#include <memory>

struct X { int v = 0; };

int main() {
    X* raw = new X;

    std::shared_ptr<X> a(raw);
    std::shared_ptr<X> b(raw); // 两个控制块，最后会 delete 两次
}
```

### ✅ 修复：只构造一次，然后拷贝 shared_ptr

```cpp
#include <memory>

struct X { int v = 0; };

int main() {
    auto a = std::make_shared<X>();
    auto b = a; // 共享同一控制块
}
```

## 4) `enable_shared_from_this` 使用不当：对象并非由 `shared_ptr` 管理时调用 `shared_from_this`

这通常会抛 `std::bad_weak_ptr` 或导致逻辑错误；另一个常见坑是把 `this` 再包一层 `shared_ptr(this)`（又回到第3条的“双控制块”灾难）。

### ❌ 错误：手动 `shared_ptr(this)`

```cpp
#include <memory>

struct Foo : std::enable_shared_from_this<Foo> {
    std::shared_ptr<Foo> get() {
        return std::shared_ptr<Foo>(this); // 错：新控制块
    }
};

int main() {
    auto p = std::make_shared<Foo>();
    auto q = p->get(); // 将来可能 double delete
}
```

### ✅ 修复：用 `shared_from_this()`

```cpp
#include <memory>

struct Foo : std::enable_shared_from_this<Foo> {
    std::shared_ptr<Foo> get() {
        return shared_from_this();
    }
};

int main() {
    auto p = std::make_shared<Foo>();
    auto q = p->get(); // 正确
}
```

## 5) 回调/全局容器/线程把 `shared_ptr` 捕获住，导致对象生命周期被无限延长（“像泄漏”）

比如注册了回调但从不注销；或线程/定时器一直持有 `shared_ptr`，对象永远不析构。

### ❌ 回调一直持有

```cpp
#include <memory>
#include <functional>
#include <vector>

std::vector<std::function<void()>> g_callbacks;

struct Obj {
    void start() {
        auto self = shared_from_this();
        g_callbacks.push_back([self] {
            // self 被永久保存在全局回调里
        });
    }
    ~Obj() {}
    std::shared_ptr<Obj> shared_from_this() { return std::shared_ptr<Obj>(this); } // 这里只是示意，实际更糟
};
```

更实际一点的模式（合法但会“延长生命周期”）：

```cpp
#include <memory>
#include <functional>
#include <vector>

std::vector<std::function<void()>> g_callbacks;

struct Obj : std::enable_shared_from_this<Obj> {
    void start() {
        auto self = shared_from_this();
        g_callbacks.push_back([self] {
            // 只要 g_callbacks 不清空，Obj 就不会析构
        });
    }
    ~Obj() {}
};

int main() {
    auto o = std::make_shared<Obj>();
    o->start();
    o.reset(); // 这里不会析构，因为回调里还持有
}
```

### ✅ 修复思路

* 回调里改捕获 `weak_ptr`，用时 `lock()`，对象没了就不执行；
* 或提供注销机制（返回 token/handle，析构时自动 unregister）。

```cpp
#include <memory>
#include <functional>
#include <vector>

std::vector<std::function<void()>> g_callbacks;

struct Obj : std::enable_shared_from_this<Obj> {
    void start() {
        std::weak_ptr<Obj> weak = weak_from_this();
        g_callbacks.push_back([weak] {
            if (auto self = weak.lock()) {
                // 使用 self
            } else {
                // 对象已销毁，跳过
            }
        });
    }
};
```

## 6) `shared_ptr` 与 C API/第三方库混用：库内部缓存引用或未释放资源

比如你把 `shared_ptr` 的指针交给库，库把它存成全局/静态，或者引用计数/释放协议不一致，导致“你以为释放了”，但库还持有。

修复原则：

* 明确所有权：谁负责释放、何时释放；
* 尽量使用**单一所有权入口**（工厂函数返回 `shared_ptr`）；
* 边界用 `unique_ptr + 自定义 deleter` 或 `shared_ptr + 正确 deleter`，并让“拥有者”唯一。

---

### 一句话总结

`shared_ptr` 出问题的核心就两类：

1. **引用计数无法归零**（循环引用、被容器/回调/线程长时间持有）
2. **控制块/释放策略错误**（重复从裸指针构造、deleter 错、错误的 `this` 包装）
