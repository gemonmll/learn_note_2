关键在这句：

```cpp
auto self = shared_from_this();
```

然后回调捕获了它：

```cpp
[this, self](boost::system::error_code ec, std::size_t length) {
    if (!ec) {
        do_read();
    }
}
```

这里的 `self` 是一个 `std::shared_ptr<Session>`。

只要 lambda 还没被销毁，lambda 里面捕获的 `self` 就还活着。
只要 `self` 还活着，它指向的 `Session` 对象就不会析构。

所以生命周期链条是：

```text
async_read_some 内部保存 handler
handler 是这个 lambda
lambda 捕获了 self
self 是 shared_ptr<Session>
shared_ptr 保持 Session 不析构
异步操作完成后，io_context 执行 lambda
lambda 执行结束后被释放
self 被释放
如果没有其他 shared_ptr，Session 才析构
```

也就是说，这段代码保证的是：

> 在这个异步读取回调执行之前和执行期间，`Session` 对象不会被析构。

---

看一个危险写法：

```cpp
void do_read() {
    socket_.async_read_some(
        boost::asio::buffer(data_),
        [this](boost::system::error_code ec, std::size_t length) {
            if (!ec) {
                do_read();
            }
        }
    );
}
```

这里只捕获了：

```cpp
[this]
```

`this` 是裸指针，不增加引用计数。

如果外部的 `shared_ptr<Session>` 被释放了：

```cpp
auto s = std::make_shared<Session>(...);
s->start();

s.reset(); // Session 可能析构
```

但异步读还没完成。之后网络事件来了，回调执行：

```cpp
[this] {
    do_read();
}
```

这里的 `this` 可能已经指向被销毁的对象，程序就有未定义行为。

---

安全写法：

```cpp
void do_read() {
    auto self = shared_from_this();

    socket_.async_read_some(
        boost::asio::buffer(data_),
        [this, self](boost::system::error_code ec, std::size_t length) {
            if (!ec) {
                do_read();
            }
        }
    );
}
```

即使外部释放了：

```cpp
s.reset();
```

也没关系。

因为异步回调还持有一份：

```cpp
self
```

所以 `Session` 还不会析构。

---

为什么还要捕获 `this`？
因为写成员函数方便：

```cpp
do_read();
```

实际上你也可以不用 `this`，改成：

```cpp
[self](boost::system::error_code ec, std::size_t length) {
    if (!ec) {
        self->do_read();
    }
}
```

这更直观：

```cpp
void do_read() {
    auto self = shared_from_this();

    socket_.async_read_some(
        boost::asio::buffer(data_),
        [self](boost::system::error_code ec, std::size_t length) {
            if (!ec) {
                self->do_read();
            }
        }
    );
}
```

这个版本更明显地表达：

```text
回调持有 self
通过 self 调用成员函数
对象一定还活着
```

很多代码写成：

```cpp
[this, self]
```

只是为了少写 `self->`。

---

但是要注意一个前提：

这个类必须继承：

```cpp
std::enable_shared_from_this<Session>
```

并且对象一开始必须由 `std::shared_ptr` 管理：

```cpp
auto session = std::make_shared<Session>(std::move(socket));
session->start();
```

不能这样：

```cpp
Session s;
s.start(); // 错
```

也不能这样：

```cpp
auto p = new Session(...);
p->start(); // 错
```

否则：

```cpp
shared_from_this()
```

会出问题，通常会抛出：

```cpp
std::bad_weak_ptr
```

---

再看递归读的生命周期：

```cpp
void do_read() {
    auto self = shared_from_this();

    socket_.async_read_some(
        boost::asio::buffer(data_),
        [self](boost::system::error_code ec, std::size_t length) {
            if (!ec) {
                self->do_read();
            }
        }
    );
}
```

每一轮都是：

```text
第 1 次 do_read()
  创建 self1
  async_read_some 保存回调1
  回调1 持有 self1

读完成
  执行回调1
  self1 保证对象活着
  调用 self->do_read()

第 2 次 do_read()
  创建 self2
  async_read_some 保存回调2
  回调2 持有 self2

回调1 结束
  self1 释放
  但 self2 已经被下一次回调持有
```

所以只要异步读链条一直继续，`Session` 就一直活着。
一旦出错，不再调用 `do_read()`：

```cpp
if (!ec) {
    self->do_read();
}
```

如果没有其他地方持有 `Session`，当前回调结束后 `self` 释放，对象就析构。

这正好符合网络连接的生命周期：

```text
连接正常：继续 async_read，Session 活着
连接断开：不再 async_read，Session 自动释放
```

一句话总结：

> `[this, self]` 能保证生命周期，不是因为 `this` 安全，而是因为 `self` 是 `shared_ptr`，lambda 持有 `self`，Asio 又持有 lambda，所以异步回调执行完之前对象不会被销毁。
