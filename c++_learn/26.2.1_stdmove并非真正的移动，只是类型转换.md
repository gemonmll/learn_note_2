```cpp
#ifndef CYBER_DATA_CACHE_BUFFER_H_
#define CYBER_DATA_CACHE_BUFFER_H_

#include <functional>
#include <memory>
#include <mutex>
#include <vector>

namespace apollo {
namespace cyber {
namespace data {

template <typename T>
class CacheBuffer {
 public:
  using value_type = T;
  using size_type = std::size_t;
  using FusionCallback = std::function<void(const T&)>;

  explicit CacheBuffer(uint64_t size) {
    capacity_ = size + 1;
    buffer_.resize(capacity_);
  }

  CacheBuffer(const CacheBuffer& rhs) {
    std::lock_guard<std::mutex> lg(rhs.mutex_);
    head_ = rhs.head_;
    tail_ = rhs.tail_;
    buffer_ = rhs.buffer_;
    capacity_ = rhs.capacity_;
    fusion_callback_ = rhs.fusion_callback_;
  }

  T& operator[](const uint64_t& pos) { return buffer_[GetIndex(pos)]; }
  const T& at(const uint64_t& pos) const { return buffer_[GetIndex(pos)]; }

  uint64_t Head() const { return head_ + 1; }
  uint64_t Tail() const { return tail_; }
  uint64_t Size() const { return tail_ - head_; }

  const T& Front() const { return buffer_[GetIndex(head_ + 1)]; }
  const T& Back() const { return buffer_[GetIndex(tail_)]; }

  bool Empty() const { return tail_ == 0; }
  bool Full() const { return capacity_ - 1 == tail_ - head_; }
  uint64_t Capacity() const { return capacity_; }

  void SetFusionCallback(const FusionCallback& callback) {
    fusion_callback_ = callback;
  }

  void Fill(const T& value) {
    if (fusion_callback_) {
      fusion_callback_(value);
    } else {
      if (Full()) {
        buffer_[GetIndex(head_)] = value;
        ++head_;
        ++tail_;
      } else {
        buffer_[GetIndex(tail_ + 1)] = value;
        ++tail_;
      }
    }
  }

  std::mutex& Mutex() { return mutex_; }

 private:
  CacheBuffer& operator=(const CacheBuffer& other) = delete;
  uint64_t GetIndex(const uint64_t& pos) const { return pos % capacity_; }

  uint64_t head_ = 0;
  uint64_t tail_ = 0;
  uint64_t capacity_ = 0;
  std::vector<T> buffer_;
  mutable std::mutex mutex_;
  FusionCallback fusion_callback_;
};

}  // namespace data
}  // namespace cyber
}  // namespace apollo
```

## 你的 `CacheBuffer` 里只有拷贝构造

代码里定义了：

```cpp
CacheBuffer(const CacheBuffer& rhs)
```

但**没有定义**：

```cpp
CacheBuffer(CacheBuffer&& rhs)
```

所以 `std::move(buffer)` 只是把 `buffer` 强制转成了一个 **右值**，它本身**不执行移动**。

编译器接下来会找：

1. 有没有 `CacheBuffer(CacheBuffer&&)` —— 没有
2. 那能不能用拷贝构造 `CacheBuffer(const CacheBuffer&)` —— 可以，因为 `const T&` 能绑定右值

于是最后调用的是：

```cpp
CacheBuffer(const CacheBuffer& rhs)
```

也就是**拷贝构造**，不是移动构造。

```cpp
buffer_ = rhs.buffer_;
capacity_ = rhs.capacity_;
fusion_callback_ = rhs.fusion_callback_;
```

这意味着：

* `buffer1` 拿到一份和 `buffer` 一样的数据
* `buffer` 自己一点没变

## `std::move` 的常见误区

很多人会以为：

```cpp
std::move(x)
```

就一定会把 `x` 清空。

其实不是。

`std::move` 只做一件事：

> 把 `x` 转成“可以被移动”的右值引用。

至于后面到底是：

* 真 move
* 还是 copy
* 还是根本编译不过

取决于类有没有提供合适的移动操作。

---

## 为什么编译器没自动生成 move constructor

因为你手写了拷贝构造函数：

```cpp
CacheBuffer(const CacheBuffer& rhs)
```

一旦用户自定义了拷贝构造，编译器通常就**不会再自动给你生成你期望的移动构造**了。

而且这个类里还有：

```cpp
std::mutex mutex_;
```

`std::mutex` 是**不可拷贝、不可移动**的，这也让默认生成 move/copy 行为更受限制，所以作者才手写了一个拷贝构造，只拷贝那些能拷贝的成员，而不给 `mutex_` 做复制。

---

## 这段测试本质上测的是啥

这段测试虽然写了 `std::move(buffer)`，但实际测出来的是：

> “把一个右值传给拷贝构造函数后，新对象内容和原对象一致。”

也就是说，这更像是在测：

* 类能不能从“右值形式的源对象”构造
* 由于没有 move constructor，最后退化成 copy constructor

所以测试后原 `buffer` 还能正常读，完全合理。

---

## 如果真想让 move 后源对象“被搬空”

那得显式提供移动构造，例如：

```cpp
CacheBuffer(CacheBuffer&& rhs) noexcept {
  std::lock_guard<std::mutex> lg(rhs.mutex_);
  head_ = rhs.head_;
  tail_ = rhs.tail_;
  capacity_ = rhs.capacity_;
  buffer_ = std::move(rhs.buffer_);
  fusion_callback_ = std::move(rhs.fusion_callback_);

  rhs.head_ = 0;
  rhs.tail_ = 0;
  rhs.capacity_ = 0;
}
```

但这里还要注意：

* `mutex_` 不能 move，只能给新对象默认构造一个新的 mutex
* move 以后源对象要保持“有效但未指定状态”
* 不能随便假设 `Front()` / `Back()` 还能安全访问

---

`buffer` 在 `std::move(buffer)` 后还有数据，是因为 `std::move` 只是类型转换；你的 `CacheBuffer` 没有移动构造，最终调用的是**拷贝构造**，所以源对象 `buffer` 根本没被搬空。

你这个例子其实很适合顺便理解一句话：

##  rhs.head_ = 0;
##  rhs.tail_ = 0;

可以。因为这里的 `rhs` 不是“新对象副本”，而是**绑定到原对象本体的引用**。

你看到的 move 构造一般长这样：

```cpp
CacheBuffer(CacheBuffer&& rhs) noexcept
```

这里的 `rhs` 类型是：

```cpp
CacheBuffer&&
```

也就是**右值引用**。引用的本质和 `T&` 一样，都是对象的别名，只不过它绑定的是右值。

所以当你写：

```cpp
rhs.head_ = 0;
rhs.tail_ = 0;
rhs.capacity_ = 0;
```

改的就是**原来的那个对象**。

## 类比普通左值引用更容易懂

比如：

```cpp
void f(int& x) {
  x = 100;
}

int a = 1;
f(a);
```

调用完后，`a` 就变成 100 了。因为 `x` 是 `a` 的引用。

右值引用同理：

```cpp
void g(int&& x) {
  x = 200;
}

int a = 1;
g(std::move(a));
```

调用完后，`a` 也会变成 200。因为 `std::move(a)` 只是把 `a` 转成右值表达式，`x` 仍然绑定的是 **a 本身**。

---

## 对象版看一下

```cpp
struct A {
  int n = 10;
};

A a;

A&& r = std::move(a);
r.n = 42;
```

这之后：

```cpp
a.n == 42
```

因为 `r` 和 `a` 指向的是同一个对象。

## 所以 move 构造里发生了什么

比如：

```cpp
CacheBuffer(CacheBuffer&& rhs) noexcept {
  buffer_ = std::move(rhs.buffer_);
  rhs.head_ = 0;
}
```

当你写：

```cpp
CacheBuffer<int> b(std::move(a));
```

可以理解成：

* `rhs` 绑定到 `a`
* `buffer_ = std::move(rhs.buffer_)` 实际是在“从 `a.buffer_` 挪资源”
* `rhs.head_ = 0` 实际就是 `a.head_ = 0`

所以当然会影响原对象 `a`。

---

## 为什么很多人会误解

因为 `std::move(a)` 看起来像“生成了一个临时对象”。
其实不是。

它更接近于一种**强制类型转换**：

```cpp
static_cast<CacheBuffer&&>(a)
```

没有创建新对象，只是把 `a` 标记成“你可以把我当右值用了”。

---

## 但要注意一点

move 之后，原对象**不是消失**，而是处于：

> 有效但值未指定的状态

也就是说：

* 对象还活着
* 可以析构
* 可以重新赋值
* 但里面具体值是什么，不该随便依赖

所以像你自己写 move 构造时：

```cpp
rhs.head_ = 0;
rhs.tail_ = 0;
rhs.capacity_ = 0;
```

这是在主动把源对象整理成一个你可控的状态。

---

## 最关键的一句

`rhs` 能影响原对象，是因为它是**引用**，不是副本。
`std::move(buffer)` 也没有创建新对象，它只是让这个原对象以右值引用的方式被绑定和修改。

# **问题2 我还是不理解右值引用和普通引用**

把它先和“能不能取地址、能不能长期存在”分开。
**引用本质上都是别名**，区别主要在于：**它们能绑定什么样的表达式**。

---

## 先看普通引用

普通引用一般指左值引用：

```cpp
int a = 10;
int& r = a;
```

这里：

* `a` 是一个有名字、能持续存在的对象
* `r` 是 `a` 的别名
* 改 `r` 就是在改 `a`

```cpp
r = 20;
// a 也变成 20
```

左值引用 `T&` 的特点是：

* **只能绑定左值**
* 左值可以简单理解成：**有身份、能反复使用的对象**

比如：

```cpp
int a = 10;
int& r1 = a;   // 可以
int& r2 = 5;   // 不可以
```

因为 `5` 是临时值，不是左值。

---

## 再看右值引用

右值引用写法是：

```cpp
int&& rr = 5;
```

它的特点是：

* **主要绑定右值**
* 右值可以先粗略理解成：**临时值、将亡值、没有稳定身份的表达式**

例如：

```cpp
int&& rr1 = 5;        // 可以
int&& rr2 = a + 1;    // 可以
int&& rr3 = a;        // 不可以
```

因为：

* `5` 是右值
* `a + 1` 结果是临时值，也是右值
* `a` 是左值，不行

但如果你强制把左值转成右值：

```cpp
int a = 10;
int&& rr = std::move(a);
```

这就可以。

---

# 最核心的区别

你现在卡住的点，大概率是这个：

> 为什么 `int&` 和 `int&&` 都像“引用”，但又不一样？

答案是：

* `T&`：给“**正常活着的对象**”起别名
* `T&&`：给“**临时值/可被搬走的对象**”起别名

它们都是别名。
不是说右值引用就不是引用，也不是说它引用的是副本。

---

## 一个最直观的例子

### 左值引用

```cpp
int a = 10;
int& r = a;
r = 30;
```

结果：

```cpp
a == 30
```

因为 `r` 就是 `a` 的别名。

---

### 右值引用绑定临时值

```cpp
int&& rr = 5;
rr = 20;
```

这里也完全合法。
虽然 `5` 本来是临时值，但一旦被右值引用绑定，这个临时对象的生命周期会延长到 `rr` 的生命周期内。

所以：

```cpp
std::cout << rr;  // 20
```

---

## 右值引用也能改原对象

这个最重要。

```cpp
int a = 10;
int&& rr = std::move(a);
rr = 99;
```

结果：

```cpp
a == 99
```

因为：

```cpp
std::move(a)
```

**没有创建新对象**，只是把 `a` 转成“右值形式”。
`rr` 绑定的还是 `a` 本体。

所以：

* `rr` 改了
* 就等于 `a` 改了

---

# 为什么要发明右值引用

因为 C++ 想区分两类对象：

一种是：

* 我还要继续用
* 你别乱拿我资源

另一种是：

* 这是临时的
* 或者我已经明确说“你可以拿走它的资源”

右值引用就是为第二类服务的。
这使得“移动语义”成为可能。

---

## 没有右值引用时

比如一个大 vector：

```cpp
std::vector<int> v1 = {1,2,3,4,5};
std::vector<int> v2 = v1;   // 拷贝
```

这里通常会复制整块数据，成本高。

---

## 有右值引用时

```cpp
std::vector<int> v1 = {1,2,3,4,5};
std::vector<int> v2 = std::move(v1);   // 移动
```

意思是：

* `v1` 的资源你可以接管
* 不必深拷贝

这就是右值引用最大的价值。

---

# 你可以先这样记

## 左值引用 `T&`

像是：

> “这个对象我还要继续正常使用，你只是给它取个别名。”

---

## 右值引用 `T&&`

像是：

> “这个对象是临时的，或者我已经声明它可以被搬空，你也给它取个别名吧。”

---

# 为什么 `std::move(a)` 后还能改到 `a`

因为：

```cpp
int&& rr = std::move(a);
```

本质近似：

```cpp
int&& rr = static_cast<int&&>(a);
```

这只是类型转换，不是复制。

所以：

* `a` 还是那个 `a`
* `rr` 只是它的另一个名字
* 只是这个名字带着“可移动”的语义

---

# 再对比一下

看这个函数：

```cpp
void f(int& x) {
  x = 1;
}

void g(int&& x) {
  x = 2;
}
```

调用：

```cpp
int a = 10;

f(a);            // 可以
// f(5);         // 不可以

g(5);            // 可以
g(std::move(a)); // 可以
// g(a);         // 不可以
```

结果：

```cpp
g(std::move(a));
```

执行后，`a` 可能就被改了，因为 `x` 绑定的还是 `a`。

---

# 一句最本质的话

**左值引用和右值引用，都是“引用”，都是原对象的别名。**
它们的主要区别不在“是不是副本”，而在于：

**它们能绑定哪一类表达式。**

---

# 用你现在最该抓住的版本总结

* `T&`：绑定左值
* `T&&`：绑定右值
* 它们都不是副本，都是别名
* `std::move(x)` 不会创建新对象，只是把 `x` 转成右值
* 所以右值引用参数 `rhs` 修改成员时，修改的就是原对象

## 问题3 那如果右值引用的生命周期比引用对象的生命周期长怎么把
不能这样用；那会变成悬空引用。

先分两种情况看。

## 1. 右值引用绑定的是临时对象

例如：

```cpp
int&& r = 5;
```

这里 `5` 本来是个临时量，但**被引用直接绑定后，临时对象生命周期会延长**，延长到 `r` 的生命周期结束。

所以这个是安全的：

```cpp
int&& r = 5;
std::cout << r << '\n';
```

可以把它理解成：编译器为这个临时值安排了更长的生存期，因为它被 `r` 接住了。

---

## 2. 右值引用绑定的是 `std::move(a)`

例如：

```cpp
int a = 10;
int&& r = std::move(a);
```

这时 `r` 绑定的不是临时新对象，而是 **a 本体**。

所以 `r` 的安全性取决于 `a` 还在不在。

例如这个安全：

```cpp
int a = 10;
int&& r = std::move(a);
r = 20;
std::cout << a << '\n';
```

因为 `a` 还活着。

但这个不安全：

```cpp
int&& f() {
    int a = 10;
    return std::move(a);
}
```

这里返回的是对局部变量 `a` 的右值引用。
函数结束时 `a` 已经销毁了，返回出去的引用就悬空了。

---

## 你问的“如果右值引用的生命周期比引用对象长怎么办”

答案是：

**引用本身可以活得更久，但一旦被引用对象先死了，这个引用就失效了。**

不管是：

* 左值引用 `T&`
* 右值引用 `T&&`

都一样。

引用不会“托管”对象生命，除非它绑定的是那种可被延长生命周期的临时对象初始化场景。

---

## 看几个例子

### 安全：临时对象生命周期被延长

```cpp
const std::string& r = std::string("hello");
```

以及：

```cpp
std::string&& r = std::string("hello");
```

这类直接用临时对象初始化引用，临时对象生命周期会延长到引用作用域结束。

---

### 不安全：返回局部对象引用

```cpp
std::string&& f() {
    std::string s = "hello";
    return std::move(s);   // 错
}
```

`s` 在函数结束就没了，返回的引用悬空。

---

### 不安全：绑定到已销毁对象

```cpp
int&& r = []() -> int&& {
    int x = 1;
    return std::move(x);
}();
```

这里也悬空。

---

## 一个很重要的误区

很多人以为：

```cpp
T&& r = std::move(x);
```

会“生成一个新的可延长生命周期的对象”。

不是。

它只是让 `r` 绑定到 `x`。
`x` 什么时候死，`r` 就什么时候不能用了。

---

## 你可以这样记

### 右值引用有两种常见来源

一种是绑定临时对象：

```cpp
T&& r = T{};
```

这种临时对象可延长生命周期。

另一种是绑定现有对象：

```cpp
T obj;
T&& r = std::move(obj);
```

这种不会创建新对象，也不会延长 `obj` 生命周期。

---

## 最后一句

**右值引用不会自动延长它所引用的一切对象的生命周期。**
它只在“直接绑定临时对象”时触发生命周期延长；如果它绑定的是 `std::move(a)` 后的 `a`，那还是看 `a` 自己活多久。
