`std::lock_guard` 和 `std::scoped_lock` 都是 C++ 的 RAII 锁管理器：构造时加锁，析构时自动解锁。主要区别在于**支持的锁数量**和**死锁避免能力**。

## 核心区别

| 对比项      | `std::lock_guard` | `std::scoped_lock`            |
| -------- | ----------------- | ----------------------------- |
| 引入版本     | C++11             | C++17                         |
| 可锁对象数量   | 只能管理一个 mutex      | 可管理一个或多个 mutex                |
| 多锁死锁避免   | 不支持               | 支持，内部使用类似 `std::lock` 的死锁避免算法 |
| 是否可手动解锁  | 不可                | 不可                            |
| 是否可移动/拷贝 | 不可                | 不可                            |
| 常见用途     | 简单单 mutex 加锁      | 单 mutex 或多个 mutex 同时加锁        |

## `lock_guard` 示例

```cpp
std::mutex m;

void f() {
    std::lock_guard<std::mutex> lock(m);
    // 临界区
}
```

它适合最普通的情况：只需要锁一个 mutex。

## `scoped_lock` 示例：锁一个 mutex

```cpp
std::mutex m;

void f() {
    std::scoped_lock lock(m); // C++17，可自动推导类型
    // 临界区
}
```

对于单个 mutex，`scoped_lock` 基本可以看作是更现代的 `lock_guard`。

## `scoped_lock` 示例：同时锁多个 mutex

```cpp
std::mutex m1, m2;

void f() {
    std::scoped_lock lock(m1, m2);
    // 同时持有 m1 和 m2
}
```

这比下面这种写法更安全：

```cpp
std::lock_guard<std::mutex> l1(m1);
std::lock_guard<std::mutex> l2(m2);
```

因为两个线程如果以不同顺序锁 `m1` 和 `m2`，就可能死锁：

```cpp
// 线程 A
std::lock_guard<std::mutex> l1(m1);
std::lock_guard<std::mutex> l2(m2);

// 线程 B
std::lock_guard<std::mutex> l1(m2);
std::lock_guard<std::mutex> l2(m1);
```

而 `std::scoped_lock lock(m1, m2);` 会避免这种典型死锁问题。

## 什么时候用哪个？

C++17 及以后，通常可以优先用：

```cpp
std::scoped_lock lock(m);
```

尤其是将来可能扩展为锁多个 mutex 时。

如果项目是 C++11/C++14，或者团队代码风格偏传统，用：

```cpp
std::lock_guard<std::mutex> lock(m);
```

就足够了。

## 补充：和 `unique_lock` 的区别

如果你需要手动 `unlock()`、延迟加锁、条件变量 `std::condition_variable`，应该用 `std::unique_lock`，不是 `lock_guard` 或 `scoped_lock`。

```cpp
std::unique_lock<std::mutex> lock(m);
cv.wait(lock);
```

一句话总结：**`lock_guard` 是单锁的简单 RAII 封装；`scoped_lock` 是 C++17 中更通用的 RAII 锁，最大优势是可以安全地同时锁多个 mutex。**
