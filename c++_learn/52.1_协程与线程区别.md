在 **Apollo Cyber RT** 里，**协程（CRoutine）和线程（std::thread / pthread）** 的区别主要体现在 **调度方式、资源开销、并发模型** 上。Cyber 的设计核心就是：**少量线程 + 大量协程** 来处理消息流。

下面用通俗方式解释，并给你一个接近 Cyber 的例子。

# 一、Cyber 协程 vs 线程核心区别

| 项目   | Cyber协程（CRoutine）      | 线程（Thread）   |
| ---- | ---------------------- | ------------ |
| 调度   | 用户态调度（Cyber Scheduler） | OS 内核调度      |
| 切换开销 | 很小（swapcontext）        | 很大（内核上下文切换）  |
| 数量   | 可以几万                   | 通常几十         |
| 栈空间  | 小（几十 KB）               | 大（1MB左右）     |
| 切换方式 | **主动 Yield**           | OS 抢占        |
| 使用场景 | 消息处理 / pipeline        | CPU任务 / IO任务 |

一句话总结：

```
线程 = OS调度的执行单位
协程 = 用户态调度的轻量任务
```

# 三、Cyber协程的典型执行流程

一个 Reader callback 会被包装成一个 CRoutine：

```cpp
factory.create_routine = [=]() {
  return [=]() {
    for (;;) {
      if (dv->TryFetch(msg)) {
        callback(msg);
        CRoutine::Yield(RoutineState::READY);
      } else {
        CRoutine::Yield();
      }
    }
  };
};
```

运行流程：

```
worker thread
     |
scheduler
     |
run CRoutine
     |
TryFetch
  |        |
成功      失败
  |        |
处理数据  Yield()
  |
Yield(READY)
```

协程执行特点：

* 处理一条消息
* 主动 `Yield`
* scheduler 切换到别的 CRoutine

# 问题 那协程cpu不会空转，那不会把其他的线程饿死吗，举一个例子说明下

# 问题 如果你愿意，我可以再给你讲一个 Cyber 协程里最关键的一个机制：为什么 TryFetch + Yield 能做到“数据一来立刻唤醒协程”。

这个其实是 Cyber DataVisitor 的核心设计，理解后你基本就能读懂 Cyber 的调度源码了。

可以，我把这套机制按 **“数据怎么一路传到 DataVisitor、怎么把协程从 DATA_WAIT 立刻变成 READY 并进入 runnable 队列”** 讲清楚。你会发现 `TryFetch + Yield` 只是表象，真正“立刻唤醒”的关键是 **DataVisitor 注册的 notifier 回调 + Scheduler::NotifyProcessor + CRoutine::SetUpdateFlag/UpdateState** 这一链路。

---

## 核心结论

* `TryFetch()` **不负责唤醒**，它只是协程被调度起来之后，去 **“取数据/融合数据”**。
* “数据一来立刻唤醒协程”靠的是：**数据到达时触发 Notify → Scheduler 唤醒/标记对应 CRoutine 为 READY → 放入可运行队列**。([Sigtry][1])

---

## 1) 先看你熟悉的协程主体：TryFetch + Yield

典型 Cyber component 的协程循环（你之前贴的）：

* 每轮先把状态设成 `DATA_WAIT`
* `TryFetch` 成功就跑回调 `f(...)`
* 然后 `Yield(READY)` 或 `Yield()` 让出执行权

这个模式的意义是：

* **没数据**：协程自己让出，并保持“我在等数据”的状态（DATA_WAIT）
* **有数据**：处理一批，然后让出（READY）保证公平，不独占线程 ([apollo.baidu.com][2])

---

## 2) 真正的“立刻唤醒”：数据到达时发生了什么

当 Writer 发送消息、底层 transport 收到消息后，Cyber 会把消息分发到对应的 reader/visitor 的 buffer（内部通常是 ChannelBuffer 一类），接着触发一条 **Notify 链**：

**Transport / Receiver → DataDispatcher → DataVisitor（buffer 入队）→ DataNotifier::Notify → notifier->callback → Scheduler::NotifyProcessor → 唤醒对应协程** ([CSDN博客][3])

其中关键点在于：

* **DataVisitor 在构造/初始化时就绑定了一个 notifier（回调）**，这个回调最终指向 scheduler 的 “唤醒某个 task/routine” 的逻辑。([Sigtry][1])
* 一旦数据进来，DataDispatcher/Notifier 就会执行这个回调，于是 scheduler 能“点名”对应的协程，把它变成可运行状态。([CSDN博客][3])

---

## 3) “从 DATA_WAIT 变 READY”的核心：CRoutine 的 updated_ 标志

Cyber 的 `CRoutine::UpdateState()` 有一个很关键的逻辑（文档里能看到）：

* 如果当前处于 `DATA_WAIT` 或 `IO_WAIT`
* 并且检测到 “被更新/被唤醒” 的标志
* 就把状态改回 `READY` ([apollo.baidu.com][4])

同时 `croutine.h` 里还有 `SetUpdateFlag()` 这类接口（文档可见），就是配合这个机制用的：([apollo.baidu.com][5])

所以你可以把“立刻唤醒”理解成两步：

1. **数据线程/分发线程**在数据到达时调用 notify 回调 → scheduler 找到目标 CRoutine，调用类似 `SetUpdateFlag()` / `Wake()` 的操作
2. **下一次调度点**（甚至马上）该 CRoutine 的 `UpdateState()` 会把 `DATA_WAIT → READY`，然后它就进入 runnable 队列等待执行 ([apollo.baidu.com][5])

> 这也是为什么你在协程里 `Yield()` 回去后，不需要忙等：因为“有人”会在数据到来时把你标记为可运行。

---

## 4) 用一段“时间线”把 TryFetch + Yield 为啥能立刻响应讲清楚

假设一个协程 `R` 在等 camera 数据：

**T0：R 在跑**

* `set_state(DATA_WAIT)`
* `TryFetch` 失败（没数据）
* `Yield()` → 切回 scheduler（R 现在是 DATA_WAIT，不占 CPU）

**T1：数据到达（另一个线程/回调路径）**

* DataDispatcher 把消息写入 DataVisitor 的 buffer
* DataNotifier::Notify 调 notifier->callback
* callback 触发 `Scheduler::NotifyProcessor(task_id)` 一类逻辑
* scheduler 把 R 做“更新标记/唤醒”，让 R 变成可运行候选 ([CSDN博客][3])

**T2：scheduler 选中 R**

* R 被切回运行
* 再次执行到 `TryFetch`，这次成功取到消息（或完成多通道 fusion）
* 执行业务回调
* `Yield(READY)` 让出（公平调度） ([Sigtry][1])

这就是“数据一来立刻唤醒”的本质：**唤醒发生在数据到达的那条 Notify 链上，而不是 TryFetch 里。**

---

## 5) 一个很直观的类比

把它类比成“条件变量/事件”：

* `Yield()` 相当于：我现在等事件，先挂起
* 数据到达的 Notify 相当于：`notify_one()` 点名叫醒你
* 被叫醒后你再 `TryFetch()` 去拿数据

只是 Cyber 这套是 **用户态协程 + 自己的调度器**，唤醒不是 OS 条件变量，而是 scheduler 的任务队列和 routine 状态机。([apollo.baidu.com][6])
