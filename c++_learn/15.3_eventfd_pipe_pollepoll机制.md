能，而且这几个就是 Linux 里最常见的“**阻塞等待事件**”工具。

你可以先这样理解：

* **pipe / eventfd**：负责“发通知”
* **poll / epoll**：负责“等通知”
* 都可以**阻塞等待**，不需要一直 while 死循环查。([man7.org][1])

---

## 1. 先说 `pipe` 是什么

`pipe()` 会创建一个**单向数据通道**，返回两个 fd：

* `pipefd[0]`：读端
* `pipefd[1]`：写端

写到写端的数据会先放在**内核缓冲区**里，读端再把它读出来。它本来就是给进程间通信用的。([man7.org][1])

最小例子：

```cpp
#include <unistd.h>
#include <cstdio>
#include <cstring>

int main() {
    int pfd[2];
    if (pipe(pfd) == -1) {
        perror("pipe");
        return 1;
    }

    const char* msg = "hello";
    write(pfd[1], msg, strlen(msg));

    char buf[64] = {0};
    read(pfd[0], buf, sizeof(buf));
    printf("read: %s\n", buf);

    close(pfd[0]);
    close(pfd[1]);
    return 0;
}
```

### 它怎么“阻塞”

* `read(pfd[0], ...)`：如果当前没数据可读，会阻塞
* `write(pfd[1], ...)`：如果 pipe 缓冲区满了，也可能阻塞
* 如果用 `O_NONBLOCK`，就不阻塞，而是返回 `EAGAIN`。([man7.org][1])

---

## 2. `eventfd` 是什么

`eventfd()` 创建的是一个**事件计数器 fd**。内核里维护一个 `uint64_t` 计数值：

* `write()` 往里写一个 8 字节整数：把这个值**加到计数器上**
* `read()` 从里读一个 8 字节整数：把计数器值取出来（或按 semaphore 模式减 1）

它特别适合做“**只通知，不传大块数据**”的场景。man page 也明确提到：很多“只是为了发信号”的 pipe 场景，可以用 eventfd 代替，而且内核开销更低，只需要一个 fd，不像 pipe 要两个。([man7.org][2])

最小例子：

```cpp
#include <sys/eventfd.h>
#include <unistd.h>
#include <cstdint>
#include <cstdio>

int main() {
    int efd = eventfd(0, 0);
    if (efd == -1) {
        perror("eventfd");
        return 1;
    }

    uint64_t one = 1;
    write(efd, &one, sizeof(one));   // 通知一次

    uint64_t v = 0;
    read(efd, &v, sizeof(v));        // 读出计数
    printf("eventfd read: %llu\n", (unsigned long long)v);

    close(efd);
    return 0;
}
```

### 它怎么“阻塞”

* 计数器是 0 时，`read()` 会阻塞，直到计数器变成非 0
* 计数器太大、再写会溢出时，`write()` 也会阻塞
* 如果 fd 设置成非阻塞，则失败并返回 `EAGAIN`。([man7.org][2])

---

## 3. `pipe` 和 `eventfd` 有什么区别

### `pipe`

更像“字节流通道”：

* 可以传实际数据
* 读多少、写多少都比较灵活
* 要两个 fd（读端、写端）([man7.org][1])

### `eventfd`

更像“计数型门铃”：

* 主要用来通知“有事发生了”
* 读写固定都是 8 字节整数
* 常用于唤醒线程/进程、配合 epoll
* 只要一个 fd，开销通常比 pipe 小。([man7.org][2])

所以工程里常见模式是：

* **共享内存放大数据**
* **eventfd 或 pipe 发通知**

比如：

```text
进程A:
  写共享内存
  -> write(eventfd, 1)

进程B:
  epoll_wait(...)
  -> 发现 eventfd 可读
  -> read(eventfd)
  -> 去读共享内存
```

---

## 4. `poll` 是什么

`poll()` 是“**同时等多个 fd**”的接口。
它的作用不是传数据，而是：

> 帮你等一组 fd，看看哪个现在可读、可写、报错、挂断。

它会等待“一组文件描述符中的某一个变成可执行 I/O”，如果都还没准备好，就阻塞。超时参数单位是毫秒：

* `< 0`：无限阻塞
* `= 0`：立即返回，不阻塞
* `> 0`：最多等这么久。([man7.org][3])

基本用法：

```cpp
#include <poll.h>
#include <unistd.h>
#include <cstdio>

int main() {
    int pfd[2];
    pipe(pfd);

    struct pollfd fds[1];
    fds[0].fd = pfd[0];
    fds[0].events = POLLIN;

    // 另一个线程/进程如果往 pfd[1] 写数据，这里就会醒
    int ret = poll(fds, 1, -1);  // 无限阻塞等待
    if (ret > 0 && (fds[0].revents & POLLIN)) {
        char c;
        read(pfd[0], &c, 1);
        printf("got data: %c\n", c);
    }
}
```

## 5. `epoll` 是什么

`epoll` 也是“等多个 fd”，但它比 `poll` 更适合**大量 fd**。
man page 里说得很直接：`epoll` 和 `poll` 做的是类似的事，但 `epoll` 对大数量的被监视 fd 扩展性更好。它内部有：

* **interest list**：你关心哪些 fd
* **ready list**：当前已经就绪的 fd

`epoll_wait()` 会在没有事件时阻塞等待。([man7.org][4])

基本三步：

1. `epoll_create1()` 创建 epoll 实例
2. `epoll_ctl()` 把 pipe/eventfd/socket 等 fd 加进去
3. `epoll_wait()` 阻塞等事件

最小骨架：

```cpp
#include <sys/epoll.h>
#include <sys/eventfd.h>
#include <unistd.h>
#include <cstdint>
#include <cstdio>

int main() {
    int efd = eventfd(0, 0);
    int ep = epoll_create1(0);

    epoll_event ev{};
    ev.events = EPOLLIN;
    ev.data.fd = efd;
    epoll_ctl(ep, EPOLL_CTL_ADD, efd, &ev);

    uint64_t one = 1;
    write(efd, &one, sizeof(one));   // 模拟通知

    epoll_event out[8];
    int n = epoll_wait(ep, out, 8, -1);  // 无限阻塞
    for (int i = 0; i < n; ++i) {
        if (out[i].data.fd == efd && (out[i].events & EPOLLIN)) {
            uint64_t v;
            read(efd, &v, sizeof(v));
            printf("eventfd notified: %llu\n", (unsigned long long)v);
        }
    }

    close(efd);
    close(ep);
}
```

## 6. `poll` 和 `epoll` 的区别

你可以先用一句话记：

* `poll`：**每次都把整张 fd 表交给内核看一遍**
* `epoll`：**先注册关心的 fd，以后只取“已经就绪的”**

更直观点：

### `poll`

* 简单
* fd 少时很好用
* 每次调用都要传整组 `pollfd[]`
* 大量 fd 时效率一般。([man7.org][3])

### `epoll`

* 更适合大量连接/大量 fd
* 先 `epoll_ctl` 注册，再 `epoll_wait`
* 内核维护 ready list
* 支持 **LT** 和 **ET** 两种触发模式。([man7.org][4])

---

## 7. LT 和 ET 是什么

`epoll` 有两种常见模式：

### LT（Level Triggered，水平触发）

默认模式。
只要 fd 还处于“可读/可写”状态，每次 `epoll_wait` 都可能继续告诉你。

好处是简单，不容易漏事件。

### ET（Edge Triggered，边沿触发）

只在“状态发生变化”时通知你一次。
比如 pipe 里从“空”变成“有数据”，给你一个事件；你如果没把数据一次读干净，下次可能就不再提醒你了。man page 特别举了 pipe 的例子说明：ET 下如果上次没读完，再次 `epoll_wait` 可能会挂住，所以通常应配合非阻塞 fd 并持续读到 `EAGAIN`。([man7.org][4])

所以入门建议：

* 先用 **LT**
* 以后再碰 **ET**

---

## 8. 它们能阻塞等待吗

能，而且这正是它们的常见用法。

### `pipe`

* `read()` 可以阻塞等数据
* `write()` 可以阻塞等缓冲区可写。([man7.org][1])

### `eventfd`

* `read()` 可以阻塞等计数器非 0
* `write()` 在极端情况下也可能阻塞。([man7.org][2])

### `poll`

* 可以阻塞，直到某个 fd ready、被信号打断、或超时
* `timeout < 0` 表示无限阻塞。([man7.org][3])

### `epoll_wait`

* 可以阻塞，直到有就绪事件
* 没事件时就睡着。([man7.org][4])

所以你前面问“共享内存是不是只能一直查”，答案就很自然了：

* **共享内存本身**不会主动叫醒别人
* 但你可以用 `eventfd` / `pipe` + `poll` / `epoll` 来**阻塞等待通知**
* 被唤醒后再去读共享内存

## 9. 最实用的搭配方式

### 方案 A：共享内存 + pipe

适合简单 demo、小系统。

```text
写进程:
  写共享内存
  -> 往 pipe 写 1 字节

读进程:
  poll/epoll 等 pipe 可读
  -> read(pipe)
  -> 读共享内存
```

### 方案 B：共享内存 + eventfd

更常见、更干净。

```text
写进程:
  写共享内存
  -> eventfd 写入 1

读进程:
  epoll_wait 阻塞等待
  -> eventfd 可读
  -> read(eventfd)
  -> 读共享内存
```

这通常比 pipe 更像“事件通知器”，也更轻量。([man7.org][2])

---

## 10. 什么时候选哪个

### 想传实际小数据

用 `pipe`

### 只是想“响铃通知一下”

用 `eventfd`

### fd 不多，逻辑简单

用 `poll`

### fd 多，或做 Linux 事件循环

用 `epoll`

---

## 11. 一句话总结

**`pipe` 和 `eventfd` 是可被等待的通知/通信 fd，`poll` 和 `epoll` 是等待这些 fd 发生事件的机制；它们都支持阻塞等待，所以并不需要一直轮询。** ([man7.org][1])
