```
插件编译时只保留了一个外部函数调用；等插件加载到进程里以后，这个调用再解析到 主程序函数的实现。我不理解这部分，so文件不会有主程序该函数的地址，那是如何调用的
```

你卡住的点其实正是**动态链接**的核心：

**`.so` 在编译/链接时确实不知道主程序里那个函数的最终地址；它保留的不是“地址”，而是“对某个符号名的引用 + 一条重定位记录”。加载到进程后，动态链接器再把这条引用解析成实际地址。**

下面按过程讲。

---

## 先看现象

假设插件里这样写：

```cpp
// plugin.so
extern "C" void host_log(const char*);

extern "C" void plugin_entry() {
    host_log("hello");
}
```

插件编译时：

* `plugin_entry` 的机器码里需要“调用 `host_log`”
* 但 `host_log` 的地址不知道，因为它在主程序里

所以 `.so` 里不会直接写死一个绝对地址，而是留下：

* 一个**未定义符号**：`host_log`
* 一条或多条**重定位信息**：告诉加载器“这里将来要填成 `host_log` 的地址”

---

## 1. `.so` 里保存的不是地址，而是“符号名 + 待修补位置”

你可以把它想成这样：

```text
plugin_entry:
    call ???     // 这里先不能确定
```

同时 ELF 里还有元数据大概表示：

```text
未定义符号: host_log
重定位表项: 代码/PLT/GOT 中某个位置需要绑定到符号 host_log
```

所以编译产物不是“不完整”，而是**故意把最终绑定推迟到加载时**。

---

## 2. 加载 `.so` 时，动态链接器会帮它“找人”

当主程序启动后加载这个插件时，动态链接器会做符号解析：

它会看：

* 主程序自己导出的符号
* 已加载的共享库导出的符号
* 当前加载顺序和符号可见性规则

然后它发现：

* `plugin.so` 需要 `host_log`
* 主程序里正好有 `host_log`

于是建立绑定关系：

```text
plugin.so: host_log  --->  main executable: host_log
```

注意这里用的是**符号名匹配**，不是插件提前知道地址。

---

## 3. 真正调用靠的是 GOT/PLT 或直接重定位

在 ELF/Linux 下，常见有两种实现方式，最典型的是 **PLT + GOT**。

### 情况 A：通过 PLT 调用函数

插件里的调用指令通常不会直接写成：

```text
call 0x7f12345678
```

而更像是：

```text
call host_log@plt
```

这里的 `@plt` 是一个跳板。

流程可理解为：

1. `plugin_entry` 调用 `host_log@plt`
2. `host_log@plt` 再去查 `GOT` 里对应槽位
3. `GOT` 槽位里最终会被动态链接器写入 `host_log` 的真实地址
4. 然后跳到主程序中的 `host_log`

也就是：

```text
plugin_entry
   -> PLT stub
      -> GOT entry
         -> 主程序中的 host_log 实际地址
```

---

## 4. 第一次调用前后，地址是怎么填进去的

有两种绑定策略。

### 立即绑定

加载 `.so` 时，动态链接器就把 `host_log` 的地址写进 GOT。

于是第一次调用就直接跳过去。

### 延迟绑定

第一次走到 `host_log@plt` 时，先进入动态解析器；
动态解析器这时才去查 `host_log` 真正在哪，然后把 GOT 改写好。
后面再调用时就直接跳真实地址了。

所以“插件一开始没有地址”完全没问题，因为**地址可以在加载时，甚至第一次调用时才确定**。

---

## 5. 为什么主程序的函数能被插件找到

前提是这个符号对动态链接器**可见**。

例如主程序：

```cpp
extern "C" void host_log(const char* s) {
    printf("HOST: %s\n", s);
}
```

如果这个符号被放进主程序的动态符号表里，插件就有机会解析到它。

在 Linux 上常见要求是主程序链接时带：

```bash
-rdynamic
```

或等价选项：

```bash
-Wl,--export-dynamic
```

否则：

* 主程序虽然自己能用 `host_log`
* 但它未必把这个符号导出给动态加载的 `.so`
* 插件加载时就可能报 `undefined symbol: host_log`

---

## 6. 所以 `.so` 到底“知道”什么

它知道的通常只有这些：

* 需要一个名字叫 `host_log` 的外部函数
* 某个调用点要绑定到这个符号
* 绑定类型是什么（函数调用、数据引用等）

它**不知道**：

* 主程序装载到哪个基址
* `host_log` 运行时的绝对地址
* ASLR 之后实际虚拟地址是多少

这些都由动态链接器在进程启动/`dlopen` 时解决。

---

## 7. 用一个更贴近底层的伪过程表示

### 编译/链接插件时

生成：

```text
符号表:
  host_log: UND   // undefined

重定位表:
  offset 0x1234 -> symbol host_log
```

### 插件加载时

动态链接器做：

```text
查找 symbol "host_log"
-> 在 main executable 中找到地址 0x555555556840

修补：
  将 GOT[host_log] = 0x555555556840
```

### 插件运行时

```text
call host_log@plt
-> jmp *GOT[host_log]
-> 跳到 0x555555556840
-> 执行主程序里的 host_log
```

---

## 8. 如果不是函数，而是主程序里的全局变量呢

原理相同，也是符号解析 + 重定位。

比如插件里：

```cpp
extern int host_state;
```

插件并不知道 `host_state` 的地址，但会留下：

* 未定义符号 `host_state`
* 对该符号的重定位项

加载时动态链接器把访问位置修补为主程序中 `host_state` 的实际地址。

函数常见走 PLT/GOT，数据符号更常见直接通过 GOT 或数据重定位访问。

---

## 9. 这和“静态链接”最大的区别

静态链接时：

* 最终链接阶段就把调用点改好
* 可执行文件里已经有确定的目标地址关系

动态链接时：

* 最终地址推迟到装载时
* `.so` 只带“我要找谁”的信息，不带最终绝对地址

---

## 10. 你可以这样记一句话

**插件里不是保存“主程序函数地址”，而是保存“对主程序函数名的未解析引用”；当插件被加载到进程里时，动态链接器根据符号表和重定位表，把调用路径修补到主程序函数的实际地址。**

---

## 11. 一个容易踩坑的点

如果你用 `dlopen` 加载插件，而插件要反向调用主程序函数，通常需要：

```bash
gcc main.c -rdynamic
```

否则主程序符号不导出，插件即使写了：

```cpp
extern void host_log(const char*);
```

也可能在加载时报：

```text
undefined symbol: host_log
```

---

## 12. 最后给你一张脑图式流程

```text
编译 plugin.so
  -> 生成对 host_log 的未定义符号引用
  -> 生成重定位记录

主程序启动 / dlopen(plugin.so)
  -> 动态链接器装载 plugin.so
  -> 查找 host_log 符号
  -> 在主程序导出符号中找到实现
  -> 修补 GOT/PLT 或相关重定位项

调用 plugin_entry()
  -> plugin 内部 call host_log@plt
  -> 经过 GOT 跳到主程序里的 host_log
```

如果你愿意，我下一条可以直接用一份**最小可运行例子**把 `main`、`plugin.so`、`readelf -Ws`、`objdump -d` 的输出对应起来讲，这样你会一下看懂 `UND`、`PLT`、`GOT`、`R_X86_64_JUMP_SLOT` 分别是什么。
