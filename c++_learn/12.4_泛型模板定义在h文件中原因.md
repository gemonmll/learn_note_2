通常要写在 `.h` 里，原因不是语法规定“只能写头文件”，而是：

**模板只有在“看到完整定义”时，编译器才能实例化出具体代码。**

所以你会感觉模板类/模板函数几乎都写在 `.h` 里。

---

## 先说为什么

普通函数比如：

```cpp
int add(int a, int b);
```

你可以：

* `.h` 里放声明
* `.cc` 里放定义

因为它就是一个已经确定的函数，编译器链接时去找实现就行。

---

但模板不一样。

比如：

```cpp
template <typename T>
T add(T a, T b) {
  return a + b;
}
```

这里真正的函数不是一个，而是很多潜在版本：

* `add<int>`
* `add<double>`
* `add<std::string>`

编译器只有在看到你实际调用：

```cpp
add<int>(1, 2);
```

时，才会生成对应实例。

### 关键点

**生成实例的那个编译单元，必须能看到模板定义。**

如果它只看到声明：

```cpp
template <typename T>
T add(T a, T b);
```

但看不到函数体，那它就没法生成 `add<int>` 的代码。

所以模板定义通常放头文件，这样每个使用它的 `.cc` 都能看到完整定义。

---

# 为什么普通函数不需要这样

因为普通函数的机器码可以在某个 `.cc` 单独编译出来，链接时大家共享。

模板函数则更像：

> “给编译器一张造函数的模具，等你用到某个具体类型时，现场造一个。”

所以“模具”必须可见。

---

# 你也可以不写在 `.h` 吗

可以，但要满足特殊条件。

## 方式 1：写在 `.h`

最常见：

```cpp
// a.h
template <typename T>
T add(T a, T b) {
  return a + b;
}
```

---

## 方式 2：声明放 `.h`，实现放 `.tcc/.ipp/.inl`，最后再被 `.h` include

这其实本质还是“放头文件体系里”。

例如：

```cpp
// a.h
template <typename T>
T add(T a, T b);

#include "a.tcc"
```

```cpp
// a.tcc
template <typename T>
T add(T a, T b) {
  return a + b;
}
```

这是很多库喜欢用的组织方式。逻辑上分文件了，但编译时还是让使用者看到完整定义。

---

## 方式 3：显式实例化

这时可以把模板定义放 `.cc`，但你要明确告诉编译器只生成哪些类型。

例如：

### 头文件

```cpp
// a.h
template <typename T>
T add(T a, T b);
```

### 源文件

```cpp
// a.cc
template <typename T>
T add(T a, T b) {
  return a + b;
}

template int add<int>(int, int);
template double add<double>(double, double);
```

这样只会生成：

* `add<int>`
* `add<double>`

别的类型比如 `add<std::string>` 就不能用了，除非你继续显式实例化。

所以这种方式适合：

* 模板参数种类有限
* 你想控制编译时间或二进制体积

---

# 你问“这种能 gdb 调试吗”

**能。**

模板函数、模板类实例化出来以后，对 GDB 来说就是正常函数、正常类型，只是名字会更长、更复杂。

例如你写：

```cpp
template <typename T>
T add(T a, T b) {
  return a + b;
}
```

调用：

```cpp
int x = add<int>(1, 2);
```

编译后实际会有类似：

* `add<int>(int, int)`

GDB 完全可以调。

---

## 你能做的调试操作

和普通函数差不多：

* 在模板定义处打断点
* 单步进入模板实例
* 看模板参数对应的具体类型
* 看局部变量
* 查看调用栈

---

## 断点长什么样

有时你直接在源码行打断点就行。

有时 GDB 里也可以按具体实例名断，比如：

```gdb
break add<int>
```

或者更复杂点的模板类成员函数：

```gdb
break 'Client<apollo::Req, apollo::Resp>::SendRequest(apollo::Req const&, std::chrono::seconds const&)'
```

模板函数名因为符号展开会比较长，所以在 VS Code 里点行号打断点通常更省事。

---

# 模板调试时常见现象

## 1）名字很长

因为实例化后的符号会带完整类型参数。

## 2）一个模板函数可能有多个实例

比如：

```cpp
add<int>
add<double>
```

你打在模板定义那一行，可能会命中多个类型版本。

## 3）优化会影响体验

如果编译开了优化，比如 `-O2`，模板代码尤其容易：

* 变量被优化掉
* 函数被内联
* 单步跳来跳去

所以调试时最好带：

```bash
-g -O0
```

或者至少低优化。

---

# 为什么很多人会误以为“模板不好调试”

不是不能调，而是：

* 名字太长
* 类型太复杂
* 很多代码被内联
* 报错和调用栈更啰嗦

尤其像你现在看的 Apollo/C++ 工程里，模板再叠：

* 智能指针
* lambda
* typedef / using
* protobuf 类型

GDB 输出会很长，所以看起来难。

但本质上是能调的。

# 一个最小例子

### 头文件

```cpp
// add.h
#pragma once

template <typename T>
T add(T a, T b) {
  T c = a + b;
  return c;
}
```

### 使用

```cpp
#include "add.h"
#include <iostream>

int main() {
  int x = add<int>(1, 2);
  std::cout << x << std::endl;
}
```

编译：

```bash
g++ -g -O0 main.cc -o main
```

GDB 里完全可以：

* 在 `add` 那行打断点
* step in
* print a
* print b
* print c

# 你可以记住这两句话

## 为什么模板常写 `.h`

因为**谁用模板，谁就得在编译时看到模板定义**，否则没法实例化。

## 能不能 GDB 调试

**能。** 模板实例化后就是普通可执行代码，只是符号名更复杂，优化开高了更难看。
