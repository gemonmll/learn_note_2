# 一、lock 是线程同步吗？

✅ **是的。**

`std::mutex` 这种锁的核心作用就是：

> 🔒 保证同一时间只有一个线程进入临界区
> 🔄 在多个线程之间进行同步

例如：

```cpp
std::mutex mtx;
int counter = 0;

void worker() {
    std::lock_guard<std::mutex> lock(mtx);
    counter++;
}
```

这里：

* 多个线程同时调用 `worker`
* `mutex` 保证 `counter++` 不会并发执行
* 避免数据竞争

这就是 **线程同步（thread synchronization）**

---

# 二、如果没取到锁，线程会等待吗？

### 对 `std::mutex::lock()` 来说：

✅ **会等待（阻塞）**

```cpp
mtx.lock();
```

如果：

* 另一个线程已经持有这个锁

当前线程就会：

> 💤 被操作系统挂起（block）
> 等待锁释放
> 然后被唤醒继续执行

这叫：

> 🔹 阻塞式等待（blocking wait）

---

# 三、lock 的几种行为对比

---

## 1️⃣ 普通 lock（会等待）

```cpp
mtx.lock();
```

拿不到锁：

> → 线程睡眠
> → 等待
> → 被唤醒

---

## 2️⃣ try_lock（不会等待）

```cpp
if (mtx.try_lock()) {
    // 成功
} else {
    // 失败，不会阻塞
}
```

特点：

> ❌ 不会等待
> ❌ 不会阻塞
> 立即返回

---

## 3️⃣ timed_mutex（可以限时等）

```cpp
std::timed_mutex mtx;

if (mtx.try_lock_for(std::chrono::seconds(2))) {
    // 两秒内拿到锁
}
```

---

# 四、底层到底发生了什么？

当线程执行：

```cpp
mtx.lock();
```

如果锁被占用：

1. 线程进入等待队列
2. 被 OS 挂起
3. 不再占用 CPU
4. 等持锁线程 unlock
5. OS 唤醒它

所以：

> ❗ mutex 不是自旋
> ❗ 它通常是阻塞的

---

# 五、和 atomic 对比一下

你刚刚学 atomic，我们对比一下：

| 对比     | mutex | atomic |
| ------ | ----- | ------ |
| 会不会阻塞  | 会     | 不会     |
| 会不会睡眠  | 会     | 不会     |
| 是否涉及内核 | 是     | 否      |
| 适合     | 复杂逻辑  | 简单变量   |

---

# 六、什么是自旋锁（spin lock）？

自旋锁不会睡眠，而是一直循环：

```cpp
while (!flag.exchange(true)) {
}
```

特点：

* 不会阻塞
* 一直占用 CPU
* 适合短时间锁

---

# 七、总结一句话

> 🔒 std::mutex 是线程同步工具
> ⏳ lock() 拿不到锁时线程会阻塞等待
> ⚡ atomic 不会阻塞

---

# 八、一个重要理解

很多人误解：

> “锁就是 while 循环等”

❌ 不对。

`std::mutex`：

* 不是纯自旋
* 是 OS + futex + 阻塞机制
