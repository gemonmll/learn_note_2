```cpp
namespace {
struct ProxyType123 {
  ProxyType123() {
    apollo::cyber::class_loader::utility::RegisterClass<Pear, Base>(
        "Pear", "Base");
  }
};
static ProxyType123 g_register_class_123;
```

关键点只有一个：

**它不是“注册进另一个程序”，而是“插件被 `dlopen` 到主程序所在的同一个进程后，直接往这个进程里的全局注册表写入一条记录”。**

流程是这样：

1. 主程序调用 `ClassLoader::LoadLibrary()`，最终走到 `utility::LoadLibrary()`，里面会执行 `dlopen`。[class_loader_utility.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.cc#L194C1) [shared_library.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/shared_library/shared_library.cc#L42C1)

2. `dlopen` 时，插件里的全局静态对象 `g_register_class_123` 会在**当前进程内**构造，于是调用 `RegisterClass<Pear, Base>("Pear", "Base")`。[class_loader_register_macro.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/class_loader_register_macro.h#L22C1)

3. `RegisterClass` 里会先 `new ClassFactory<Pear, Base>(...)`，再把这个指针塞进注册表。[class_loader_utility.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.h#L76C1)

4. 这个“注册表”本体在 `GetClassFactoryMapMap()` 里，是一份进程内的静态对象。[class_loader_utility.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.cc#L34C1)

所以运行时实际发生的是：

```cpp
// 插件的静态构造函数里执行
auto* f = new ClassFactory<Pear, Base>();
GetClassFactoryMapByBaseClass(typeid(Base).name())["Pear"] = f;
```

之后主程序再创建对象时，走的是同一张表：

```cpp
auto* factory = registry["Base"]["Pear"];
Base* obj = factory->CreateObj();
```

对应查表在这里：[class_loader_utility.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.h#L95C1)

你可以把它理解成：

- `ClassFactory<Pear, Base>` 对象是在插件代码里 `new` 出来的
- 这个对象存活在**同一个进程的堆上**
- 它的指针被放进 `utility` 的全局 map
- 主程序后来只是从 map 里拿出这个指针来用

所以不是“把工厂复制进主程序”，而是：

**插件加载后，在主程序所在进程里创建了工厂对象，并把它挂到了共享注册表上。**

再压缩成一句话：

**`dlopen` 让插件代码在主程序进程里执行；静态注册对象在这个时刻把 `ClassFactory<Pear, Base>*` 写进进程级注册表，主程序随后就能查到。**

## 疑问1 dlopen 时，插件里的全局静态对象 g_register_class_123 会在当前进程内构造，于是调用 RegisterClass<Pear, Base>("Pear", "Base")。class_loader_register_macro.h (line 22, column 1)这句话我不理解，插件中有全局变量的话，是如何注册的

你这里要区分两件事：

1. `g_register_class_123` 这个全局静态对象本身  
2. 它的构造函数带来的“注册副作用”

**Apollo 并没有去“注册这个全局变量”**。  
真正发生的是：**共享库被 `dlopen` 时，C++ 运行时会自动执行这个全局对象的构造函数；构造函数里再去调用 `RegisterClass(...)`。**

可以按这条时间线理解：

1. 插件源码里有这个对象：
```cpp
namespace {
struct ProxyType123 {
  ProxyType123() {
    RegisterClass<Pear, Base>("Pear", "Base");
  }
};
static ProxyType123 g_register_class_123;
}
```
位置来自宏展开，[class_loader_register_macro.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/class_loader_register_macro.h#L22C1)。

2. 编译插件时，编译器发现 `g_register_class_123` 是“带构造函数的全局静态对象”，就会额外生成一段“全局初始化代码”。
概念上类似：
```cpp
static void __cxx_global_var_init() {
  new (&g_register_class_123) ProxyType123();
}
```
这不是你手写的源码，而是编译器自动生成的初始化函数。

3. 链接成 `.so` 时，这个初始化函数会被放进共享库的初始化表里，通常是 ELF 的 `.init_array` 一类机制。
也就是说，so 自己带着一份“加载后要先执行哪些初始化函数”的清单。

4. 主程序调用 `dlopen()` 时，动态加载器把这个 so 映射进当前进程，然后会自动执行这张初始化清单。[shared_library.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/shared_library/shared_library.cc#L42C1)

5. 于是 `__cxx_global_var_init()` 被执行，`g_register_class_123` 被构造。

6. 构造函数里调用 `RegisterClass<Pear, Base>("Pear", "Base")`。

7. `RegisterClass` 再去创建 `ClassFactory<Pear, Base>`，并把它放进进程内全局注册表。[class_loader_utility.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.h#L76C1)  
这个注册表本体在 [class_loader_utility.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.cc#L34C1)。

所以本质不是：

“主程序发现插件里有个全局变量，然后把它注册起来”

而是：

“`dlopen` 触发插件全局静态对象构造，构造函数主动调用框架注册接口”

你可以把它压缩成一句伪代码：

```cpp
dlopen(libplugin2.so)
-> 运行 so 的全局初始化函数
-> 构造 g_register_class_123
-> ProxyType123::ProxyType123()
-> RegisterClass<Pear, Base>("Pear", "Base")
-> registry["Base"]["Pear"] = new ClassFactory<Pear, Base>()
```

顺手补一个容易混的点：

- 普通全局变量比如 `static int x = 1;` 不会有这种“注册效果”
- 这里能触发，是因为它是**类对象**，而且构造函数里有副作用

## 问题2 so文件中的全局对象会在主程序加载过程中会构造一份对象出来吗

会，但要分清楚“**构造一份**”和“**另外复制/new 一份**”不是一回事。

## 结论

**会构造。**
但这个构造通常是：

* 在 `.so` 自己的数据区里那块已经预留好的内存上
* 由动态链接器在加载阶段调用初始化代码
* 对这块内存执行构造函数

**不是**：

* 主程序再单独拷贝一份对象
* 或者默认在堆上 `new` 一份新的全局对象本体

---

## 更准确地说

例如 `.so` 里有：

```cpp
Foo g_obj(42);
std::string g_str = "hello";
```

当主程序启动并加载这个 `.so` 时，大致过程是：

1. 动态链接器把 `.so` 映射到进程地址空间
2. 为 `g_obj` / `g_str` 提供它们在 `.data/.bss` 中对应的内存位置
3. 处理重定位
4. 调用 `.init_array` / 全局构造相关代码
5. `Foo::Foo(42)`、`std::string` 构造函数在那块内存上执行

所以：

* **对象会被构造出来**
* 但构造位置是 `.so` 预留的静态存储区
* 不是又生成了另一份独立对象

---

## 分两种情况看最清楚

### 1. 平凡初始化对象

```cpp
int g = 123;
static int x = 0;
```

这种一般不需要真正调用构造函数：

* `g=123` 的值直接在 `.data`
* `x=0` 在 `.bss` 清零

这种可以认为“加载后值就已经在那里了”。

### 2. 非平凡构造对象

```cpp
std::string s = "abc";
Foo obj(1, 2);
```

这种才需要运行初始化代码：

* 动态链接器会调用构造逻辑
* 于是对象在静态存储区上“完成构造”

你问的“会构造一份对象出来吗”，主要对应的是这一类。
**答案是会。**

---

## 但不是“再额外多出一份”

更像这样理解：

```cpp
alignas(Foo) unsigned char storage[sizeof(Foo)];
new (storage) Foo(42);   // 近似理解
```

这里本质是：

* 先有一块固定内存
* 再在这块内存上构造对象

不是：

```cpp
Foo* p = new Foo(42);    // 这不是全局对象默认装载方式
```

---

## 一个容易混淆的点

虽然**全局对象本体**不会额外 `new` 一份，但对象内部可能在构造时申请堆内存。

例如：

```cpp
std::string s = "a long long string";
std::vector<int> v = {1,2,3,4};
```

这里：

* `s` / `v` 对象本体在 `.so` 的静态区
* 但它们内部维护的字符缓冲区、元素数组，可能在堆上分配

所以你看到构造阶段发生堆分配，不代表“全局对象本体被重新 new 了一份”。

---

## 一句话回答

**会，`.so` 中需要动态初始化的全局对象会在主程序加载该共享库时被构造；但构造是在共享库自身的静态存储区上完成的，不是主程序另外再复制或 `new` 一份同样的对象。**
