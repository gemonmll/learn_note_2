因为它们是**两个时代留下来的两套 IPC 设计**，后来都被广泛使用，结果 Linux/Unix 就一直同时保留了。

你可以先记一句：

* **System V**：更老，来自早期 UNIX System V 的 IPC 体系
* **POSIX**：更后来的标准化接口，想把接口设计得更统一、更现代一些

所以不是“必须同时发明两套”，而是：

> **先有老方案，后有新标准；但老方案用得太广，不能轻易废掉，于是两套都留下来了。**

---

## 1. System V 是老一代 IPC 风格

System V 这一套不只是共享内存，还有：

* `msgget` / 消息队列
* `semget` / 信号量
* `shmget` / 共享内存

它们有一套很统一的风格：

* 用 `key_t` 找对象
* 返回一个内核 id
* 再用这个 id 做后续操作

比如共享内存就是：

```text
key -> shmget -> shmid -> shmat
```

它的特点是：

* 历史很久
* 在很多老项目里大量存在
* 风格偏“内核对象编号”

---

## 2. POSIX 是后来想做统一标准

后来 POSIX 想把很多系统接口标准化，于是又定义了一套更“统一”的 IPC 风格。

POSIX 共享内存更像“文件对象”：

```text
name -> shm_open -> fd -> mmap
```

这个思路和别的 POSIX 接口更一致：

* 名字像路径
* 返回的是 fd
* 后续很多操作跟文件/mmap 模型接近

所以 POSIX 的设计通常让人觉得：

* 更直观
* 更容易和文件描述符、mmap、poll 体系融合
* 接口风格更统一

---

## 3. 为什么老的没被新的完全替代

因为现实里 API 一旦广泛使用，就很难彻底消失。

主要有几个原因。

### 第一，兼容性

大量老程序、老框架、老中间件已经在用 System V：

* 改动成本高
* 升级风险大
* 业务没动力重写

所以操作系统必须继续支持。

### 第二，功能上也不是完全重复得毫无差别

虽然都能做“共享内存”，但使用模型不完全一样：

* System V：`key + id`
* POSIX：`name + fd + mmap`

很多人会根据项目习惯选不同方案。

### 第三，Unix 世界本来就很重视向后兼容

Linux/Unix 很少因为“有了新接口”就把老接口彻底砍掉。

---

## 4. 两套的核心区别是什么

### System V 更像“内核登记对象”

你先拿一个 `key` 去系统里登记/查找，系统返回一个 `shmid`。

特点：

* 接口比较“古典”
* 生命周期和内核对象绑定得更明显
* 常见于老项目、教材、底层框架

### POSIX 更像“命名对象 + 文件映射”

你先 `shm_open` 得到一个 fd，再 `mmap`。

特点：

* 和文件描述符模型接近
* 更容易理解成“特殊共享文件”
* 和现代 Linux 编程习惯更一致

---

## 5. 为什么很多人更喜欢 POSIX

因为它通常更顺手。

比如新手看这两套：

### System V

```cpp
shmget(...)
shmat(...)
shmdt(...)
shmctl(...)
```

### POSIX

```cpp
shm_open(...)
ftruncate(...)
mmap(...)
munmap(...)
shm_unlink(...)
```

很多人会觉得 POSIX 这套更容易串起来，因为：

* `shm_open` 像 `open`
* `mmap` 是熟悉的内存映射
* `fd` 能融入很多已有抽象

所以**新项目常更倾向 POSIX**。

---

## 6. 那为什么还有人继续用 System V

因为它也有自己的工程惯性：

* 很多老代码已经稳定运行很多年
* 某些中间件、老组件就是基于它写的
* 某些团队对它非常熟悉
* 某些场景下直接 `key -> shmid` 也很直接

特别是在一些老式 Unix/Linux C 项目里，System V 非常常见。

---

## 7. 你可以把这理解成 C++ 世界里的类比

有点像：

* 老接口：一直能用，很多遗留项目都在用
* 新接口：设计更现代一些，写起来更顺手
* 系统为了兼容，两个都保留

不是因为其中一个“错了”，而是因为历史演化就是这样。

---

## 8. 实际开发怎么选

### 如果你在读老项目

大概率会碰到 System V：

* `shmget`
* `shmat`
* `semget`
* `msgget`

你需要会看。

### 如果你自己写新 Linux 项目

很多时候会优先考虑 POSIX：

* `shm_open + mmap`
* `sem_open`
* 再配合 `eventfd/epoll`

因为整体风格更统一。

---

## 9. 一句话总结

**之所以有 System V 和 POSIX 两套共享内存 API，是因为它们来自不同历史阶段的 Unix IPC 设计：System V 更老，POSIX 是后来的标准化接口；由于兼容性和大量存量代码的存在，两套都一直保留到了今天。**
