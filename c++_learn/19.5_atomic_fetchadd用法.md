`std::atomic` 里的 `fetch_add` 用来**原子地把当前值加上一个增量**。

最常见写法：

```cpp
std::atomic<int> cnt{0};

int old = cnt.fetch_add(1);
```

这句的意思是：

1. 以原子方式把 `cnt` 增加 1
2. `old` 得到的是 **加之前的旧值**

所以如果原来 `cnt == 0`，执行后：

* `old == 0`
* `cnt == 1`

---

## 基本语法

```cpp
obj.fetch_add(arg);
obj.fetch_add(arg, memory_order);
```

比如：

```cpp
std::atomic<int> x{10};

int a = x.fetch_add(3);   // a = 10, x = 13
int b = x.fetch_add(2);   // b = 13, x = 15
```

---

## 返回值要特别注意

很多人一开始会以为返回的是“加完后的值”，其实不是。

它返回的是：

> **修改前的值**

例如：

```cpp
std::atomic<int> id{100};

int old = id.fetch_add(1);
```

结果：

* `old == 100`
* `id == 101`

如果你想拿“加完后的值”，可以自己再加一下：

```cpp
int new_val = id.fetch_add(1) + 1;
```

---

## 和 `++` 有什么关系

这几个写法都可能用到：

```cpp
std::atomic<int> x{0};

++x;        // 原子前置++
x++;        // 原子后置++
x.fetch_add(1);
```

可以这样理解：

* `x.fetch_add(1)` 类似于 `x++`
* 都是“先返回旧值，再加 1”

例子：

```cpp
std::atomic<int> x{5};

int a = x.fetch_add(1);  // a = 5, x = 6
int b = x++;             // b = 6, x = 7
int c = ++x;             // x = 8, c = 8
```

---

## 多线程里为什么要用它

因为它是**原子操作**，多个线程同时加也不会丢。

### 错误写法

```cpp
int cnt = 0;

// 多线程同时 cnt++
```

普通 `cnt++` 不是原子的，可能发生竞争，结果不对。

### 正确写法

```cpp
std::atomic<int> cnt{0};

cnt.fetch_add(1);
```

---

## 典型例子：多线程计数器

```cpp
#include <atomic>
#include <thread>
#include <vector>
#include <iostream>

int main() {
    std::atomic<int> cnt{0};

    std::vector<std::thread> threads;
    for (int i = 0; i < 10; ++i) {
        threads.emplace_back([&]() {
            for (int j = 0; j < 1000; ++j) {
                cnt.fetch_add(1, std::memory_order_relaxed);
            }
        });
    }

    for (auto& t : threads) {
        t.join();
    }

    std::cout << cnt.load() << "\n";  // 10000
}
```

这里常用 `memory_order_relaxed`，因为只是做计数，不需要额外同步别的数据。

## memory_order 怎么选

### 1. 默认写法

```cpp
x.fetch_add(1);
```

默认是：

```cpp
std::memory_order_seq_cst
```

最强、最容易理解，但有时性能不是最优。

---

### 2. 只做计数：`relaxed`

如果只是统计次数、发号器、引用计数中的某些场景：

```cpp
x.fetch_add(1, std::memory_order_relaxed);
```

意思是：

* 加法本身仍然原子
* 但不额外提供复杂的线程同步顺序保证

---

### 3. 需要同步时

如果这个加法还承担“线程间发布/可见性”的作用，就不能随便用 `relaxed`，通常要结合：

* `memory_order_acquire`
* `memory_order_release`
* `memory_order_acq_rel`
* 或默认 `seq_cst`

这个就和你的并发设计有关了。

## 支持哪些类型

`fetch_add` 主要用于：

### 1. 整数原子类型

```cpp
std::atomic<int>
std::atomic<long>
std::atomic<unsigned>
```

### 2. 指针类型

```cpp
std::atomic<T*>
```

对指针做 `fetch_add(n)`，表示**按元素步长移动指针**。

例如：

```cpp
int arr[10];
std::atomic<int*> p{arr};

int* old = p.fetch_add(2);
```

结果：

* `old == &arr[0]`
* `p == &arr[2]`

注意不是加 2 字节，而是加 2 个 `int`。

---

## 指针例子

```cpp
#include <atomic>
#include <iostream>

int main() {
    int arr[5] = {10, 20, 30, 40, 50};
    std::atomic<int*> p{arr};

    int* old = p.fetch_add(2);

    std::cout << *old << "\n";        // 10
    std::cout << *(p.load()) << "\n"; // 30
}
```

## 常见用途

### 发号器

```cpp
std::atomic<int> next_id{0};

int id = next_id.fetch_add(1);
```

每个线程拿到一个唯一 id。

### 计数器

```cpp
counter.fetch_add(1, std::memory_order_relaxed);
```

### 工作分配

```cpp
std::atomic<int> idx{0};

int i = idx.fetch_add(1);
if (i < n) {
    // 处理第 i 个任务
}
```

## 常见误区

### 误区 1：以为返回新值

错。

```cpp
std::atomic<int> x{1};
int y = x.fetch_add(2);
```

结果是：

* `y == 1`
* `x == 3`

---

### 误区 2：原子就等于“什么线程问题都解决了”

不是。

`fetch_add` 只保证这个“加法”本身是原子的。
如果你还依赖它去同步别的共享数据，就要认真设计 memory order。

---

### 误区 3：普通变量和 atomic 混用

不要一会儿原子访问，一会儿非原子访问同一个对象，否则还是可能数据竞争。

---

## 一句话记忆

`fetch_add(n)` 可以记成：

> **“把当前原子值加 n，并返回加之前的旧值”**
