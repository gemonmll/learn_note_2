条件变量和锁一起用，核心是为了解决一类问题：

> 一个线程要“等某个条件成立”，
> 另一个线程在条件成立后“通知它”。

最经典的场景就是：

* 生产者放数据
* 消费者没数据时阻塞等待
* 有数据后被唤醒继续干活

---

# 1. 为什么条件变量一定要配锁

因为线程等待的不是“通知”本身，而是**某个共享条件**，比如：

* 队列是否非空
* 某个标志位是否变成 true
* 某个资源是否准备好了

这个“条件”是共享变量，所以必须保护它，避免竞态。

条件变量本身只负责：

* 让线程睡眠
* 在被通知时唤醒

真正要检查的是那个共享条件，而共享条件要靠 mutex 保护。

---

# 2. 最基本使用套路

记住这个模板就够了：

## 等待方

```cpp
std::unique_lock<std::mutex> lk(mtx);
cv.wait(lk, [] { return 条件成立; });
```

## 通知方

```cpp
{
    std::lock_guard<std::mutex> lk(mtx);
    修改共享条件;
}
cv.notify_one();
```

这就是最标准的写法。

---

# 3. 一个完整例子：一个线程等数据，一个线程放数据

```cpp
#include <iostream>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>

std::mutex mtx;
std::condition_variable cv;
std::queue<int> q;

void producer() {
    {
        std::lock_guard<std::mutex> lk(mtx);
        q.push(42);
        std::cout << "producer: pushed 42\n";
    }
    cv.notify_one();
}

void consumer() {
    std::unique_lock<std::mutex> lk(mtx);

    cv.wait(lk, [] {
        return !q.empty();
    });

    int value = q.front();
    q.pop();
    std::cout << "consumer: got " << value << "\n";
}

int main() {
    std::thread t1(consumer);
    std::thread t2(producer);

    t1.join();
    t2.join();
}
```

---

# 4. 这个例子里每一步在干什么

## 消费者线程

```cpp
std::unique_lock<std::mutex> lk(mtx);
```

先拿锁，准备检查共享数据 `q`。

---

```cpp
cv.wait(lk, [] {
    return !q.empty();
});
```

这句最关键，含义是：

* 如果 `q` 不空，直接继续
* 如果 `q` 空，就：

  * 自动释放 `lk`
  * 让线程睡眠
* 当收到通知后：

  * 再自动重新加锁
  * 再次检查 `!q.empty()`
* 只有条件真成立，才返回

---

## 生产者线程

```cpp
{
    std::lock_guard<std::mutex> lk(mtx);
    q.push(42);
}
cv.notify_one();
```

含义是：

1. 先加锁
2. 修改共享数据 `q`
3. 解锁
4. 通知等待线程

---

# 5. 为什么 `wait` 要传 `unique_lock`，不能传 `lock_guard`

因为 `wait()` 在等待期间要做两件事：

1. **释放锁**
2. 醒来后**重新加锁**

而 `lock_guard` 不能手动解锁/重新加锁，功能太简单。

`unique_lock` 可以，所以 `condition_variable` 要求传它。

---

# 6. `wait()` 到底做了什么

这句：

```cpp
cv.wait(lk, pred);
```

大致等价于：

```cpp
while (!pred()) {
    cv.wait(lk);
}
```

而 `cv.wait(lk)` 本身又做了这些事：

1. 检查前提：当前已经持有锁
2. 原子地释放锁并进入睡眠
3. 被唤醒后重新拿锁
4. 返回给调用者

所以你可以理解成：

> `wait()` 不是一直抱着锁睡觉，
> 它是“睡觉时放锁，醒了再抢回来”。

这很重要，否则别的线程根本没法加锁修改条件。

---

# 7. 为什么推荐用带谓词的 `wait`

推荐这样写：

```cpp
cv.wait(lk, [] { return ready; });
```

而不是只写：

```cpp
cv.wait(lk);
```

原因有两个：

## ① 防止虚假唤醒

线程可能在**没人 notify** 的情况下也醒来。

所以醒来后不能直接往下走，必须重新检查条件。

---

## ② 防止“通知先到，等待后到”

比如：

1. 线程 B 先把 `ready = true`
2. 然后 `notify_one()`
3. 线程 A 之后才开始 wait

如果你只等通知，A 可能永远睡住。
但如果你检查的是“条件 ready 是否成立”，那 A 发现已经 true，就根本不会睡。

所以：

> 条件变量等的是“条件”，不是“通知”。

这是理解条件变量的关键。

---

# 8. 一个最常见错误写法

```cpp
cv.wait(lk);
```

然后醒来直接操作共享数据：

```cpp
int value = q.front();   // 可能错
```

问题是：

* 可能虚假唤醒
* 也可能多个消费者一起被唤醒，但只有一个拿到数据

所以必须写成：

```cpp
cv.wait(lk, [] { return !q.empty(); });
```

或者：

```cpp
while (q.empty()) {
    cv.wait(lk);
}
```

---

# 9. `notify_one()` 和 `notify_all()` 区别

## `notify_one()`

只唤醒一个等待线程

适合：

* 一个资源只需要一个线程处理
* 生产者/消费者模型

## `notify_all()`

唤醒所有等待线程

适合：

* 状态变化会影响所有等待者
* 比如“系统开始退出了”“配置加载完成了”

---

# 10. 再看一个布尔标志例子

```cpp
#include <iostream>
#include <thread>
#include <mutex>
#include <condition_variable>

std::mutex mtx;
std::condition_variable cv;
bool ready = false;

void worker() {
    std::unique_lock<std::mutex> lk(mtx);
    cv.wait(lk, [] {
        return ready;
    });

    std::cout << "worker: ready is true, start working\n";
}

int main() {
    std::thread t(worker);

    {
        std::lock_guard<std::mutex> lk(mtx);
        ready = true;
    }
    cv.notify_one();

    t.join();
}
```

这个例子特别适合理解条件变量的本质：

* `ready` 是条件
* `cv` 是等待/通知机制
* `mtx` 是保护 `ready` 的锁

---

# 11. `wait_for` 怎么用

如果不想无限等，可以用超时版：

```cpp
std::unique_lock<std::mutex> lk(mtx);
bool ok = cv.wait_for(lk, std::chrono::seconds(1), [] {
    return ready;
});
```

意思是：

* 最多等 1 秒
* 如果这期间 `ready` 变 true，就返回 `true`
* 超时还没等到，就返回 `false`

这就是你前面看到 Apollo 代码里 `wait_for` 的原因：既能阻塞等待，又能周期性醒来检查状态。

---

# 12. 使用顺序最好怎么记

你可以直接背这个流程。

## 等待线程

1. 加锁
2. 检查条件，不满足就 wait
3. wait 时自动释放锁
4. 被唤醒后自动重新加锁
5. 再次检查条件
6. 条件满足后继续执行

## 通知线程

1. 加锁
2. 修改共享条件
3. 解锁
4. notify_one / notify_all

---

# 13. 为什么通常“先改条件，再通知”

正确顺序通常是：

```cpp
{
    std::lock_guard<std::mutex> lk(mtx);
    ready = true;
}
cv.notify_one();
```

因为等待线程判断的依据是 `ready`。

如果先通知、后改条件，就可能出现：

* 对方醒了
* 抢到锁一看，条件还没变
* 又睡回去

虽然有些场景也能工作，但规范写法还是：

> **先修改条件，再通知。**

---

# 14. 一个简化心智模型

把条件变量想成“门铃”，锁想成“门”。

* 锁保护门内状态
* 条件变量负责让人睡觉/叫醒
* 被叫醒后，必须重新进门看状态是不是真的变了

所以：

* **mutex**：保护共享状态
* **condition_variable**：协调等待和唤醒
* **predicate/条件**：真正决定能不能继续执行

---

# 15. 最后给你一个万能模板

```cpp
std::mutex mtx;
std::condition_variable cv;
bool ready = false;

// 等待方
void wait_thread() {
    std::unique_lock<std::mutex> lk(mtx);
    cv.wait(lk, [] {
        return ready;
    });

    // 条件满足后执行
}

// 通知方
void notify_thread() {
    {
        std::lock_guard<std::mutex> lk(mtx);
        ready = true;
    }
    cv.notify_one();
}
```

# 16. 一句话总结

**条件变量负责“睡眠/唤醒”，锁负责“保护条件”，真正等待的是共享条件本身，所以三者必须一起配合用。**

## 问题 uniquelock和lockguard区别

`std::lock_guard` 和 `std::unique_lock` 都是 **RAII 风格的加锁对象**，作用都是：

> 进入作用域时加锁，离开作用域时自动解锁。

但两者能力不一样。

---

# 1. 先说结论

## `lock_guard`

* 轻量
* 简单
* 只能“拿着这把锁直到作用域结束”
* 适合最普通的互斥保护

## `unique_lock`

* 更灵活
* 功能更多
* 可以手动 `unlock()` / `lock()`
* 可以延迟加锁、尝试加锁、转移锁所有权
* `condition_variable` 必须配它用

---

# 2. `lock_guard` 的用法

最常见：

```cpp
#include <mutex>

std::mutex mtx;
int count = 0;

void f() {
    std::lock_guard<std::mutex> lk(mtx);
    ++count;
}
```

意思是：

* 构造 `lk` 时自动加锁
* `f()` 结束时自动解锁

它非常适合这种场景：

> 我只想进来加锁，出去解锁，中间不要搞别的。

---

# 3. `unique_lock` 的用法

```cpp
#include <mutex>

std::mutex mtx;
int count = 0;

void f() {
    std::unique_lock<std::mutex> lk(mtx);
    ++count;
}
```

这也能正常工作，和 `lock_guard` 表面上差不多。

但它还能做更多事。

---

# 4. 最大区别：`unique_lock` 可以手动控制锁

例如：

```cpp
std::unique_lock<std::mutex> lk(mtx);
++count;
lk.unlock();

// 这里不再持有锁

lk.lock();
++count;
```

而 `lock_guard` 不行，它没有 `unlock()`。

所以：

## `lock_guard`

像一次性封条，贴上就到作用域结束

## `unique_lock`

像一把可操作的钥匙，可以开、关、再开

---

# 5. 为什么条件变量要用 `unique_lock`

你前面问过 `condition_variable`。

标准写法是：

```cpp
std::unique_lock<std::mutex> lk(mtx);
cv.wait(lk, [] { return ready; });
```

不能写成：

```cpp
std::lock_guard<std::mutex> lk(mtx);
cv.wait(lk);   // 不行
```

因为 `wait()` 在等待时要：

1. 先释放锁
2. 睡眠
3. 被唤醒后再重新加锁

`lock_guard` 做不到“中途解锁再加锁”，
`unique_lock` 可以，所以条件变量接口要求它。

---

# 6. `unique_lock` 支持延迟加锁

```cpp
std::unique_lock<std::mutex> lk(mtx, std::defer_lock);

// 此时还没加锁

lk.lock();
```

这在你想自己控制加锁时机时很有用。

`lock_guard` 没这个能力。

---

# 7. `unique_lock` 支持尝试加锁

```cpp
std::unique_lock<std::mutex> lk(mtx, std::try_to_lock);

if (lk.owns_lock()) {
    // 成功拿到锁
} else {
    // 没拿到
}
```

`lock_guard` 不支持。

---

# 8. `unique_lock` 支持所有权转移

```cpp
std::unique_lock<std::mutex> lk1(mtx);
std::unique_lock<std::mutex> lk2 = std::move(lk1);
```

锁的所有权可以 move 给别人。

`lock_guard` 基本就是一个很固定的作用域对象，不玩这个。

---

# 9. 性能上怎么选

通常：

* `lock_guard` 更轻、更直接
* `unique_lock` 稍重一点，因为它要保存更多状态

所以如果你只是这种场景：

```cpp
{
    std::lock_guard<std::mutex> lk(mtx);
    // 临界区
}
```

优先用 `lock_guard` 就很好。

如果你需要：

* 条件变量
* 手动解锁
* 延迟加锁
* 尝试加锁
* move 锁对象

就用 `unique_lock`。

---

# 10. 最常见选择原则

## 用 `lock_guard` 的场景

“我只想简单保护一段代码”

```cpp
void add() {
    std::lock_guard<std::mutex> lk(mtx);
    ++count;
}
```

---

## 用 `unique_lock` 的场景

“我需要更灵活地操作锁”

```cpp
void worker() {
    std::unique_lock<std::mutex> lk(mtx);
    cv.wait(lk, [] { return ready; });
}
```

或者：

```cpp
std::unique_lock<std::mutex> lk(mtx);
do_something();
lk.unlock();
do_other_thing();
```

---

# 11. 一个对比表

## `std::lock_guard`

* 自动加锁：有
* 自动解锁：有
* 手动 `unlock()`：没有
* 手动 `lock()`：没有
* `condition_variable`：不能配合
* 延迟加锁：不支持
* try_lock：不支持
* move：不常用/不支持这种灵活语义
* 使用成本：更低

## `std::unique_lock`

* 自动加锁：有
* 自动解锁：有
* 手动 `unlock()`：有
* 手动 `lock()`：有
* `condition_variable`：可以
* 延迟加锁：支持
* try_lock：支持
* move：支持
* 使用成本：略高

---

# 12. 一句话记忆

你可以这样记：

> **`lock_guard` = 简单保险栓**
> **`unique_lock` = 功能完整的锁控制器**

# 13. 实战建议

平时写代码时可以这样选：

* 默认先想 `lock_guard`
* 一旦需要更灵活的控制，就换成 `unique_lock`

也就是：

> **能用 `lock_guard` 就用它；不够用了再上 `unique_lock`。**
