可以，我用**从不会到会用**的方式带你学 `poll` 和 `epoll`。你前面已经在学 Linux IPC、共享内存、eventfd、pipe，这一块正好是同一个体系里的核心内容。

---

# 1. 先理解：poll / epoll 到底是干什么的

它们本质上都是：

> **让你一次等待很多文件描述符(fd)上的事件，而不是一个一个等。**

比如你有这些 fd：

* socket
* pipe
* eventfd
* timerfd
* 串口
* 某些设备文件

你想知道：

* 哪个 fd 可读了？
* 哪个 fd 可写了？
* 哪个 fd 出错了？
* 有没有对端关闭？

如果你不用 `poll/epoll`，就只能：

* 要么一个一个轮询，CPU 浪费大
* 要么一个线程盯一个 fd，线程太多
* 要么阻塞在一个 fd 上，别的 fd 不能同时处理

所以它们就是 **I/O 多路复用**。

---

# 2. 先学 poll，因为它更直观

## 2.1 poll 的基本思想

`poll` 让你传进去一个 fd 数组，然后内核帮你等：

```c
int poll(struct pollfd *fds, nfds_t nfds, int timeout);
```

结构体：

```c
struct pollfd {
    int   fd;         // 要监听的 fd
    short events;     // 关心什么事件
    short revents;    // 实际发生了什么事件（内核填）
};
```

最常见的事件：

* `POLLIN`：可读
* `POLLOUT`：可写
* `POLLERR`：错误
* `POLLHUP`：挂断，对端关闭
* `POLLNVAL`：fd 无效

---

## 2.2 poll 怎么理解

你可以把它想成：

> 我把一堆 fd 的“监听清单”交给内核，内核帮我睡眠等待。
> 有事件发生了，就返回，我再看是哪几个 fd 就绪了。

---

## 2.3 一个最小例子：监听标准输入

```cpp
#include <poll.h>
#include <unistd.h>
#include <iostream>

int main() {
    struct pollfd pfd;
    pfd.fd = STDIN_FILENO;   // 监听标准输入
    pfd.events = POLLIN;

    while (true) {
        int ret = poll(&pfd, 1, 5000);  // 等 5 秒
        if (ret < 0) {
            perror("poll");
            return 1;
        } else if (ret == 0) {
            std::cout << "timeout\n";
        } else {
            if (pfd.revents & POLLIN) {
                char buf[128] = {0};
                ssize_t n = read(STDIN_FILENO, buf, sizeof(buf) - 1);
                if (n > 0) {
                    std::cout << "read: " << buf;
                }
            }
        }
    }
}
```

---

## 2.4 poll 的工作流程

### 第一步：准备监听数组

```cpp
struct pollfd fds[2];
fds[0].fd = fd1;
fds[0].events = POLLIN;

fds[1].fd = fd2;
fds[1].events = POLLIN;
```

### 第二步：调用 poll

```cpp
int ret = poll(fds, 2, 1000);
```

### 第三步：检查 revents

```cpp
if (fds[0].revents & POLLIN) {
    // fd1 可读
}
if (fds[1].revents & POLLIN) {
    // fd2 可读
}
```

# 3. poll 的优缺点

## 优点

* 比 `select` 更好用
* 不受 `FD_SETSIZE` 那种位图限制
* 接口不复杂，容易理解

## 缺点

### 1）每次调用都要把整个 fd 数组传给内核

你有 10000 个 fd，即使只有 1 个活跃，也要每次都遍历那 10000 个。

### 2）返回后你还得自己遍历数组找哪个就绪

内核不会直接告诉你“第几个就绪”，你还得扫描。

### 3）fd 多时效率差

所以大规模网络服务器通常不会用 `poll`，而会用 `epoll`。

---

# 4. epoll 是为了解决 poll 的效率问题

`epoll` 可以理解成：

> 先把“要监听哪些 fd”注册到内核里，之后等待时不需要每次重复传整张表。
> 哪些 fd 就绪了，内核直接告诉你。

这就是它高效的原因。

---

# 5. epoll 的三个核心函数

## 5.1 创建 epoll 对象

```c
int epoll_create1(int flags);
```

例如：

```cpp
int epfd = epoll_create1(0);
```

这个 `epfd` 本身也是一个 fd，表示一个 epoll 实例。

---

## 5.2 注册 / 修改 / 删除监听项

```c
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
```

`op` 有三种：

* `EPOLL_CTL_ADD`：添加
* `EPOLL_CTL_MOD`：修改
* `EPOLL_CTL_DEL`：删除

---

## 5.3 等待事件

```c
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);
```

它会把已经就绪的事件直接放进 `events` 数组里。

---

# 6. 一个最小 epoll 例子

监听标准输入：

```cpp
#include <sys/epoll.h>
#include <unistd.h>
#include <iostream>
#include <cstring>

int main() {
    int epfd = epoll_create1(0);
    if (epfd < 0) {
        perror("epoll_create1");
        return 1;
    }

    epoll_event ev;
    std::memset(&ev, 0, sizeof(ev));
    ev.events = EPOLLIN;
    ev.data.fd = STDIN_FILENO;

    if (epoll_ctl(epfd, EPOLL_CTL_ADD, STDIN_FILENO, &ev) < 0) {
        perror("epoll_ctl add");
        return 1;
    }

    epoll_event events[10];

    while (true) {
        int n = epoll_wait(epfd, events, 10, 5000);
        if (n < 0) {
            perror("epoll_wait");
            return 1;
        } else if (n == 0) {
            std::cout << "timeout\n";
        } else {
            for (int i = 0; i < n; ++i) {
                if (events[i].data.fd == STDIN_FILENO &&
                    (events[i].events & EPOLLIN)) {
                    char buf[128] = {0};
                    ssize_t r = read(STDIN_FILENO, buf, sizeof(buf) - 1);
                    if (r > 0) {
                        std::cout << "read: " << buf;
                    }
                }
            }
        }
    }

    close(epfd);
    return 0;
}
```

# 7. poll 和 epoll 最核心的区别

## poll

每次都要：

* 把所有 fd 传给内核
* 内核检查所有 fd
* 返回后用户再扫一遍所有 fd

所以像：

> “每次考试都把全班名单重新交一遍，再挨个点名”

---

## epoll

先注册一次，之后：

* 内核长期记住你关心哪些 fd
* 事件发生时，只把就绪的 fd 返回给你

所以像：

> “先把班级名单录进系统，以后只通知你今天到场的人”

---

# 8. 为什么 epoll 更快

不是因为它神奇，而是因为它减少了两件事：

## 8.1 不用每次复制大数组

`poll` 每次传 `pollfd[]`
`epoll` 注册一次后，不用反复传

## 8.2 不用每次全量扫描

`poll`：你和内核都要扫全部 fd
`epoll`：直接给你“已经就绪的 fd 列表”

所以当 fd 数量很大时，差距明显。

---

# 9. epoll 的两种触发方式：最重要的点之一

这块必须搞懂。

## 9.1 水平触发 LT（Level Triggered）

默认模式。

意思是：

> 只要 fd 仍然处于“可读/可写”状态，`epoll_wait` 每次都会继续通知你。

例如 pipe 里有 100 字节，你只读了 20 字节，还剩 80 字节。
那么下次 `epoll_wait` 还会继续告诉你：这个 fd 可读。

### 特点

* 好理解
* 不容易写错
* 默认就是它

---

## 9.2 边沿触发 ET（Edge Triggered）

通过 `EPOLLET` 开启。

意思是：

> 只在“状态发生变化”的瞬间通知一次。

例如：

* pipe 从“空”变成“非空”时通知一次
* 你如果没把数据一次性读完，后面可能不再提醒你

所以 ET 模式下通常必须：

* 把 fd 设成非阻塞
* 一直 `read` 到返回 `EAGAIN`

---

## 9.3 ET 为什么容易出错

假设管道里来了 100 字节：

1. `epoll_wait` 通知一次
2. 你只读了 20 字节
3. 还剩 80 字节
4. 因为状态没有再次变化，可能不会再通知

结果你以为没数据了，实际上数据还在那。

---

# 10. LT 和 ET 的直观比喻

## LT

像闹钟：

> 只要门还开着，就一直提醒你“门没关”。

## ET

像门铃：

> 只在门从关变开那一瞬间响一次。
> 你没处理干净，后面不一定再响。

---

# 11. 为什么 ET 通常要配合非阻塞

因为你要“读到不能再读为止”。

例如：

```cpp
while (true) {
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n > 0) {
        // 处理数据
    } else if (n == -1 && errno == EAGAIN) {
        // 已经读空了
        break;
    } else if (n == 0) {
        // 对端关闭
        break;
    } else {
        // 其他错误
        break;
    }
}
```

如果不用非阻塞，你可能在循环里卡死。

---

# 12. epoll 常见事件

常见的有：

* `EPOLLIN`：可读
* `EPOLLOUT`：可写
* `EPOLLERR`：错误
* `EPOLLHUP`：挂断
* `EPOLLRDHUP`：对端关闭写端，socket 常用
* `EPOLLET`：边沿触发
* `EPOLLONESHOT`：触发一次后自动失效，需手动重新 arm

---

# 13. poll / epoll 能监听什么

只要是支持“等待事件”的 fd，一般都可以：

* pipe
* socket
* eventfd
* timerfd
* signalfd
* 某些字符设备

你之前问过：

* `eventfd`
* `pipe`
* 是否能阻塞等待

答案就是：

> 非常适合配合 `poll/epoll` 用。

这也是 Linux 事件驱动编程的核心。

---

# 14. pipe + poll 的理解

比如父子进程通信：

* 子进程往 pipe 写数据
* 父进程用 `poll` 等 pipe 变可读
* 一旦 `POLLIN`，父进程再去 `read`

这样就不用自己写死循环：

```cpp
while (1) {
    // 不停 read / 检查
}
```

而是：

> 没数据就睡眠，有数据再醒来

---

# 15. eventfd + epoll 非常经典

`eventfd` 是一个**事件计数器 fd**。
线程或进程 A 往里面 write 一个 8 字节整数，线程或进程 B 就能通过 `epoll_wait` 被唤醒。

这个模式常用于：

* 线程池唤醒
* reactor 线程通知
* 跨线程退出通知
* 代替某些 pipe 唤醒场景

---

# 16. 一个 eventfd + epoll 的直觉模型

你可以把 `eventfd` 想成：

> 一个可以被 `epoll` 监听的“信号门铃”

别人一按铃（write），你这边 `epoll_wait` 就醒。

这比共享内存一直轮询强很多，因为：

* 不忙等
* 不浪费 CPU
* 事件到来再处理

---

# 17. poll 和 epoll 是否会阻塞

会。

## poll

```cpp
poll(fds, nfds, timeout);
```

* `timeout > 0`：最多阻塞这么久
* `timeout = 0`：立即返回，不阻塞
* `timeout = -1`：一直阻塞

## epoll_wait

```cpp
epoll_wait(epfd, events, maxevents, timeout);
```

规则一样。

所以你之前问“`poll/epoll` 能不能阻塞”，答案是：

> 它们最常见的用法就是阻塞等待。

---

# 18. 它们和“回调”是什么关系

你之前还问过 RTPS 接收回调、本质像不像 socket 接收通知。

在 Linux 用户态里，`poll/epoll` 本身**不是回调机制**，而是：

> 你主动调用 `epoll_wait` 阻塞等待，返回后自己处理。

所以更准确说是：

* 内核负责事件就绪检测
* 用户线程通过 `epoll_wait` 被唤醒
* 醒来后你自己调用对应处理函数

很多框架把这一步再封装成“回调”，例如：

```cpp
if (events[i].data.fd == sockfd) {
    on_socket_readable(sockfd);
}
```

所以**回调是你在用户层封装出来的**，不是 epoll 内核直接回调你。

---

# 19. 一个典型 reactor 模型

很多网络库本质上是这个结构：

```cpp
while (true) {
    int n = epoll_wait(epfd, events, MAX_EVENTS, -1);
    for (int i = 0; i < n; ++i) {
        int fd = events[i].data.fd;
        if (fd == listenfd) {
            accept_new_connection();
        } else if (events[i].events & EPOLLIN) {
            handle_read(fd);
        } else if (events[i].events & EPOLLOUT) {
            handle_write(fd);
        }
    }
}
```

这就是：

* 一个线程
* 管很多连接
* 谁就绪就处理谁

nginx、redis、很多中间件都是这类思想。

# 20. poll 和 epoll 怎么选

## 小规模、教学、简单程序

可以先用 `poll`

因为：

* 接口直接
* 逻辑简单
* 易于理解

## fd 数量大、长期运行、高性能服务

优先 `epoll`

因为：

* 可扩展性更好
* 大量连接时效率高
* Linux 服务端编程主流方案

# 21. 学习时最容易混淆的几个点

## 1）“可读”不等于“已经帮你读了”

`POLLIN` / `EPOLLIN` 只是告诉你：

> 现在读不会卡住，或者已经有数据/状态可读

你还是要自己 `read()`。

---

## 2）“可写”不等于“你应该一直写”

`POLLOUT` / `EPOLLOUT` 只说明当前发送缓冲区有空间。
很多程序平时不关心可写，只在发送缓冲区满时才监听 `EPOLLOUT`。

---

## 3）ET 模式不是“更高级就一定更好”

很多人一学 epoll 就想直接上 ET。
实际上：

* LT 更稳
* ET 更容易丢处理机会
* ET 对非阻塞和循环读写要求更严格

学习顺序建议：

> 先把 LT 用熟，再学 ET

---

## 4）epoll 不是“替代 read/write”

它只负责等事件，不负责真正 I/O。

---

# 23. 给你一个简洁对比表

| 项目                 | poll        | epoll        |
| ------------------ | ----------- | ------------ |
| 监听集合存放位置           | 用户态数组，每次传入  | 内核维护         |
| 每次 wait 是否重复传所有 fd | 是           | 否            |
| 返回后是否要扫全部 fd       | 通常要         | 不用扫全部，只拿到就绪项 |
| 适合 fd 数量           | 小中等         | 大量           |
| 接口复杂度              | 简单          | 稍复杂          |
| Linux 特性           | 通用 POSIX 风格 | Linux 特有     |
| 是否支持 LT/ET         | 没有 ET 概念    | 支持 LT / ET   |

# 25. 最后给你一个非常实用的理解框架

以后你看到 `poll/epoll`，脑子里就按这四步想：

## 第 1 问：我在等谁？

等一组 fd。

## 第 2 问：等什么？

等可读、可写、错误、关闭。

## 第 3 问：等到后干嘛？

自己 `read/write/accept`。

## 第 4 问：为什么不用死循环轮询？

因为可以阻塞睡眠，省 CPU。
