[plugin1.cc](cyber/class_loader/test/plugin1.cc) CLASS_LOADER_REGISTER_CLASS(Circle, Base);z这个自动注册机制我有些疑问，插件中注册了Circle, Base这个模版，在动态库加载时，这个模板是如何注册进入全局map中的，这是模块在内存中的位置是怎样的

核心点先说结论：`CLASS_LOADER_REGISTER_CLASS(Circle, Base)` 注册的不是“模板本身”，而是一个静态注册对象。这个静态对象在插件 `.so` 被 `dlopen` 时构造，构造函数里调用 `RegisterClass<Circle, Base>()`，再把一个 `ClassFactory<Circle, Base>` 指针塞进进程级全局表。

**流程**
你这行注册点在 [plugin1.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/test/plugin1.cc#L42)，宏定义在 [class_loader_register_macro.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/class_loader_register_macro.h#L22)。它可以近似理解成：

```cpp
namespace {
struct Proxy {
  Proxy() {
    RegisterClass<Circle, Base>("Circle", "Base");
  }
};
static Proxy g_register_class_xxx;
}
```

真正的加载链路是：
1. [class_loader_manager.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/class_loader_manager.cc#L56) 创建 `ClassLoader`。
2. [class_loader.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/class_loader.cc#L34) 进入 `utility::LoadLibrary()`。
3. [class_loader_utility.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.cc#L212) 和 [class_loader_utility.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.cc#L214) 先把“当前 loader”和“当前库路径”写到静态变量里，然后通过 [shared_library.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/shared_library/shared_library.cc#L42) 调 `dlopen(path, RTLD_LAZY | RTLD_GLOBAL)`。
4. `dlopen` 把插件映射进进程地址空间后，会执行该 `.so` 的静态初始化代码，于是 `g_register_class_xxx` 开始构造，进入 [class_loader_utility.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.h#L77) 的 `RegisterClass()`。
5. `RegisterClass()` 做三件事：
   - `new ClassFactory<Circle, Base>`，[class_loader_utility.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.h#L82)
   - 记录 owner loader 和所属库路径，[class_loader_utility.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.h#L84)
   - 插入全局表，[class_loader_utility.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.h#L87)

这个“全局 map”实际是两层：
- 外层 `BaseToClassFactoryMapMap`，key 是 `typeid(Base).name()`，定义见 [class_loader_utility.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.h#L47)
- 内层 `ClassClassFactoryMap`，key 是 `"Circle"`，value 是 `AbstractClassFactoryBase*`

注意一个细节：外层 key 不是字符串 `"Base"`，而是 `typeid(Base).name()`；`"Base"` 这个字符串更多是工厂对象里的元数据。

**内存**
内存里大致是这样：

- 插件 `.so` 自己会被映射成若干段：`.text/.rodata/.data/.bss`
- 宏生成的静态对象 `g_register_class_xxx` 属于这个插件的静态存储区，通常在 `.data`/`.bss`
- 触发它构造的入口在 ELF 的 `.init_array`，`dlopen` 时执行
- 真正的全局注册表不是在插件里，而是在 [class_loader_utility.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.cc#L34) 那些函数级静态对象里，进程生命周期内一直存在
- `ClassFactory<Circle, Base>` 本体是 `new` 出来的，在堆上；`std::map` 的节点和字符串也在堆上

所以更准确地说，注册过程不是“把模板放进 map”，而是“插件加载时执行静态构造，把一个堆上的工厂对象指针挂到进程级 registry 里”。

卸载时则反过来走：[class_loader_utility.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.cc#L250) 会先把这个 loader 从工厂对象 owner 列表里移掉；如果没人再持有，就从 map 里删掉并 `delete` 工厂对象，最后 `dlclose`。

## 问题2 第二部分 
```cpp
dlopen 把插件映射进进程地址空间后，会执行该 .so 的静态初始化代码，于是 g_register_class_xxx 开始构造，进入 class_loader_utility.h (line 77) 的 RegisterClass()。
RegisterClass() 做三件事：new ClassFactory<Circle, Base> 在调用RegisterClass时 主程序不知道有 circle这部分是怎么能够执行的
```

关键不是“主程序调用模板时知道 `Circle`”，而是：

`Circle` 相关的模板实例化和机器码，早就在插件编译阶段生成进 `.so` 里了。运行时执行的不是主程序那边“现想出来”的代码，而是插件自己带来的初始化代码。

可以拆成两层看。

**1. 编译期**
在 [plugin1.cc](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/test/plugin1.cc#L42) 里，编译器同时看到了 `Circle` 和宏 `CLASS_LOADER_REGISTER_CLASS(Circle, Base)`。

因为 `RegisterClass` 是头文件模板，定义在 [class_loader_utility.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.h#L77)，所以编译插件时就会在插件内部实例化出大致这样的代码：

```cpp
void RegisterClass_Circle_Base() {
  auto* f = new ClassFactory<Circle, Base>("Circle", "Base");
  ...
  global_map[typeid(Base).name()]["Circle"] = f;
}
```

`ClassFactory<Circle, Base>` 也不是运行时临时生成的，它在编译插件时就已经变成具体类了，核心能力类似：

```cpp
struct ClassFactory<Circle, Base> : AbstractClassFactory<Base> {
  Base* CreateObj() const override {
    return new Circle;
  }
};
```

所以，`new Circle` 这段代码其实也早就编进插件 `.so` 了。

**2. 运行期**
宏在 [class_loader_register_macro.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/class_loader_register_macro.h#L22) 里会生成一个静态对象 `g_register_class_xxx`。

`dlopen` 加载插件后，会执行这个静态对象的构造函数。这个构造函数属于插件自己的代码段，不是主程序源码里显式写的调用。它再去调用刚才已经实例化好的 `RegisterClass<Circle, Base>()`。

所以准确说是：

- 主程序只负责 `dlopen`
- 插件加载后，插件自己的静态初始化代码开始跑
- 这段插件代码知道 `Circle`
- 它把一个 `ClassFactory<Circle, Base>` 挂进全局注册表

主程序根本不需要包含 `Circle` 的定义，它只需要知道 `Base` 和工厂基类接口。

**为什么后面还能创建出 `Circle`**
后面主程序调用 [class_loader_utility.h](/home/jzm/workspace/apollo_learn/apollo-11.0.0/cyber/class_loader/utility/class_loader_utility.h#L95) 的 `CreateClassObj<Base>("Circle", loader)` 时：

- 先按字符串 `"Circle"` 从 map 里取出工厂指针
- map 里存的是 `AbstractClassFactoryBase*` / `AbstractClassFactory<Base>*`
- 然后走虚函数 `CreateObj()`
- 实际落到插件里的 `ClassFactory<Circle, Base>::CreateObj()`
- 那个函数里执行 `new Circle`
- 返回给主程序的是 `Base*`

这就是典型的“字符串查找 + 基类接口 + 虚函数分发”的插件工厂模式。  
主程序不知道 `Circle` 的具体类型，但它知道“这里有个能产出 `Base*` 的工厂”。
