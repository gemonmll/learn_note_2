## 基础用法
你这里说的 `feature` 应该是 `future`。
`std::promise` 和 `std::future` 是一对，用来做**线程之间的一次性结果传递**。

你可以先把它理解成：

* `promise`：生产者，负责“以后把结果塞进去”
* `future`：消费者，负责“将来把结果取出来”

它们像一个一次性的“取件码 + 包裹箱”。

---

# 1. 最基本概念

## `std::promise<T>`

表示“我承诺将来会提供一个 `T` 类型的结果”。

常见操作：

```cpp
std::promise<int> p;
```

然后生产者线程里：

```cpp
p.set_value(123);
```

或者发生异常时：

```cpp
p.set_exception(std::current_exception());
```

---

## `std::future<T>`

表示“我将来会拿到一个 `T` 类型的结果”。

从 `promise` 拿到：

```cpp
std::future<int> f = p.get_future();
```

然后消费者可以：

```cpp
int x = f.get();
```

`get()` 会阻塞，直到结果准备好。

---

# 2. 最小例子

这是最值得先吃透的版本：

```cpp
#include <future>
#include <iostream>
#include <thread>

int main() {
    std::promise<int> p;
    std::future<int> f = p.get_future();

    std::thread t([&p]() {
        p.set_value(42);
    });

    int result = f.get();
    std::cout << result << std::endl;  // 42

    t.join();
    return 0;
}
```

## 过程怎么理解

1. 主线程创建 `promise<int>`
2. 从它拿到一个 `future<int>`
3. 子线程里调用 `set_value(42)`
4. 主线程调用 `f.get()` 等结果
5. 一旦子线程设置了结果，`get()` 返回 42

---

# 3. 它适合解决什么问题

最典型场景：

你开了一个线程做事，但结果不是立刻有。
你希望：

* 子线程做完后把结果交回来
* 主线程可以等待这个结果

这时就很适合 `promise + future`。

---

# 4. `get()` 的特点

```cpp
int x = f.get();
```

`get()` 有两个重要特点：

## 第一，会阻塞

如果结果还没准备好，就等。

## 第二，只能调用一次

因为 `future` 是“一次性取值”。

例如：

```cpp
int x1 = f.get();
int x2 = f.get();  // 错，future 已经被取空了
```

第二次通常会抛异常或出错。

---

# 5. `wait()` / `wait_for()` / `wait_until()`

有时你不想立刻取值，只想先等它就绪。

## `wait()`

```cpp
f.wait();
```

一直等到结果就绪。

## `wait_for()`

```cpp
auto status = f.wait_for(std::chrono::seconds(1));
```

等一段时间，看准备好了没有。

例如：

```cpp
if (f.wait_for(std::chrono::seconds(1)) == std::future_status::ready) {
    std::cout << f.get() << std::endl;
} else {
    std::cout << "timeout\n";
}
```

这个很适合做超时控制。

---

# 6. 异常怎么传

这是 `promise/future` 很实用的一点。
子线程里抛异常时，可以把异常传给主线程。

```cpp
#include <future>
#include <iostream>
#include <thread>
#include <stdexcept>

int main() {
    std::promise<int> p;
    std::future<int> f = p.get_future();

    std::thread t([&p]() {
        try {
            throw std::runtime_error("something wrong");
        } catch (...) {
            p.set_exception(std::current_exception());
        }
    });

    try {
        int result = f.get();
        std::cout << result << std::endl;
    } catch (const std::exception& e) {
        std::cout << "catch: " << e.what() << std::endl;
    }

    t.join();
}
```

这里：

* 子线程没 `set_value`
* 而是 `set_exception`
* 主线程 `f.get()` 时重新抛出那个异常

---

# 7. `promise<void>` 也很常用

如果你不需要返回值，只需要“通知任务完成”，可以用 `void`。

```cpp
#include <future>
#include <iostream>
#include <thread>

int main() {
    std::promise<void> p;
    std::future<void> f = p.get_future();

    std::thread t([&p]() {
        std::cout << "worker done\n";
        p.set_value();
    });

    f.wait();
    std::cout << "main knows worker finished\n";

    t.join();
}
```

这里 `set_value()` 不带参数，表示“完成信号到了”。

---

# 8. `std::async` 和它们的关系

你以后会看到：

```cpp
auto f = std::async(std::launch::async, []() {
    return 123;
});
```

这也会返回 `std::future<int>`。

区别是：

* `std::promise`：你手动控制什么时候设置结果
* `std::async`：标准库帮你创建异步任务并返回 future

所以可以理解成：

* `async` 更方便
* `promise` 更底层、更灵活

---

# 9. `shared_future`

普通 `future` 只能 `get()` 一次。
如果你想多个地方都读同一个结果，可以用 `shared_future`。

```cpp
std::future<int> f = p.get_future();
std::shared_future<int> sf = f.share();
```

然后：

```cpp
int a = sf.get();
int b = sf.get();
```

都可以。

这个适合“一份结果，多人消费”。

---

# 10. 几个很常见的坑

## 坑 1：`get_future()` 只能拿一次

```cpp
std::promise<int> p;
auto f1 = p.get_future();
auto f2 = p.get_future();  // 错
```

一个 promise 只能配一个 future。

---

## 坑 2：`set_value()` 也只能设置一次

```cpp
p.set_value(1);
p.set_value(2);  // 错
```

因为这是一次性通道。

---

## 坑 3：promise 被销毁但没设置结果

例如：

```cpp
std::future<int> f;
{
    std::promise<int> p;
    f = p.get_future();
}  // p 提前销毁
```

然后：

```cpp
f.get();
```

会抛 `std::future_error`，因为对应的 promise 消失了，结果永远不会来了。这个叫 **broken promise**。

---

## 坑 4：别忘了线程同步对象的生命周期

比如你把 `promise` 按引用捕获到线程里：

```cpp
std::thread t([&p]() { ... });
```

那就要保证线程执行期间 `p` 还活着。

---

# 11. 一个稍微真实点的例子

比如主线程发任务给工作线程，工作线程算平方后返回。

```cpp
#include <future>
#include <iostream>
#include <thread>

void worker(std::promise<int> p, int x) {
    p.set_value(x * x);
}

int main() {
    std::promise<int> p;
    std::future<int> f = p.get_future();

    std::thread t(worker, std::move(p), 12);

    std::cout << "result = " << f.get() << std::endl;  // 144

    t.join();
    return 0;
}
```

这里有个要点：

```cpp
std::thread t(worker, std::move(p), 12);
```

`promise` 一般要 `std::move` 传进线程，因为它通常不可拷贝、可移动。

---

# 12. 为什么 `promise` 往往要 move

`std::promise` 管的是共享状态的写端，它是独占语义，所以通常：

* 不可拷贝
* 可移动

所以你经常会看到：

```cpp
std::promise<int> p;
std::thread t(worker, std::move(p));
```

这是正常用法。

---

# 13. 和条件变量相比是什么关系

它们都能做线程同步，但用途不完全一样。

## `condition_variable`

更适合“反复通知 + 共享状态 + 多次协作”

## `promise/future`

更适合“一次性结果传递”

所以你可以粗略记：

* 一次性交付结果：`promise/future`
* 多次唤醒协调：`condition_variable`

---

# 14. 一张脑图式理解

你可以把它们记成：

```cpp
std::promise<T>   // 写端
std::future<T>    // 读端
```

流程是：

1. 创建 `promise`
2. 从 `promise` 拿到 `future`
3. 把 `promise` 交给生产者线程
4. 生产者调用 `set_value` 或 `set_exception`
5. 消费者用 `get` / `wait` 等待结果

---

# 15. 最推荐你先记住的三个例子

## 返回值

```cpp
std::promise<int> p;
auto f = p.get_future();
p.set_value(10);
std::cout << f.get();
```

## 完成通知

```cpp
std::promise<void> p;
auto f = p.get_future();
p.set_value();
f.wait();
```

## 异常传递

```cpp
try {
    throw std::runtime_error("err");
} catch (...) {
    p.set_exception(std::current_exception());
}
```

然后：

```cpp
f.get(); // 这里重新抛异常
```

---

# 16. 最后用一句话总结

`std::promise` 和 `std::future` 是 C++ 里一套“一次性异步结果通道”：

* `promise` 负责写结果
* `future` 负责等结果、取结果
* 可传正常值，也可传异常
* 特别适合线程间任务结果返回

## 第二部分 cv关系以及async同步

它们三组东西都和“**异步/等待**”有关，但层级不一样。

你可以先这样抓：

* `condition_variable`：底层同步工具，负责“有人改了共享状态，来通知一下”
* `promise/future`：更高层的一次性结果通道，负责“将来给你一个结果/异常”
* `async/future`：更省事的异步任务接口，负责“帮你起任务，并给你 future”

---

# 1. `condition_variable` 和 `promise/future` 的关系

可以把 `promise/future` 看成一种**封装好的、更专用的等待机制**。
底层实现未必直接就是 `condition_variable`，但思想上很像：

* 一边生产结果
* 一边等待结果
* 结果到了就唤醒等待方

只是 `promise/future` 帮你把这些都包装好了。

---

## `condition_variable` 更像什么

它一般和：

* `mutex`
* 共享状态变量

一起用。

典型模式：

```cpp
std::mutex m;
std::condition_variable cv;
bool ready = false;
int data = 0;
```

生产者：

```cpp
{
    std::lock_guard<std::mutex> lk(m);
    data = 42;
    ready = true;
}
cv.notify_one();
```

消费者：

```cpp
std::unique_lock<std::mutex> lk(m);
cv.wait(lk, [&] { return ready; });
std::cout << data << std::endl;
```

这里重点是：

* 你自己维护 `ready`
* 你自己维护 `data`
* 你自己加锁
* 你自己决定何时通知

所以它更通用，也更底层。

---

## `promise/future` 更像什么

同样的事，用 `promise/future` 写：

```cpp
std::promise<int> p;
std::future<int> f = p.get_future();
```

生产者：

```cpp
p.set_value(42);
```

消费者：

```cpp
std::cout << f.get() << std::endl;
```

这里：

* 不用自己写 `ready`
* 不用自己写 `cv.wait(...)`
* 不用自己处理“值到了没”
* 还能自动传异常

所以它是“**一次性交付结果**”的高层抽象。

---

# 2. 它俩最本质的区别

## `condition_variable`

适合：

* 多次通知
* 多个阶段同步
* 自定义复杂状态机
* 生产者消费者队列
* 线程池唤醒

比如“队列非空就唤醒消费者”，“停止标志来了就退出”。

---

## `promise/future`

适合：

* 一次性结果返回
* 一个任务算完给一个值
* 一个任务失败给一个异常
* 请求-响应式同步

比如“子线程算完平方，主线程拿结果”。

---

# 3. 一个对比例子

## 用 `condition_variable` 实现“一次性结果返回”

```cpp
#include <condition_variable>
#include <iostream>
#include <mutex>
#include <thread>

int main() {
    std::mutex m;
    std::condition_variable cv;
    bool ready = false;
    int result = 0;

    std::thread t([&] {
        int x = 42;
        {
            std::lock_guard<std::mutex> lk(m);
            result = x;
            ready = true;
        }
        cv.notify_one();
    });

    {
        std::unique_lock<std::mutex> lk(m);
        cv.wait(lk, [&] { return ready; });
        std::cout << result << std::endl;
    }

    t.join();
}
```

---

## 用 `promise/future` 实现同样的事

```cpp
#include <future>
#include <iostream>
#include <thread>

int main() {
    std::promise<int> p;
    std::future<int> f = p.get_future();

    std::thread t([&] {
        p.set_value(42);
    });

    std::cout << f.get() << std::endl;

    t.join();
}
```

后者明显更像“结果通道”。

---

# 4. 所以你可以这么记

`condition_variable` 提供的是：

> “条件满足了，叫醒别人。”

`future` 提供的是：

> “这个异步结果好了，你可以取了。”

前者关心“条件”，后者关心“结果”。

---

# 5. `future` 和 `async` 是怎么联系起来的

这个关系很直接：

## `async` 的返回值就是 `future`

例如：

```cpp
auto f = std::async([] {
    return 42;
});
```

这里 `f` 的类型就是：

```cpp
std::future<int>
```

也就是说：

* `async` 负责启动任务
* `future` 负责持有任务结果

---

## 最小例子

```cpp
#include <future>
#include <iostream>

int main() {
    std::future<int> f = std::async(std::launch::async, [] {
        return 21 * 2;
    });

    std::cout << f.get() << std::endl;  // 42
}
```

流程是：

1. `std::async` 启动一个异步任务
2. 这个任务将来返回 `int`
3. 返回值被放进共享状态
4. `future.get()` 取出结果

---

# 6. `async` 和 `promise` 的关系

可以把它理解成：

* `promise + future`：你自己搭“结果通道”
* `async + future`：标准库帮你搭

也就是：

## 手工版

```cpp
std::promise<int> p;
auto f = p.get_future();

std::thread t([p = std::move(p)]() mutable {
    p.set_value(42);
});
```

## 简化版

```cpp
auto f = std::async(std::launch::async, [] {
    return 42;
});
```

后者本质上就是“帮你起任务 + 帮你管理结果状态 + 给你 future”。

---

# 7. `async` 为什么只看到 `future` 看不到 `promise`

因为 `promise` 被藏起来了。

你可以理解成标准库内部帮你做了类似的事：

* 创建共享状态
* 启动函数
* 函数返回时把结果塞进共享状态
* 如果抛异常也塞进去
* 返回一个 `future`

所以你不需要显式写 `promise`。

---

# 8. `async` 和 `future` 典型用法

## 例 1：异步算值

```cpp
#include <future>
#include <iostream>

int calc() {
    return 100;
}

int main() {
    auto f = std::async(std::launch::async, calc);

    std::cout << "doing something else...\n";

    std::cout << f.get() << std::endl;
}
```

主线程可以先干别的，最后再 `get()`。

---

## 例 2：异常自动传递

```cpp
#include <future>
#include <iostream>
#include <stdexcept>

int main() {
    auto f = std::async(std::launch::async, []() -> int {
        throw std::runtime_error("boom");
    });

    try {
        std::cout << f.get() << std::endl;
    } catch (const std::exception& e) {
        std::cout << e.what() << std::endl;
    }
}
```

这里不需要手写 `set_exception`，`async` 会帮你传到 `future` 里。

---

# 9. `async` 的两个启动策略

这个很重要。

```cpp
std::async(std::launch::async, func);
std::async(std::launch::deferred, func);
```

---

## `std::launch::async`

真的开异步任务，通常会新开线程或类似调度。

```cpp
auto f = std::async(std::launch::async, [] {
    return 42;
});
```

---

## `std::launch::deferred`

不立刻执行，等你 `get()` 或 `wait()` 时才执行。

```cpp
auto f = std::async(std::launch::deferred, [] {
    return 42;
});
```

这时如果你从不 `get()`，它可能根本不运行。

---

## 默认策略

如果你写：

```cpp
auto f = std::async([] {
    return 42;
});
```

默认可能是：

* `async`
* 或 `deferred`
* 或两者由实现决定

所以在你想明确异步时，最好写：

```cpp
std::launch::async
```

---

# 10. 这三者放一起怎么理解

## `condition_variable`

你手工组织同步协议：

* 锁
* 条件
* 通知
* 共享状态

适合复杂协作。

---

## `promise/future`

你手工组织“一次性结果”：

* promise 写
* future 读

适合自定义线程/任务之间传结果。

---

## `async/future`

标准库直接帮你跑任务并返回 future。

适合“我就是想异步跑个函数拿结果”。

---

# 11. 一个并排对比

## A. `condition_variable`

```cpp
std::mutex m;
std::condition_variable cv;
bool ready = false;
int result = 0;
```

你自己管一切。

---

## B. `promise/future`

```cpp
std::promise<int> p;
auto f = p.get_future();
```

你自己决定何时 `set_value()`。

---

## C. `async/future`

```cpp
auto f = std::async(std::launch::async, [] { return 42; });
```

你连线程和 promise 都不用管。

---

# 12. 实战上怎么选

你可以按这个简单判断：

## 想做线程间复杂协调

用 `condition_variable`

例如：

* 队列非空唤醒消费者
* 多线程反复协作
* 线程池任务分发

---

## 想给某个任务返回一个结果

用 `promise/future`

例如：

* 某线程处理完后回传结果
* 你自己控制结果何时完成
* 你自己控制线程生命周期

---

## 想快速异步执行一个函数并拿返回值

用 `async/future`

例如：

* 并行算几个独立任务
* 临时异步跑函数
* 自动拿结果和异常

---

# 13. 一句最短总结

* `cv` 是“条件通知器”
* `promise/future` 是“手工版一次性结果通道”
* `async/future` 是“自动版异步任务 + 结果通道”

---

# 14. 你现在最该记住的一张图

```cpp
condition_variable
    -> 关注 条件满足没
    -> 更底层，更通用

promise -> future
    -> 关注 结果什么时候到
    -> 一次性传值/异常

async -> future
    -> 关注 函数异步执行后的结果
    -> 比 promise 更省事
```

---

# 15. 最后补一句容易混的点

`future` 不是只能和 `promise` 搭配。
`future` 的来源常见有两个：

```cpp
std::promise<T> -> std::future<T>
std::async(...) -> std::future<T>
```

所以 `future` 是“读结果”的统一接口，
而 `promise` 和 `async` 只是两种不同的“生产结果”的方式。
