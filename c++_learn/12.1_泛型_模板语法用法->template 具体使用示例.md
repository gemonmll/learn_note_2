## 先直接给结论

这句里的 `template` 是在告诉编译器：

# 一、为什么你会觉得“去掉也行”

因为这里的 `this` 不是依赖类型。

经典必须写 `template` 的情况一般是这样：

```cpp
template <typename T>
void foo(T* p) {
    p->template bar<int>();
}
```

这里 `p` 的类型是 `T*`，而 `T` 要等模板实例化后才知道。
所以编译器第一眼看到：

```cpp
p->bar<int>()
```

没法确定：

* `bar` 是模板
* 还是 `bar < int` 是比较

所以必须写：

```cpp
p->template bar<int>()
```

---

但你这个不是。

你这里是：

```cpp
this->template CreateReader<MessageT>(...)
```

而 `this` 在这个成员函数里，类型就是：

```cpp
NodeChannelImpl*
```

这不是依赖模板参数的类型。
所以编译器通常是知道 `CreateReader` 是成员模板的。

通常有几种可能：

```cpp
this->template CreateReader<MessageT>(...)
```

比

```cpp
this->CreateReader<MessageT>(...)
```

> 这里调用的是另一个模板重载。

```cpp

```

就顺手加 `template`，哪怕某些地方 technically 可省。

## 3）兼容性/风格上的保守写法

有些项目会倾向写得更“防解析歧义”，即使这里未必严格需要。

## 重载 2

第一个重载做的事其实只是“转发参数”：

* 填 `role_attr`

# 四、为什么这里用 `this->`

这个你也顺便可以一起理解。

它本来甚至可能写成：

```cpp
return CreateReader<MessageT>(role_attr, reader_func,
                              config.pending_queue_size);
```

或者：

```cpp
return this->CreateReader<MessageT>(role_attr, reader_func,
                                    config.pending_queue_size);
```

加 `this->` 一般是为了明确：

* 这是当前对象的成员函数
* 不是全局函数
* 不是别的名字查找结果

再进一步加 `template`，就是：

* 当前对象的成员模板函数

```cpp
    }
```

前提是 `bar` 来自依赖基类或依赖对象，编译器在第一阶段不确定它是不是模板。

# 八、最简判断法

你以后可以这样判断：

## 如果左边是依赖类型

像：



→ **往往可以不加，写不写看上下文和风格**
