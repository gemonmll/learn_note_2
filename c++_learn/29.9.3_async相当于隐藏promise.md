## 第三部分 promise和async（把promise隐藏）
可以。你就抓住一句：

**`async` 是“把 `promise` 藏起来并自动帮你用掉”的更高层封装。**

---

## 先说结论

这两种写法，本质目标一样：

### 手工版：`promise + future`

你自己负责：

* 建立结果通道
* 启动线程
* 在线程里 `set_value()` / `set_exception()`

### 自动版：`async + future`

标准库替你负责：

* 建立结果通道
* 启动任务
* 把函数返回值放进去
* 把异常放进去

所以你只看到了 `future`，没看到 `promise`，因为 **`promise` 那一半被 `async` 内部接管了**。

---

# 1. 先看手工版

比如异步算一个值。

```cpp
#include <future>
#include <thread>
#include <iostream>

int main() {
    std::promise<int> p;
    std::future<int> f = p.get_future();

    std::thread t([p = std::move(p)]() mutable {
        int x = 42;
        p.set_value(x);
    });

    std::cout << f.get() << std::endl;

    t.join();
}
```

这里你自己做了三件事：

1. 创建 `promise`
2. 从它拿 `future`
3. 在线程里手工 `set_value(42)`

---

# 2. 再看 `async` 版

```cpp
#include <future>
#include <iostream>

int main() {
    std::future<int> f = std::async(std::launch::async, [] {
        return 42;
    });

    std::cout << f.get() << std::endl;
}
```

你少写了很多代码，因为这些步骤都被 `async` 包了。

---

# 3. 它们的对应关系

你可以把 `async` 脑补成“内部大概做了下面这些事”：

```cpp
template <class F>
auto fake_async(F func) {
    using R = decltype(func());

    std::promise<R> p;
    std::future<R> f = p.get_future();

    std::thread([p = std::move(p), func]() mutable {
        try {
            p.set_value(func());
        } catch (...) {
            p.set_exception(std::current_exception());
        }
    }).detach();

    return f;
}
```

这不是标准库源码，只是帮助理解。

核心意思是：

* `async` 内部也需要某种“共享状态”
* 任务执行完，要把结果写进去
* `future` 从那里把结果读出来

这和 `promise/future` 的思想是同一个模型。

---

# 4. 为什么 `async` 返回的是 `future`

因为 `future` 才是“对外暴露的读端”。

对调用者来说，你最关心的是：

* 任务完成了吗
* 结果是什么
* 有没有异常

这些都由 `future` 提供：

* `get()`
* `wait()`
* `wait_for()`

而“写端”不需要你手工操作，所以 `async` 没必要把 `promise` 暴露给你。

---

# 5. 所以它们的关系到底是什么

最准确的理解是：

**`future` 是结果句柄，`promise` 是手工写结果的工具，`async` 是自动写结果的工具。**

也就是：

```cpp
promise  ----写---->  shared state  ----读----> future
async任务 ----写----> shared state  ----读----> future
```

`promise` 和 `async` 都像“生产者”；
`future` 是“消费者接口”。

---

# 6. 为什么说 `async` 不是单纯的 `promise`

因为 `async` 不只是“写值”而已，它还负责：

* 调度函数执行
* 可能创建线程
* 管理延迟执行策略
* 自动捕获异常
* 自动把返回值放入共享状态

所以它比 `promise` 更高一层。

你可以理解成：

* `promise`：我自己已经有线程/任务了，只差一个结果通道
* `async`：我连任务启动都想让标准库帮我做

---

# 7. 一个特别重要的区别

## `promise`

你可以在任何时机手工决定结果：

```cpp
p.set_value(10);
```

这个值不一定是函数返回值，也可能来自：

* 某个回调
* 某个 IO 结果
* 某个事件触发
* 多步处理后的最终结果

所以它更灵活。

---

## `async`

结果通常就是“函数执行的返回值”：

```cpp
auto f = std::async(..., [] {
    return 10;
});
```

你不能中途手工 `set_value`，因为这部分控制权已经交给标准库了。

---

# 8. 一句话区分使用场景

## 用 `async`

当你想说：

> “帮我异步跑这个函数，我以后拿它的结果。”

## 用 `promise`

当你想说：

> “任务/线程/回调我自己控制，但我要把最终结果交给别人等。”

---

# 9. 再看异常，就更清楚了

### `promise` 手工版

```cpp
std::promise<int> p;
auto f = p.get_future();

std::thread t([p = std::move(p)]() mutable {
    try {
        throw std::runtime_error("err");
    } catch (...) {
        p.set_exception(std::current_exception());
    }
});
```

这里异常要你自己抓，再手工塞给 `future`。

---

### `async` 版

```cpp
auto f = std::async(std::launch::async, []() -> int {
    throw std::runtime_error("err");
});
```

然后：

```cpp
f.get();
```

异常会自动重新抛出。

说明 `async` 内部帮你做了“异常写入共享状态”的工作。

---

# 10. 你可以把它想成两层 API

## 底层一点

`promise/future`

你自己管生产过程。

## 高层一点

`async/future`

标准库替你管生产过程。

---

# 11. 最后用一句最像人话的总结

`promise` 像你自己拿着快递往柜子里放，`future` 是取件码。
`async` 则像你把“打包、投递、入柜”整套流程都交给快递员，最后你仍然只拿到那个取件码 `future`。

---

# 12. 你现在最该记住的图

```cpp
手工模式:
promise -> 把结果写进共享状态 -> future 取结果

自动模式:
async启动函数 -> 函数返回值/异常写进共享状态 -> future 取结果
```

**“shared state 到底是什么，为什么 promise 和 async 最终都能产出 future”**

这个一旦懂了，你对它们的关系会彻底通。
