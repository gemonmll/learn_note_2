好，我们把 **poll / epoll 和 pipe / socket 怎么一起用** 这件事彻底串起来。

你现在要建立的是这个核心观念：

> **pipe、socket 本质上都是 fd。**
>
> 只要是 fd，而且支持“就绪事件”，就可以放进 `poll/epoll` 里统一等待。

所以它们能一起用，不是因为它们长得像，而是因为在 Linux 里都走 **“文件描述符 + 事件通知”** 这套模型。

---

# 1. 先把四个东西的关系摆正

## 1）pipe 是什么

`pipe` 是内核提供的一种**单向字节流通道**，常用于：

* 父子进程通信
* 同进程不同线程之间简单唤醒
* 做一个“通知通道”

创建后会得到两个 fd：

* `pipefd[0]`：读端
* `pipefd[1]`：写端

写端写入，读端读出。

---

## 2）socket 是什么

`socket` 也是一种 fd，但它更通用，用于：

* 本机进程间通信（Unix domain socket）
* 网络通信（TCP / UDP）
* 服务端监听连接、收发消息

比如 TCP 里常见两个层次：

* `listenfd`：监听新连接
* `connfd`：已经建立好的连接，用来收发数据

---

## 3）poll / epoll 是什么

它们不负责传数据，只负责：

> **帮你等一堆 fd，看看谁先有事。**

“有事”通常指：

* 可读
* 可写
* 出错
* 对端关闭

所以：

* pipe 里来了数据 → 可读
* socket 收到网络数据 → 可读
* listen socket 有新连接到来 → 可读
* pipe/socket 对端关闭 → hangup / rdhup

---

# 2. 为什么 pipe 和 socket 能一起放进 poll/epoll

因为 `poll/epoll` 不关心你这个 fd 背后到底是：

* pipe
* socket
* eventfd
* timerfd
* stdin

它只关心：

> **这个 fd 现在有没有事件。**

所以一个事件循环里完全可以同时监听：

* 一个 pipe：做“内部通知”
* 一个 socket：做“网络通信”
* 一个 eventfd：做线程唤醒
* 一个 timerfd：做定时器

这就是 Linux 事件驱动程序的核心。

---

# 3. 最常见的实际搭配方式

最经典的组合是：

## 方案 A：socket 做业务数据，pipe 做控制通知

例如主线程跑 `epoll_wait`，同时监听：

* `listenfd`：新客户端连接
* `connfd`：客户端数据
* `pipefd[0]`：别的线程通知“退出”或“刷新配置”

这样一个线程就能统一处理：

* 网络数据
* 内部命令
* 退出信号

这非常常见。

---

## 方案 B：socketpair / Unix socket 代替 pipe

有时也不用 `pipe`，而用 `socketpair()`。

因为 `socketpair` 是**双向**的，而 pipe 天生通常是单向的。

# 4. 先看 pipe 和 poll 怎么一起用

---

## 4.1 基本思路

### 创建 pipe

```cpp
int pipefd[2];
pipe(pipefd);
```

得到：

* `pipefd[0]` 读
* `pipefd[1]` 写

### 把读端加入 poll

因为你通常关心的是：

> 什么时候有数据可读

```cpp
struct pollfd pfd;
pfd.fd = pipefd[0];
pfd.events = POLLIN;
```

### 别处写数据

```cpp
write(pipefd[1], "hello", 5);
```

### poll 返回后 read

```cpp
if (pfd.revents & POLLIN) {
    char buf[128];
    int n = read(pipefd[0], buf, sizeof(buf));
}
```

---

## 4.2 一个完整例子：父子进程 + pipe + poll

```cpp
#include <unistd.h>
#include <poll.h>
#include <sys/wait.h>
#include <iostream>
#include <cstring>

int main() {
    int pipefd[2];
    if (pipe(pipefd) < 0) {
        perror("pipe");
        return 1;
    }

    pid_t pid = fork();
    if (pid < 0) {
        perror("fork");
        return 1;
    }

    if (pid == 0) {
        close(pipefd[0]);  // 子进程不用读端
        sleep(2);
        const char* msg = "message from child";
        write(pipefd[1], msg, strlen(msg));
        close(pipefd[1]);
        return 0;
    }

    close(pipefd[1]);  // 父进程不用写端

    struct pollfd pfd;
    pfd.fd = pipefd[0];
    pfd.events = POLLIN;

    while (true) {
        int ret = poll(&pfd, 1, 5000);
        if (ret < 0) {
            perror("poll");
            break;
        } else if (ret == 0) {
            std::cout << "timeout\n";
            continue;
        }

        if (pfd.revents & POLLIN) {
            char buf[128] = {0};
            ssize_t n = read(pipefd[0], buf, sizeof(buf) - 1);
            if (n > 0) {
                std::cout << "parent read: " << buf << "\n";
            } else if (n == 0) {
                std::cout << "pipe closed\n";
                break;
            }
        }

        if (pfd.revents & POLLHUP) {
            std::cout << "writer closed\n";
            break;
        }
    }

    close(pipefd[0]);
    wait(nullptr);
    return 0;
}
```

# 5. socket 和 poll 怎么一起用

---

## 5.1 socket 最重要的两个“可读”场景

### 场景 1：监听 socket（listenfd）可读

表示：

> 有新连接到了，可以 `accept()`

123
### 场景 2：已连接 socket（connfd）可读

表示：

> 对端发数据来了，可以 `read()` / `recv()`

所以 socket 的“可读”不止一种含义，要看它是什么类型的 socket。

---

## 5.2 TCP 服务端 + poll 的基本模型

你通常会：

1. `socket()`
2. `bind()`
3. `listen()`
4. 把 `listenfd` 放进 `poll`
5. `poll` 返回后：

   * 如果是 `listenfd` 可读 → `accept()`
   * 如果是某个 `connfd` 可读 → `recv()`

---

## 5.3 最小思路代码

```cpp
if (fds[i].fd == listenfd && (fds[i].revents & POLLIN)) {
    int connfd = accept(listenfd, nullptr, nullptr);
    // 把 connfd 也加入 poll
}
else if (fds[i].revents & POLLIN) {
    char buf[1024];
    int n = recv(fds[i].fd, buf, sizeof(buf), 0);
    if (n > 0) {
        // 收到数据
    } else if (n == 0) {
        // 对端关闭
    }
}
```

# 6. 真正关键：pipe 和 socket 如何一起用

这才是你最想学的部分。

假设你有一个线程在跑事件循环，它既要处理网络连接，又要处理内部控制命令。

那就可以同时监听：

* `listenfd`：新连接
* `connfd`：客户端消息
* `pipefd[0]`：内部线程通知

这叫：

> **一个 event loop 统一处理不同来源的事件**

---

# 7. 一个很典型的使用场景

比如你写一个服务程序，主线程阻塞在 `epoll_wait`：

* 客户端发消息过来 → socket 就绪
* 管理线程发“退出”命令 → pipe 就绪

主线程不需要忙轮询，也不需要开两个阻塞线程分别等。

它只需要：

```cpp
epoll_wait(...)
```

谁来了就处理谁。

---

# 8. pipe + socket + epoll 的完整思路图

可以脑子里这样想：

```text
             +--------------------+
             |    epoll_wait      |
             +--------------------+
                 /      |       \
                /       |        \
               /        |         \
        listenfd     connfd     pipefd[0]
       新连接到来    收到业务数据   内部通知/退出命令
```

所以 epoll 像一个总调度器。

---

# 9. 一个能看懂结构的例子：socket + pipe + epoll

下面我给你一个“教学版”的骨架。不是完整商用服务器，但结构是对的。

```cpp
#include <sys/epoll.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <unistd.h>
#include <fcntl.h>
#include <arpa/inet.h>
#include <cstring>
#include <iostream>
#include <vector>

int set_nonblock(int fd) {
    int flags = fcntl(fd, F_GETFL, 0);
    if (flags < 0) return -1;
    return fcntl(fd, F_SETFL, flags | O_NONBLOCK);
}

int main() {
    int pipefd[2];
    if (pipe(pipefd) < 0) {
        perror("pipe");
        return 1;
    }

    int listenfd = socket(AF_INET, SOCK_STREAM, 0);
    if (listenfd < 0) {
        perror("socket");
        return 1;
    }

    int opt = 1;
    setsockopt(listenfd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    sockaddr_in addr{};
    addr.sin_family = AF_INET;
    addr.sin_port = htons(8888);
    addr.sin_addr.s_addr = INADDR_ANY;

    if (bind(listenfd, (sockaddr*)&addr, sizeof(addr)) < 0) {
        perror("bind");
        return 1;
    }

    if (listen(listenfd, 16) < 0) {
        perror("listen");
        return 1;
    }

    set_nonblock(listenfd);
    set_nonblock(pipefd[0]);

    int epfd = epoll_create1(0);
    if (epfd < 0) {
        perror("epoll_create1");
        return 1;
    }

    epoll_event ev{};
    ev.events = EPOLLIN;
    ev.data.fd = listenfd;
    epoll_ctl(epfd, EPOLL_CTL_ADD, listenfd, &ev);

    ev.events = EPOLLIN;
    ev.data.fd = pipefd[0];
    epoll_ctl(epfd, EPOLL_CTL_ADD, pipefd[0], &ev);

    epoll_event events[32];
    std::vector<int> clients;

    std::cout << "server listen on 8888\n";
    std::cout << "write to pipefd[1] from another place to notify loop\n";

    while (true) {
        int n = epoll_wait(epfd, events, 32, -1);
        if (n < 0) {
            perror("epoll_wait");
            break;
        }

        for (int i = 0; i < n; ++i) {
            int fd = events[i].data.fd;

            if (fd == listenfd) {
                while (true) {
                    int connfd = accept(listenfd, nullptr, nullptr);
                    if (connfd < 0) {
                        break;
                    }

                    set_nonblock(connfd);
                    epoll_event cev{};
                    cev.events = EPOLLIN;
                    cev.data.fd = connfd;
                    epoll_ctl(epfd, EPOLL_CTL_ADD, connfd, &cev);
                    clients.push_back(connfd);

                    std::cout << "new client: " << connfd << "\n";
                }
            } 
            else if (fd == pipefd[0]) {
                char buf[128] = {0};
                ssize_t r = read(pipefd[0], buf, sizeof(buf) - 1);
                if (r > 0) {
                    std::cout << "pipe notify: " << buf << "\n";
                    if (std::string(buf, r) == "quit") {
                        std::cout << "exit loop\n";
                        goto END;
                    }
                }
            } 
            else {
                char buf[1024] = {0};
                ssize_t r = recv(fd, buf, sizeof(buf) - 1, 0);
                if (r > 0) {
                    std::cout << "client " << fd << " says: " << buf << "\n";
                    send(fd, buf, r, 0); // echo
                } else if (r == 0) {
                    std::cout << "client closed: " << fd << "\n";
                    epoll_ctl(epfd, EPOLL_CTL_DEL, fd, nullptr);
                    close(fd);
                } else {
                    std::cout << "recv error on fd " << fd << "\n";
                    epoll_ctl(epfd, EPOLL_CTL_DEL, fd, nullptr);
                    close(fd);
                }
            }
        }
    }

END:
    for (int fd : clients) close(fd);
    close(pipefd[0]);
    close(pipefd[1]);
    close(listenfd);
    close(epfd);
    return 0;
}
```

# 10. 这个例子里最核心的学习点

你别急着抄代码，先看懂这三个分支：

## 分支 1：`fd == listenfd`

说明：

> 有新客户端连接来了

所以做：

```cpp
accept(...)
```

然后把新的 `connfd` 再加进 epoll。

---

## 分支 2：`fd == pipefd[0]`

说明：

> 内部有通知消息来了

所以做：

```cpp
read(pipefd[0], ...)
```

比如读到 `"quit"`，就退出 event loop。

---

## 分支 3：其他 fd

这些一般都是客户端连接 fd。

说明：

> 某个客户端发数据来了

所以做：

```cpp
recv(fd, ...)
```

这三个分支就构成了一个完整事件循环。

---

# 11. 为什么很多程序要“socket + pipe”一起用

因为单纯 socket 只能处理网络事件，但程序内部也常有控制需求：

* 让事件循环退出
* 让工作线程唤醒主线程
* 重新加载配置
* 通知有任务要处理

如果主线程卡在 `epoll_wait`，你总得有办法把它叫醒。

这时 pipe 就很方便：

* 管理线程往 `pipefd[1]` 写一个字节
* 主线程监听 `pipefd[0]`
* epoll 立刻醒来

这叫 **self-pipe trick** 的思路，虽然现代里也常用 `eventfd`。

# 12. pipe 和 socket 的共同点与区别

---

## 共同点

* 都是 fd
* 都能 `read/write` 或类似操作
* 都能放进 `poll/epoll`
* 都能变成非阻塞
* 都有“可读/可写/关闭”等状态

---

## 区别

### pipe

* 更简单
* 常用于本机内部通信
* 默认单向
* 没有“连接建立”这个概念

### socket

* 更通用
* 可本地可网络
* TCP 有连接概念
* 会有 `listen` / `accept` / `connect`

---

# 13. 学这块时最容易糊涂的地方

---

## 1）listenfd 可读，不是“有数据来了”

而是：

> 有新连接可以 accept 了

这和 `connfd` 可读不同。

---

## 2）pipe 可读，不代表只读一次就完

如果写端写了很多数据，你可能要循环读。

LT 模式下问题不大，ET 模式下尤其要读到 `EAGAIN`。

---

## 3）socket 收到 0 字节，不是“没数据”

对于 TCP 来说：

```cpp
recv(fd, buf, size, 0) == 0
```

通常表示：

> 对端关闭连接了

这个非常关键。

---

## 4）pipe 读到 0，也常表示写端都关了

这相当于“对端结束”。

---

# 14. 为什么非阻塞常和 epoll 配套

因为你通常不想：

* `epoll_wait` 返回了
* 然后你 `read/accept/recv` 又卡死

所以常把：

* `listenfd`
* `connfd`
* `pipefd[0]`

都设成非阻塞。

特别是 ET 模式下，几乎是必须的。

---

# 15. pipe / socket / epoll 这套东西在工程里怎么理解

你可以用一句话记住：

> **epoll 是总调度器，socket 负责外部通信，pipe 负责内部通知。**

这句话很重要。

一个进程里往往同时有：

* 外部世界：网络客户端 → socket
* 内部世界：线程/模块之间通知 → pipe 或 eventfd
* 时间事件：timerfd
* 信号事件：signalfd

全部都可以统一进一个 epoll 循环。

# 17. 给你一个最终脑图

以后看到这套东西，你脑子里就这么想：

```text
事件循环线程:
    epoll_wait(...)

返回后:
    1. listenfd 有事件
       -> accept 新连接

    2. connfd 有事件
       -> recv / send 业务数据

    3. pipefd[0] 有事件
       -> read 内部通知，例如 quit / reload
```

这就是“pipe socket 如何一起使用”的本质。

> **pipe 和 socket 是事件源，poll/epoll 是等待和分发事件的机制。**
