```cpp
class Node { public: template <typename M0, typename M1, typename M2, typename M3>
friend class Component; 
friend class TimerComponent; 
friend bool Init(const char*, const std::string&); 
friend std::unique_ptr<Node> CreateNode(const std::string&, const std::string&);
```

因为 **`private` 构造函数只能在类内部，或者它的 friend 里调用**。

你这段里：

```cpp
class Node {
 private:
  explicit Node(const std::string& node_name,
                const std::string& name_space = "");
};
```

所以类外普通代码这样写：

```cpp
Node n("test");
auto p = std::make_unique<Node>("test");
```

都不行。

## 为什么不行

因为这两句本质上都在 **类外调用 `Node` 的构造函数**。

而 `Node(...)` 是 `private`，访问控制不允许。

编译器会把它理解成：

* 你想调用 `Node::Node(...)`
* 但这个构造函数对当前作用域不可见/不可访问

---

## 那为什么 friend 可以

比如这句：

```cpp
friend std::unique_ptr<Node> CreateNode(const std::string&,
                                        const std::string&);
```

等于 `Node` 明确授权：

> `CreateNode` 这个函数虽然在类外，但我允许它访问我的 private 成员。

所以它可以：

```cpp
std::unique_ptr<Node> CreateNode(const std::string& name,
                                 const std::string& ns) {
    return std::unique_ptr<Node>(new Node(name, ns));
}
```

这里 `new Node(name, ns)` 是合法的吗？
**合法**，因为它发生在 friend 函数里。

---

## 你可以把它理解成“门禁”

`private` 像一扇上锁的门。

* 普通外部代码：没有钥匙，进不去
* `friend`：被主人单独发了钥匙，可以进

---

## 一个最小例子

```cpp
class A {
private:
    A() {}
    friend A makeA();
};

A makeA() {
    return A();   // 可以，makeA 是 friend
}

int main() {
    A a;          // 不可以，main 不是 friend
}
```

这里：

* `main` 不能直接 `A a;`
* `makeA()` 可以 `return A();`

原因就只是：`A()` 是 private，而 `makeA` 被授予了 friend 权限。

---

## 为什么作者要这么设计

通常是为了 **不让别人乱创建对象**，强制大家走指定入口。

比如 `Node` 可能创建时要顺便：

* 检查名字是否合法
* 注册拓扑
* 初始化内部资源
* 建立 channel/service 结构

所以作者不想让你随便：

```cpp
Node n("abc");
```

而是要求你走：

```cpp
auto node = CreateNode("abc", "");
```

这样创建流程可控。
