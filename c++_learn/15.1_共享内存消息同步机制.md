不是，只查共享内存里的值只是**最原始的一种办法：轮询**。
更完整地说：

**共享内存解决的是“数据怎么共享”，不解决“谁先谁后、什么时候通知对方”。**

所以进程间共享内存通信通常分成两部分：

1. **数据通道**：共享内存
2. **同步/通知通道**：锁、条件变量、信号量、eventfd、pipe、futex 等

---

# 1. 共享内存到底是什么

正常情况下，不同进程的虚拟地址空间彼此隔离：

* 进程 A 看不到进程 B 的普通变量
* 各自有各自的堆、栈、全局区

而共享内存是：

* 内核准备一块物理内存页
* 把它同时映射到多个进程的地址空间
* 于是多个进程都能访问同一份数据

所以：

* A 写进去的数据
* B 可以直接看到

这也是它快的原因：**不需要像 socket/pipe 那样多一次“拷贝到内核缓冲区、再拷贝出来”**。

---

# 2. 共享内存能做什么

最常见用途：

* 高频数据交换
* 大块数据传输
* 低延迟 IPC
* 共享状态区
* 环形队列 / 缓冲区

比如：

* 图像帧
* 激光点云
* 音视频数据
* 传感器状态
* 多进程日志缓存
* 实时模块间消息通道

---

# 3. 共享内存为什么“不能单独完成通信”

因为你虽然共享了一块内存，但会立刻碰到几个问题：

## 问题 1：对方怎么知道你写完了？

你把 `data` 写进去时，对方可能正在读。

## 问题 2：谁先写谁先读？

如果读进程先开始读，读到半截怎么办？

## 问题 3：多个进程同时写怎么办？

会不会互相覆盖？

## 问题 4：怎么避免忙等？

如果对方没写新数据，难道一直 `while(1)` 地查？

所以共享内存只是“共用仓库”，
但仓库还需要：

* 门锁
* 红绿灯
* 门铃
* 规则

---

# 4. 最原始的做法：轮询

比如共享内存里放一个结构：

```cpp
struct SharedData {
  int ready;
  int value;
};
```

写进程：

```cpp
shared->value = 42;
shared->ready = 1;
```

读进程：

```cpp
while (shared->ready == 0) {
  // 一直等
}
printf("%d\n", shared->value);
```

这就是轮询。

---

## 轮询的优点

* 简单
* 容易实现
* 在极短路径、极高实时要求下，有时可接受

---

## 轮询的缺点

* 浪费 CPU
* 延迟和功耗都不好
* 容易写出竞态
* 不适合复杂生产环境

所以实际工程里，**一般不会只靠轮询**。

---

# 5. 更常见的做法：共享内存 + 同步机制

这才是正统方案。

---

## 方案 A：共享内存 + 互斥锁 + 条件变量

这是最经典的“有数据就唤醒”。

共享内存里放：

```cpp
struct SharedBlock {
  pthread_mutex_t mutex;
  pthread_cond_t cond;
  int ready;
  int value;
};
```

注意这里的 `pthread_mutex_t` 和 `pthread_cond_t` 需要设置成**进程共享**属性：

* `PTHREAD_PROCESS_SHARED`

### 写进程

```cpp
pthread_mutex_lock(&blk->mutex);
blk->value = 42;
blk->ready = 1;
pthread_cond_signal(&blk->cond);
pthread_mutex_unlock(&blk->mutex);
```

### 读进程

```cpp
pthread_mutex_lock(&blk->mutex);
while (!blk->ready) {
  pthread_cond_wait(&blk->cond, &blk->mutex);
}
printf("%d\n", blk->value);
pthread_mutex_unlock(&blk->mutex);
```

### 优点

* 不需要忙等
* 写完直接唤醒
* 逻辑清晰

这是“共享内存不是只能查”的最直接答案。

---

## 方案 B：共享内存 + 信号量

比如 POSIX semaphore：

* `sem_init(..., 1, ...)` 或命名信号量 `sem_open`

适合：

* 生产者-消费者
* 计数资源管理
* 缓冲区槽位控制

例如环形队列：

* `empty_sem` 表示空槽数量
* `full_sem` 表示已写数据数量
* `mutex` 保护 head/tail

写者：

1. `sem_wait(empty_sem)`
2. 写数据
3. `sem_post(full_sem)`

读者：

1. `sem_wait(full_sem)`
2. 读数据
3. `sem_post(empty_sem)`

这在多进程共享内存队列里特别常见。

---

## 方案 C：共享内存 + eventfd

Linux 下很实用。

思路：

* 大数据放共享内存
* “有新数据了”通过 `eventfd` 通知
* 对方可以 `read/poll/epoll` 等待这个通知

优点：

* 很适合事件驱动
* 可以纳入 `epoll`
* 比轮询优雅很多

典型模式：

* 共享内存里存消息或帧
* 写者写完后往 `eventfd` 写一个整数
* 读者在 `epoll_wait` 中被唤醒，再去读共享内存

---

## 方案 D：共享内存 + futex

这是更底层、更高性能的做法。

* 共享内存中放一个整型状态位
* 用户态先原子判断
* 需要睡眠时调用 `futex`
* 内核在状态变化时唤醒等待者

很多高性能锁/同步原语底层都依赖 futex。

你可以理解成：

* 比 pthread 更底层
* 更难写
* 但性能和控制力更强

---

## 方案 E：共享内存 + signal / pipe / socket 唤醒

也可以：

* 数据在共享内存
* 通知通过其他 IPC 完成

比如：

* 写完共享内存后发一个 UNIX socket 消息
* 或写 pipe 一个字节
* 或发信号 `SIGUSR1`

这种“数据面和控制面分离”的设计很常见。

---

# 6. 实际通信一般怎么设计

## 模式 1：单写单读状态块

适合共享配置、状态标志。

例如：

```cpp
struct Status {
  std::atomic<uint64_t> seq;
  SensorData data;
};
```

写者：

1. 更新数据
2. 更新版本号

读者：

1. 读版本号
2. 读数据
3. 再次读版本号
4. 检查是否一致

这叫**版本号/sequence lock** 风格，适合“写少读多”。

---

## 模式 2：环形队列

最常见。

共享内存中维护：

* buffer[N]
* head
* tail

写者把消息写到 `tail`
读者从 `head` 取

再配合：

* mutex + cond
* 或 semaphore
* 或 atomic + futex

这就是共享内存消息队列的基本模型。

---

## 模式 3：双缓冲 / 多缓冲

适合大块数据，如图像帧。

比如两个 buffer：

* 一个正在写
* 一个正在读

写完后切换“当前可读缓冲区编号”。

这样能减少读写冲突。

---

# 7. 为什么很多框架不是“只用共享内存”

因为共享内存很快，但同步很难。

工程里常见组合：

* **共享内存传大数据**
* **socket/eventfd/condvar 传通知**
* **atomic/mutex 保护一致性**

例如：

* ROS2 某些 zero-copy 方案
* 音视频共享缓冲
* 自动驾驶中大消息传输
* 数据采集系统

都是这种思路。

---

# 8. “只能一直查询吗”的直接回答

## 可以一直查

这叫：

* polling
* busy wait
* 自旋/轮询

简单但通常不优雅。

---

## 也可以阻塞等待通知

这才是更常见的方式，例如：

* 条件变量
* 信号量
* eventfd
* futex
* pipe/socket 唤醒

所以答案是：

**不是只能一直查询。共享内存本身不带通知能力，但几乎总是和某种等待/唤醒机制一起使用。**

---

# 9. 为什么共享内存“本身不带通知能力”

因为内存只是内存。

你往共享内存某个地址写了：

```cpp
shared->ready = 1;
```

CPU 不会自动帮另一个进程说：

> “喂，对面醒醒，有数据了。”

所以如果没有额外同步机制，对方就只能：

* 不停读这个变量
* 或偶尔 sleep 再查
* 或靠别的通知通道唤醒

---

# 10. 一个非常典型的正确思路

可以记成：

## 小数据控制，大数据共享

* 控制信号：mutex/cond/semaphore/eventfd
* 真实数据：shared memory

例如：

```text
写进程:
  写共享内存大块数据
  -> 发通知

读进程:
  阻塞等待通知
  -> 被唤醒后读取共享内存
```

这就是最实用的共享内存通信模型。

---

# 11. 什么时候还会用轮询

虽然我一直说别只轮询，但有些场景会保留轮询：

* 超低延迟
* CPU 核心专用
* 等待时间极短
* 内核切换开销比轮询还贵
* 实时系统中的自旋等待

例如：

* 高频交易
* 某些实时控制路径
* lock-free ring buffer 的短时自旋

但这通常会很谨慎，不是普通业务程序默认选择。

---

# 12. 一个简单分类表

## 只轮询

特点：

* 最简单
* 最费 CPU
* 适合 demo 或非常短的等待

## 共享内存 + 锁/条件变量

特点：

* 通用
* 易理解
* 适合大多数多进程场景

## 共享内存 + 信号量

特点：

* 很适合生产者消费者
* 适合队列/槽位计数

## 共享内存 + eventfd/epoll

特点：

* Linux 风格强
* 适合事件循环
* 很适合高性能服务

## 共享内存 + futex/atomic

特点：

* 最底层
* 高性能
* 复杂，容易写错

---

# 13. 你可以这样形成整体理解

共享内存通信不是“一个 API 完事”，而是三层：

## 第一层：把同一块内存映射给多个进程

比如：

* `shmget + shmat`
* `shm_open + mmap`

## 第二层：规定共享数据结构

比如：

* 状态块
* 环形队列
* 双缓冲
* 版本号

## 第三层：解决同步和唤醒

比如：

* mutex
* condvar
* semaphore
* eventfd
* futex

这三层拼起来，才是完整的进程间共享内存通信。

---

# 14. 一句话总结

**共享内存不是只能一直查询；共享内存只负责让多个进程访问同一份数据，真正高效可靠的通信通常要再配合同步/通知机制，比如条件变量、信号量、eventfd 或 futex。**
