它们本质上是 **SOME/IP-SD 报文体里的两张表**：

* **Entries Array**：写“**我要表达什么动作/对象**”
* **Options Array**：写“**这个动作附带的地址、端口、配置参数是什么**”

SOME/IP-SD 的负载结构本来就是由 **SD Header + Entries Array + Options Array** 组成。

---

# 1. 先说 Entries Array 是什么

你可以把 **Entries Array** 理解成：

> **SD 报文里的“主记录表”**

每一条 entry 都在描述一个“服务发现相关动作”。

常见 entry 类型有：

* `FindService`
* `OfferService`
* `SubscribeEventgroup`
* `SubscribeEventgroupAck`
* `SubscribeEventgroupNack`。

所以 entry 关心的是：

* 我现在是在 **找服务**
* 还是在 **宣告提供服务**
* 还是在 **订阅某个 eventgroup**
* 还是在 **确认订阅成功/失败**

同时，它还会带上这个动作对应的核心标识，比如：

* `Service ID`
* `Instance ID`
* `Major Version`
* `TTL`
* 对于订阅类 entry，还会有 `Eventgroup ID`。

你可以把它想成：

```text id="ou3m37"
Entry = “动作 + 目标对象”
```

比如：

```text id="7yvcr7"
OfferService:
  service=0x1234
  instance=0x0001
  major=1
  ttl=3
```

这条 entry 的意思就是：

> “我提供这个服务实例。”

---

# 2. 再说 Options Array 是什么

**Options Array** 不是动作表，而是：

> **给 entry 提供附加信息的“参数表/附件表”**

典型 option 里装的是：

* IPv4 endpoint
* IPv6 endpoint
* multicast endpoint
* configuration option 等。

最常见的就是 endpoint option，例如：

```text id="jlwmnf"
IPv4 Address = 192.168.1.10
L4 Proto     = UDP
Port         = 30509
```

它表达的不是“我要 offer service”，而是：

> “如果你要和这个服务通信，请去这个 IP 和端口。”

所以 option 更像：

```text id="2pk8pn"
Option = “附加参数”
```

---

# 3. 为什么要分成两部分

因为在 SD 里，一条动作记录和它的地址参数，不一定想硬写死在一起。

所以协议把它拆成：

* **Entries Array**：只描述动作和对象
* **Options Array**：集中放地址、端口、配置
* Entry 再通过索引去引用 option。

这有点像数据库里的感觉：

* Entry 像主表
* Option 像附表
* 通过索引关联

---

# 4. 用一个最简单的例子理解

假设一个 SD 报文里有：

## Entries Array

```text id="5gtuph"
Entry[0] = OfferService
  service=0x1234
  instance=0x0001
```

## Options Array

```text id="igib1t"
Option[0] = IPv4 Endpoint
  ip=192.168.1.10
  port=30509
  proto=UDP
```

这两者连起来的意思才完整：

> “我提供 `0x1234/0x0001` 这个服务，地址在 `192.168.1.10:30509/UDP`。”

如果只有 entry，没有 option，你只知道“有这个服务”，但不知道去哪找它。
如果只有 option，没有 entry，你只看到一个 IP/端口，但不知道它是给哪个服务用的。

所以：

* **entry 定义语义**
* **option 提供细节**

---

# 5. 它们分别像什么

你可以这样记：

## Entries Array

像“目录项”
回答的是：

> **谁在干什么？**

## Options Array

像“附件区”
回答的是：

> **相关的地址、端口、补充配置是什么？**

---

# 6. 在订阅场景里再看一次

比如客户端要订阅事件：

## Entry

```text id="yl7jwg"
SubscribeEventgroup
  service=0x1234
  instance=0x0001
  eventgroup=0x0001
```

这表示：

> “我要订阅这个 eventgroup。”

## Option

```text id="g01g61"
IPv4 Endpoint
  ip=192.168.1.20
  port=40000
  proto=UDP
```

这表示：

> “以后事件请发到我这个地址。”

所以在订阅语境下：

* entry 说“我要订什么”
* option 说“订到了以后发到哪”

---

# 7. 一句话总结

**Entries Array 是“动作和对象的列表”。**
**Options Array 是“这些动作附带参数的列表”。**

再压缩成最短的一句：

**Entry 负责“说事”，Option 负责“补细节”。**
