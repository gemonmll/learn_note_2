在 **SOME/IP-SD** 里，**Flags** 可以理解成：

**“这整个 SD 报文的全局状态标记”**。

它不是某个具体 `Entry`（比如 Offer/Subscribe）自己的字段，
而是在 **SD Header 最前面**的一个 **8 bit 字段**，用来告诉对端一些“这条 SD 消息整体应该怎么理解”的信息。AUTOSAR 规定 SOME/IP-SD Header 以 8 位的 Flags 开始。([AUTOSAR][1])

---

## 1. Flags 在哪一层

SOME/IP-SD 报文大概可以这么看：

* SOME/IP Header
* **SD Header**

  * **Flags（8 bit）**
  * Reserved（24 bit）
  * Entries Array
  * Options Array

所以 Flags 是 **SD Header 级别**的信息，不是某个 service entry 里的字段。AUTOSAR 还规定 Flags 后面紧跟 24 位 Reserved。([AUTOSAR][1])

---

## 2. Flags 里有哪些位

在你现在常见的 AUTOSAR SOME/IP-SD 版本里，重点看这两位：

* **bit7（最高位）**：**Reboot Flag**
* **bit6（次高位）**：**Unicast Flag**
* 其余未定义位：发送时置 0，接收时忽略。([AUTOSAR][1])

也就是说你可以先把它记成：

```text
bit7   bit6   bit5..bit0
R      U      保留/未定义
```

---

## 3. Reboot Flag 怎么理解

这是最重要的那个位。

它的作用不是“这条消息是不是重启消息”，而是：

**“告诉对端：我这边是一次 reboot 之后的新状态周期。”**

AUTOSAR 规定：

* Reboot Flag 是 Flags 的最高位
* 节点在 reboot 后发送的 SD 消息里，要把这个位设为 1
* 一直设到 SOME/IP Header 里的 Session-ID 回卷后，再恢复为 0。([AUTOSAR][1])

### 它为什么存在

因为 SD 是基于 UDP 的，不可靠。
如果一个 ECU 重启了，它之前维护的状态可能都没了，比如：

* 提供过哪些服务
* 接受过哪些订阅
* 记住了哪些客户端

这时对端如果还按“旧状态”理解，就会出错。
所以 Reboot Flag 的意义就是：

**“我重启过了，你要把和我相关的旧状态当成失效。”**

AUTOSAR 在 CP 的 ServiceDiscovery 规范里明确说，检测到对端 reboot 后，应让由该对端控制的本地状态失效；比如 server reboot 时，client 要像收到了 `StopOfferService` 一样处理。([AUTOSAR][2])

---

## 4. Reboot Flag 不是单独用的，要和 Session ID 一起看

这个很关键。

SOME/IP-SD 不是只看 Reboot Flag=1 就说“对方刚重启了”，
而是要结合 **旧值** 和 **新的 Session ID** 来判断。

AUTOSAR 给出的 reboot 检测规则是：

* 如果 `old.reboot=0` 且 `new.reboot=1` → 检测到 reboot
* 或者 `old.reboot=1` 且 `new.reboot=1` 且 `old.session_id >= new.session_id` → 检测到 reboot。([AUTOSAR][1])

### 为什么要这样

因为 UDP 可能乱序、丢包。
如果你只看一个 Reboot Flag，很容易误判。

所以正确理解是：

**Reboot Flag 是“重启状态标识”，Session ID 是“顺序辅助判断”。**

---

## 5. Unicast Flag 怎么理解

这个位表示：

**“我支持通过 unicast 接收 SD 消息。”**

AUTOSAR 规定：

* Unicast Flag 是第二高位
* 对所有 SD 消息，这个位都应设置为 1
* 它主要是历史兼容遗留，实际用途已经很有限。([AUTOSAR][1])

所以在实际项目里你通常可以把它理解为：

* **基本恒为 1**
* 看到它更多是说明“实现兼容标准”
* 一般不会靠它做复杂业务逻辑

很多栈里你抓包时会发现这个位几乎一直开着，就是这个原因。([AUTOSAR][3])

---

## 6. 其它位怎么处理

对未定义位，规范要求很明确：

* **发送方**：置 0
* **接收方**：忽略。([AUTOSAR][1])

所以如果你自己写解析器，不要把这些保留位当业务含义。

---

## 7. 你可以把 Flags 理解成“报文级别的路牌”

很好记：

* **Entries** 告诉你“具体在说什么”

  * FindService
  * OfferService
  * SubscribeEventgroup
  * Ack/Nack
* **Flags** 告诉你“这整车消息在什么状态下发出来”

  * 我是不是 reboot 后的新状态
  * 我支不支持 unicast SD

也就是：

**Entry 管业务内容，Flags 管全局上下文。**

---

## 8. 举个最实用的例子

假设：

* ECU-A 是 server
* ECU-B 是 client
* B 之前已经订阅了 A 的 eventgroup

### 正常时

A 周期性发 `OfferService`。
比如某一帧：

* `Flags: Reboot=0, Unicast=1`
* `Entry: OfferService`

B 看见后，认为：

* A 没重启
* 服务正常续命

---

### A 突然重启了

A 重启后，内部订阅表、服务状态都重新初始化。
这时它再次发 SD 报文：

* `Flags: Reboot=1, Unicast=1`
* `Entry: OfferService`

B 收到后，不只是“又来了一个 Offer”，而是应该理解成：

**“A 是 reboot 后重新上线的，它以前相关的旧状态不能再直接相信。”**

因此 B 要把之前依赖 A 的旧状态清掉，再重新订阅、重新建立关系。
这就是 Reboot Flag 的价值。相关处理要求在 AUTOSAR CP ServiceDiscovery 中有明确说明。([AUTOSAR][2])

---

## 9. 一个抓包视角的直观理解

如果你抓到一个 Flags 字节，比如：

### `0xC0`

二进制是：

```text
1100 0000
```

表示：

* bit7 = 1 → Reboot Flag = 1
* bit6 = 1 → Unicast Flag = 1
* 其他位 = 0

含义就是：

* 发送方处在 reboot 后的阶段
* 支持 unicast SD

---

### `0x40`

二进制是：

```text
0100 0000
```

表示：

* Reboot Flag = 0
* Unicast Flag = 1

这通常就是“正常运行中”的常见状态。

---

## 10. 你前面提过 TTL，这里顺便区分一下

很多人会把 Flags 和 Entry 里的 TTL 混在一起。

它们不是一回事：

* **Flags**：报文级别的状态位
* **TTL**：Entry 级别的有效期字段

比如一个 `OfferService` entry 里会有 TTL，表示服务还能有效多久；
但 reboot/unicast 这些，不在 entry 里，而是在 SD Header 的 Flags 里。([AUTOSAR][1])

---

## 11. 关于 “Explicit Initial Data Control Flag” 你可能会看到旧资料

这个点容易踩坑。

如果你查到一些旧 AUTOSAR 或旧博客，可能会看到：

* Explicit Initial Data Control Flag
* Initial Data Requested Flag

但在较新的 AUTOSAR FO SOME/IP-SD 规范里，这两个已经被移除。R22-11 文档的修订说明里明确写了删除了这两个 flag。([AUTOSAR][1])

所以你现在如果是在看 **较新的 SOME/IP-SD**，理解 Flags 时重点就是：

* **Reboot**
* **Unicast**
* 其他位保留

---

## 12. 一句话总结

**SOME/IP-SD 的 Flags 是 SD Header 里的全局标志位。**
你主要这样记：

* **Reboot Flag**：告诉对端“我是不是 reboot 后的新状态周期”
* **Unicast Flag**：表示支持 unicast 接收 SD，基本总是 1
* **其它位**：保留，发 0，收时忽略。([AUTOSAR][1])
