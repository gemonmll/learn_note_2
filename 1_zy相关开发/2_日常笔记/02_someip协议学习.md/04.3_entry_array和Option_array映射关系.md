它们不是“按位置肉眼对齐”的，而是 **Entry 里用 4 个字段去引用 Options Array 里的连续片段**。

最核心的一句话：

**Entry 不直接装 Option 内容，只保存“从哪开始、取几个”的索引信息。**
AUTOSAR 规范明确说，entries 通过这些字段来引用 options：`Index First Option Run`、`Index Second Option Run`、`Number of Options 1`、`Number of Options 2`。其中索引是指向 **Options Array** 的位置，数量表示这个 run 里包含几个 option；索引 0 表示该 SOME/IP-SD 消息里的第一个 option。([AUTOSAR][1])

---

## 1. 先看 Entry 里哪几个字段负责“映射”

无论是 Service Entry 还是 Eventgroup Entry，都会带这 4 个字段：

```text
Index 1st options
Index 2nd options
# of opt 1
# of opt 2
```

规范定义里，它们的意思分别是：

* `Index First Option Run`：第一个 option run 的起始索引
* `Index Second Option Run`：第二个 option run 的起始索引
* `Number of Options 1`：第一个 option run 的 option 数量
* `Number of Options 2`：第二个 option run 的 option 数量。([AUTOSAR][2])

所以某个 Entry 实际上是在说：

> 我关联两段连续的 option 子数组：
> 第一段从 `Index 1st` 开始，长度 `#opt1`
> 第二段从 `Index 2nd` 开始，长度 `#opt2`

---

## 2. 什么叫 “option run”

这里的 **run**，你可以理解成：

> **Options Array 里的一个连续片段**

不是单个 option 的名字，也不是另一层协议对象，就是一段连续区间。

比如 Options Array 有 5 个 option：

```text
Option[0]
Option[1]
Option[2]
Option[3]
Option[4]
```

某个 entry 如果写：

```text
Index 1st = 1
#opt1     = 2
```

那它的第一段 option run 就是：

```text
Option[1], Option[2]
```

如果又写：

```text
Index 2nd = 4
#opt2     = 1
```

那第二段 option run 就是：

```text
Option[4]
```

---

## 3. 为什么要设计成“两段 run”

因为一个 Entry 可能要同时引用多种 option，而且协议允许它用两段连续片段来组织，而不是只允许一整段。

这些 option 可能包括：

* IPv4 Endpoint Option
* IPv6 Endpoint Option
* Multicast Option
* Configuration Option 等。Wireshark 的 SOME/IP-SD 字段也能看到这些 option 类型。([AUTOSAR][3])

所以你可以把 Entry 看成：

* 主体部分：我是谁、我要干什么（Find/Offer/Subscribe…）
* 引用部分：和我相关的地址、端口、配置参数在哪些 option 里

---

## 4. 一个最简单的映射例子

假设某个 SOME/IP-SD 报文里，Options Array 是：

```text
Options Array:
  [0] IPv4 Endpoint: 192.168.1.10:30509/UDP
  [1] Config Option: "profile=fast"
  [2] IPv4 Multicast: 239.1.1.1:5000/UDP
  [3] Config Option: "zone=front"
```

现在有一个 Entry 写成：

```text
Index 1st options = 0
# of opt 1        = 2
Index 2nd options = 2
# of opt 2        = 1
```

那它映射到的 option 就是：

### 第一段 run

从 `0` 开始，取 `2` 个：

```text
Option[0], Option[1]
```

### 第二段 run

从 `2` 开始，取 `1` 个：

```text
Option[2]
```

所以这个 Entry 最终关联的是：

```text
Option[0], Option[1], Option[2]
```

而 **Option[3] 不属于它**。

## 5. 再看你之前那个规范图里的 OfferService 例子

你前面提到过规范图 5.2 的例子。那里面第二条 Entry 是 OfferService，大概是这样：

```text
Type              = OfferService
Index 1st options = 0
Index 2nd options = 0
# of opt 1        = 1
# of opt 2        = 0
```

而该报文的 Options Array 里只有一个 option，也就是那个 IPv4 Endpoint。
所以这条 Entry 的映射关系就是：

* 第一段：从 Option[0] 开始，取 1 个 → 取到那个 IPv4 Endpoint
* 第二段：数量 0 → 没有第二段

所以它等价于说：

> 这条 OfferService 关联了 **第 0 个 option**，也就是它的 endpoint 信息。([AUTOSAR][2])

## 6. 为什么不直接把 option 写进 entry 里面

因为 SD 想做的是 **共享和复用**。

比如一个 SOME/IP-SD 报文里可以有多个 Entries，而这些 Entries 可能引用同一个或相近的一组 Options。把 Options 单独放成一个数组，再让 Entry 用索引去指，能减少重复。规范因此把 SD 负载分成：

* `Entries Array`
* `Options Array`。([AUTOSAR][2])

你可以把它想成：

* Entries Array：目录
* Options Array：附件池
* Entry 里的索引：附件引用关系

---

## 7. “一条 Entry 对一条 Option” 这个理解为什么不对

因为现实里可能有这些情况：

### 情况 A：一条 Entry 没有任何 option

例如某些 FindService 可能不带 option：

```text
Index1=0, #opt1=0
Index2=0, #opt2=0
```

这时它虽然索引写 0，但因为数量是 0，实际上**不引用任何 option**。([AUTOSAR][1])

### 情况 B：一条 Entry 关联多个 option

比如：

* 一个 IPv4 endpoint
* 一个 config option

那就是一个 Entry -> 多个 options。

### 情况 C：多条 Entries 可能引用同一片 options

协议之所以用“索引到 options array”这种方式，本身就允许引用关系独立于 entry 排列。规范也明确要求接收方检查 **entries 是否引用了存在的 options**，否则视为 malformed message。([AUTOSAR][3])

## 8. 你抓包时怎么快速判断映射关系

看到一条 Entry，直接读这四个字段：

```text
Index1, Num1, Index2, Num2
```

然后用下面这个公式去取：

```text
Run1 = Options[Index1 : Index1 + Num1]
Run2 = Options[Index2 : Index2 + Num2]
MappedOptions = Run1 + Run2
```

比如：

```text
Index1=3, Num1=2
Index2=0, Num2=1
```

那就是：

```text
Run1 = Option[3], Option[4]
Run2 = Option[0]
```

它不是按顺序重排成 0,3,4，而是逻辑上“这个 entry 关联了这两段 run”。

## 9. 一个完整一点的“发现 + 订阅”映射示例

假设一个 SD 报文里有这些 Options：

```text
[0] IPv4 Endpoint    192.168.1.10:30509/UDP
[1] Config Option    "proto=v1"
[2] IPv4 Endpoint    192.168.1.20:40000/UDP
[3] Config Option    "client=A"
```

### Entry A: OfferService

```text
Index1 = 0, Num1 = 2
Index2 = 0, Num2 = 0
```

它映射到：

```text
[0], [1]
```

意思就是：

* 服务端 endpoint 在 [0]
* 附加配置在 [1]

### Entry B: SubscribeEventgroup

```text
Index1 = 2, Num1 = 2
Index2 = 0, Num2 = 0
```

它映射到：

```text
[2], [3]
```

意思就是：

* 客户端接收事件的 endpoint 在 [2]
* 客户端附加配置在 [3]

这样你就看出来了：

* Entries 在前面定义动作
* Options 在后面提供地址/端口/配置
* 两者靠索引字段关联

## 10. 最容易错的三个点

### 10.1 索引 0 不一定表示“引用了 Option[0]”

只有当 `#opt > 0` 时才真的引用。
如果 `#opt = 0`，那就是**没有 option**，哪怕 index 写的是 0。([AUTOSAR][1])

### 10.2 Index 说的是 “第几个 option”，不是字节偏移

它是 **Options Array 中 option 项的序号**，不是字节地址。规范原文就是 “Index into array of options”。([AUTOSAR][1])

### 10.3 一个 entry 可以有两段 option run

不是只有一段。
所以看抓包时不要只盯着第一组 `Index/#opt`，还要看第二组。

---

## 11. 你可以这样死记一句

**Entry 不装 Option，只“指向” Option。**
**映射方式就是：两个起点 + 两个长度 = 两段连续 option run。**
