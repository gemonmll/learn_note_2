2，疑问：哪些信号是在设计阶段定义好的，哪些信号是服务发现的
**设计阶段定义**
Service ID
Instance ID（很多工程里也是静态配置好的）
Method ID
Event ID
Eventgroup ID
Major/Minor Version
某个 event 属于哪个 eventgroup
某个服务有哪些 method / event / field
某个 service instance 预期使用的端口、传输方式等部署参数。

**服务发现**
某个 service instance 当前在不在
它现在是否可用
它的 endpoint 是什么
IP 地址
端口
TCP/UDP 等 option
这个服务实例当前有没有在 offer
客户端发起的 eventgroup 订阅是否被接受
订阅的 TTL 还是否有效。

3)疑问： ttl在sd中有什么作用
在SOME/IP-SD 里，TTL 不是 IP 包里的那个“路由跳数” TTL，而是 Service Discovery 条目自身的“生存时间/租约时间”.

**server端:** 这个 OfferService 还能被认为有效多久,server 在声明自己的服务租约。客户端收到后，就按这个 TTL 计时；如果在到期前没再收到新的 Offer，就认为这个服务失效
**client端:** 这个 SubscribeEventgroup 订阅还能被认为有效多久。也就是 client 在申请一个订阅租约。server 收到后，如果接受订阅，就按这个 TTL 维护这条订阅；如果到期前 client 没续订，server 就把这条订阅删掉

**server 发 OfferService 且 TTL=0** 等价于 StopOfferService意思：我不再提供这个服务了
**client 发 SubscribeEventgroup 且 TTL=0**等价于 StopSubscribeEventgroup 意思：我不订了。

### 1 四个核心动作
1) Offer Service  、Find Service 、Subscribe Eventgroup 、Subscribe Eventgroup Ack / Nack

```cpp
完整链路
Server                               Client
|                                     |
|---- OfferService (multicast) -----> |
|                                     |
|<--- FindService (optional) -------- |
|                                     |
|<--- SubscribeEventgroup ----------- |
|                                     |
|---- SubscribeEventgroupAck -------> |
|                                     |
|==== 后续真正发 SOME/IP Event =====> |
```
---

**2) Offer Service**
作用:提供某个 service instance，你们可以来用我
someipy描述：服务提供方会周期性发送带 offer entry 的 SOME/IP-SD 消息，里面带上服务位置、service ID、instance ID、版本和 TTL

**3）Find Service**
作用：客户端如果急着找某个服务，而不是等服务端下一次 offer，可以主动发，核心功能

**4）Subscribe Eventgroup**
作用: 客户端发现服务后，如果它想接收事件，不是直接说“给我 event 0x8001”，而是通过 SD 发:我要订阅这个 service instance 下的这个 eventgroup。

**5) Subscribe Eventgroup Ack / Nack**
服务端收到订阅请求后，要告诉客户端：
Ack：订阅通过
Nack：订阅失败

### 2 SD 报文关系
SOME/IP-SD 报文格式按 3 层拆开看：
1）最外层：普通 SOME/IP 头
2）中间层：SD 自己的固定头
3）最内层：Entries 数组 + Options 数组 

---

```cpp
**SOME/IP Header**
  Service ID / Method ID (对 SD 来说是固定保留值组合)
  Length
  Client ID
  Session ID
  Protocol Version
  Interface Version
  Message Type
  Return Code

**SD Payload** 
  SD Header
  Entries Array
  Options Array
```
---

**1 最外层：SOME/IP-SD 本质上还是一个 SOME/IP 消息**
2.1 Message ID
对于 SD，它不是普通的 ServiceID + MethodID 业务意义，而是固定保留成 SD 专用值。Vector 和 someipy 都给出了 0xFFFF8100 这个值。

2.2 Length
表示后面 SOME/IP 负载长度，也就是 SD 内容的总长度。

2.3 Client ID / Session ID
它们仍然存在，因为 SD 还是 SOME/IP 消息。通常抓包时你也能看到。

2.4 Message Type
SOME/IP-SD 运行在 UDP 上，AUTOSAR 明确规定 SD 只使用 UDP

**3 中间层：SD 自己的固定头**

```cpp
**SD Header**
  Flags                1 byte
  Reserved             3 bytes
  Entries Length       4 bytes
  Entries Array        variable
  Options Length       4 bytes
  Options Array        variable
```
**3.1 flags**
在你现在常见的 AUTOSAR SOME/IP-SD 版本里，重点看这两位：
bit7（最高位）：Reboot Flag
bit6（次高位）：Unicast Flag
其余未定义位：发送时置 0，接收时忽略
