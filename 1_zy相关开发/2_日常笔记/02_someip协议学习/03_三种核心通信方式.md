## 三种核心通信方式
1)method request/response
2)method fire/forget
3)event/notify

AUTOSAR SOME/IP 规范里对这些消息类型有明确编码：REQUEST (0x00)、REQUEST_NO_RETURN (0x01)、NOTIFICATION (0x02)、RESPONSE (0x80)、ERROR (0x81)。

## 03.1 Request / Response
Client                          Server
  |                               |
  |---- REQUEST ----------------->|
  |       ServiceID               |
  |       MethodID                |
  |       RequestID               |
  |       Payload(参数)           |
  |                               |
  |<--- RESPONSE / ERROR ---------|
  |       同一个 RequestID        |
  |       Payload(返回值/错误)     |
  |                               |

Message ID 决定你调用的是哪个服务、哪个方法
Request ID 决定这条响应对应的是哪一次请求

## 03.2 fire/ forget
REQUEST_NO_RETURN (0x01)
Client                          Server
  |                               |
  |---- REQUEST_NO_RETURN ------->|
  |       ServiceID               |
  |       MethodID                |
  |       Payload                 |
  |                               |
  |       (处理，但不回包)         |
  |                               |

**与event的区别在于谁主动发起**

## 03.3 Event / Notification
服务端持续对外发布某种状态，客户端有兴趣就订阅。

SOME/IP 中，事件消息通常表现为 NOTIFICATION (0x02)

Client                          Server
  |                               |
  |--- 通过 SD 订阅 eventgroup --->|
  |                               |
  |<------ NOTIFICATION ----------|
  |         Event Payload         |
  |<------ NOTIFICATION ----------|
  |         Event Payload         |
  |<------ NOTIFICATION ----------|
  |         Event Payload         |

## 04.3 Event 和 Eventgroup 的关系

Event
是“真正传输的数据项”。
例如：
vehicle_speed_event
door_status_event

Eventgroup
是“订阅管理单元”。
客户端通常不是直接说：
我要订阅 event 0x8001
而是说：
我要订阅某个 eventgroup

该组里可以包含一个或多个 event。someipy 的文档专门强调了，eventgroup 主要用于服务发现阶段的订阅管理，而不是实际的事件数据本体。
