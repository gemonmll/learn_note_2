## 1. 先建立一张总图

假设有两个 ECU：

* **ECU-A**：摄像头/感知控制器，IP `192.168.10.10`，MAC `02:11:22:33:44:55`
* **ECU-B**：座舱/融合控制器，IP `192.168.10.20`，MAC `02:aa:bb:cc:dd:ee`

它们通过车载以太网交换机相连，在**同一个 IPv4 子网**里。对同一子网内的 IPv4 通信，发送方需要先把目标 IP 解析成目标 MAC，这正是 **ARP** 的作用；ARP 是把协议地址映射到以太网地址的协议。([RFC Editor][1])

典型顺序是：

1. ECU-A 先发 **ARP Request**
2. ECU-B 回 **ARP Reply**
3. 交换机在转发这些帧时，用收到帧的**源 MAC**来学习 MAC 表；如果目的 MAC 还不知道，就在同一 VLAN 内泛洪。([Cisco][2])
4. ARP 完成后，双方才能发 IPv4/UDP 或 IPv4/TCP 业务报文。以太网上承载 IPv4 时，EtherType 是 `0x0800`；ARP 的 EtherType 是 `0x0806`。([IETF][3])
5. 如果业务是 SOME/IP：

   * **SOME/IP-SD** 负责发现和订阅，且 **只跑 UDP**
   * **SOME/IP** 负责真正的方法调用和事件通知。([AUTOSAR][4])

## 2. 第一跳：为什么一开始经常先看到 ARP

ARP 本质上是“我知道对方 IP，但还不知道对方 MAC，所以先广播问一下”。RFC 826 定义了 ARP 的核心字段：硬件类型、协议类型、硬件地址长度、协议地址长度、操作码，以及发送方/目标方的硬件地址和协议地址。以太网 + IPv4 场景下，`HTYPE=1`、`PTYPE=0x0800`、`HLEN=6`、`PLEN=4`，操作码 `1` 是 request，`2` 是 reply。([RFC Editor][1])

### 2.1 ARP Request 抓包式例子

下面是一个**教学示例**，格式和字段都按 RFC 826：

**Ethernet II 头**

```text
Dst MAC   = ff:ff:ff:ff:ff:ff
Src MAC   = 02:11:22:33:44:55
EtherType = 0x0806   # ARP
```

**ARP 负载**

```text
HTYPE = 0x0001       # Ethernet
PTYPE = 0x0800       # IPv4
HLEN  = 6
PLEN  = 4
OPER  = 0x0001       # request

SHA   = 02:11:22:33:44:55     # Sender Hardware Address
SPA   = 192.168.10.10         # Sender Protocol Address
THA   = 00:00:00:00:00:00     # Target Hardware Address (request里未知)
TPA   = 192.168.10.20         # Target Protocol Address
```

它的人话就是：

> `192.168.10.10` 在问：
> “谁是 `192.168.10.20`？请把你的 MAC 告诉我。”

ARP request 中，目标硬件地址在请求里本来就是未知的，RFC 826 明确说这个字段在 request 里没有实际意义。([IETF Datatracker][5])

### 2.2 交换机此时做了什么

因为目的 MAC 是广播地址 `ff:ff:ff:ff:ff:ff`，交换机会把这帧在同一 VLAN 内泛洪出去。同时，交换机会根据收到帧的**源 MAC** `02:11:22:33:44:55` 学到：

```text
02:11:22:33:44:55  ->  端口 P1
```

Cisco 的交换机文档就是这么描述的：交换机用收到帧的源 MAC 动态建立 MAC 地址表；如果目的 MAC 不在表里，就在该 VLAN 内泛洪。([Cisco][2])

## 3. 第二跳：ARP Reply 回来后，二层单播就建立了

当 ECU-B 发现 TPA 是自己的 IP `192.168.10.20`，它会回 **ARP Reply**。RFC 5227 也明确写了：主机收到目标 IP 等于本机地址的 ARP Request 时，必须按 RFC 826 的方式回复 ARP Reply。([RFC Editor][6])

### 3.1 ARP Reply 抓包式例子

**Ethernet II 头**

```text
Dst MAC   = 02:11:22:33:44:55
Src MAC   = 02:aa:bb:cc:dd:ee
EtherType = 0x0806
```

**ARP 负载**

```text
HTYPE = 0x0001
PTYPE = 0x0800
HLEN  = 6
PLEN  = 4
OPER  = 0x0002       # reply

SHA   = 02:aa:bb:cc:dd:ee
SPA   = 192.168.10.20
THA   = 02:11:22:33:44:55
TPA   = 192.168.10.10
```

它的人话是：

> `192.168.10.20` 回答：
> “我是 `192.168.10.20`，我的 MAC 是 `02:aa:bb:cc:dd:ee`。”

同时交换机又会学到：

```text
02:aa:bb:cc:dd:ee  ->  端口 P2
```

这样，后续 ECU-A 发给 ECU-B 的以太网帧，就不需要广播了，可以直接二层单播。([Cisco][2])

## 4. ARP 完成后，真正的 IPv4 传输才开始

以太网上承载 IPv4 时，EtherType 必须是 `0x0800`，这是 RFC 894 定义的标准以太网封装。([IETF][3])

所以一个后续业务帧的外壳会长成这样：

```text
Ethernet II
  Dst MAC   = 02:aa:bb:cc:dd:ee
  Src MAC   = 02:11:22:33:44:55
  EtherType = 0x0800

IPv4
  Src IP    = 192.168.10.10
  Dst IP    = 192.168.10.20

UDP 或 TCP
  Src Port  = ...
  Dst Port  = ...

Payload
  = SOME/IP-SD 或 SOME/IP 或别的应用数据
```

也就是说，**ARP 是为后面的 IPv4 业务打基础**，不是业务本身。([IETF][3])

## 5. 车载 ECU 常见的下一步：先 SOME/IP-SD，再 SOME/IP

AUTOSAR 定义里：

* **SOME/IP** 支持 request/response、event notification 和序列化线格式
* **SOME/IP-SD** 负责服务发现和事件订阅控制
* **SOME/IP-SD 只跑 UDP**。([AUTOSAR][7])

someipy 文档给出的常见配置里，SD 默认多播地址是 `224.224.224.245`，端口是 `30490`。这个地址/端口是很多教学和原型环境里常见的。([Someipy][8])

## 6. 真实链路示例：ARP 之后做 SOME/IP-SD 服务发现

假设 ECU-B 是服务端，提供一个温度服务：

```text
Service ID   = 0x1234
Instance ID  = 0x0001
Eventgroup   = 0x0001
Event ID     = 0x8001
```

### 6.1 第一个 SD 报文：OfferService

这个报文从 ECU-B 发出。SOME/IP-SD 报文外层的 SOME/IP 头使用固定值：

* `Service ID = 0xFFFF`
* `Method ID = 0x8100`
* `Message Type = 0x02`。([AUTOSAR][4])

**教学示例**

```text
Ethernet II
  Dst MAC   = 01:00:5e:e0:e0:f5      # IPv4多播 224.224.224.245 对应的MAC示意
  Src MAC   = 02:aa:bb:cc:dd:ee
  EtherType = 0x0800

IPv4
  Src IP    = 192.168.10.20
  Dst IP    = 224.224.224.245
  Proto     = UDP

UDP
  Src Port  = 30490
  Dst Port  = 30490

SOME/IP Header
  Message ID        = 0xFFFF8100
  Protocol Version  = 0x01
  Interface Version = 0x01
  Message Type      = 0x02
  Return Code       = 0x00

SD Entry
  Type          = OfferService
  Service ID    = 0x1234
  Instance ID   = 0x0001
  Major Version = 0x01
  TTL           = 3
  Minor Version = 0x00000001

SD Option
  IPv4 Endpoint = 192.168.10.20:30509/UDP
```

这条报文的人话是：

> “我是服务 `0x1234.0x0001`，版本 1.1，
> 后续业务数据你可以发/收在 `192.168.10.20:30509/UDP`。”

SOME/IP-SD 的 Entry/Option 结构、OfferService 动作，以及 SD 只用 UDP，这些都来自 AUTOSAR SD 规范。([AUTOSAR][4])

---

## 7. 第二个 SD 报文：客户端订阅 SubscribeEventgroup

ECU-A 想接收 ECU-B 的温度事件，就会发 `SubscribeEventgroup`。AUTOSAR 的一些示例资料明确给出了“通知交互”链路：客户端通过 SOME/IP-SD 运行时注册，再由服务端发送事件。([SOME/IP][9])

**教学示例**

```text
Ethernet II
  Dst MAC   = 02:aa:bb:cc:dd:ee
  Src MAC   = 02:11:22:33:44:55
  EtherType = 0x0800

IPv4
  Src IP    = 192.168.10.10
  Dst IP    = 192.168.10.20
  Proto     = UDP

UDP
  Src Port  = 40000
  Dst Port  = 30490

SOME/IP Header
  Message ID        = 0xFFFF8100
  Message Type      = 0x02

SD Entry
  Type          = SubscribeEventgroup
  Service ID    = 0x1234
  Instance ID   = 0x0001
  Major Version = 0x01
  TTL           = 5
  Eventgroup ID = 0x0001

SD Option
  IPv4 Endpoint = 192.168.10.10:40000/UDP
```

它的人话是：

> “我要订阅 `0x1234.0x0001` 这个服务的 `eventgroup 0x0001`，
> 以后事件请发到 `192.168.10.10:40000/UDP`。”

客户端订的是 **eventgroup**，不是直接在 SD 层订单个 event；这点是 SOME/IP-SD 的核心设计。([AUTOSAR][4])

---

## 8. 第三个 SD 报文：SubscribeEventgroupAck

如果 ECU-B 接受订阅，它会回 `SubscribeEventgroupAck`。AUTOSAR 的 SOME/IP 示例文档里就有 `SubscribeEventgroup()` 和 `SubscribeEventgroupAck()` 的事件交互示意。([SOME/IP][9])

**教学示例**

```text
IPv4/UDP
  192.168.10.20:30490  ->  192.168.10.10:40000

SOME/IP-SD Entry
  Type          = SubscribeEventgroupAck
  Service ID    = 0x1234
  Instance ID   = 0x0001
  Major Version = 0x01
  TTL           = 5
  Eventgroup ID = 0x0001
```

它的人话是：

> “我同意你订阅 `eventgroup 0x0001`，5 秒内有效。”

---

## 9. 第四步：真正的 SOME/IP 事件报文来了

订阅建立后，后面发的已经不是 SD，而是普通 SOME/IP 事件通知。AUTOSAR SOME/IP 规范里明确，SOME/IP 支持 request/response 和 event notification。([AUTOSAR][7])

**教学示例**

```text
Ethernet II
  Dst MAC   = 02:11:22:33:44:55
  Src MAC   = 02:aa:bb:cc:dd:ee
  EtherType = 0x0800

IPv4
  Src IP    = 192.168.10.20
  Dst IP    = 192.168.10.10
  Proto     = UDP

UDP
  Src Port  = 30509
  Dst Port  = 40000

SOME/IP Header
  Service ID        = 0x1234
  Method/Event ID   = 0x8001
  Protocol Version  = 0x01
  Interface Version = 0x01
  Message Type      = 0x02   # Notification
  Return Code       = 0x00

Payload
  00 00 01 09   # 例如把 26.5°C 编成 265 = 0x00000109
```

它的人话是：

> “温度事件 `0x8001` 当前值是 26.5°C。”

SOME/IP 的 `NOTIFICATION` 消息类型是协议里定义好的。([AUTOSAR][7])

## 10. 如果不是订阅事件，而是方法调用

那链路会变成：

1. ARP
2. 可选的 SOME/IP-SD 找服务
3. 普通 SOME/IP Request
4. 普通 SOME/IP Response

AUTOSAR SOME/IP 规范支持 request/response；Wireshark 也有单独的 `someip` 和 `someipsd` 过滤器字段。([AUTOSAR][7])

一个方法调用的外层报文壳大概是：

```text
Ethernet II / IPv4 / UDP(or TCP) / SOME/IP / Payload
```

跟事件通知的差别主要在 SOME/IP 头里的 `Method/Event ID` 和 `Message Type`。([AUTOSAR][7])

## 11. 从“真实抓包”角度，你应该怎么读

你抓一条车载控制器以太网包时，可以按这个顺序看：

### 11.1 先看二层

* `Dst MAC`
* `Src MAC`
* `EtherType`
* 是否广播 / 单播 / 多播

如果是 `0x0806`，那是 ARP；如果是 `0x0800`，那是 IPv4。([IETF][3])

### 11.2 再看三层四层

* 源 IP / 目的 IP
* UDP 还是 TCP
* 端口号

SOME/IP-SD 是 UDP；常见示例端口是 30490。([Someipy][8])

### 11.3 最后看应用层

* `0xFFFF8100` 往往就是 SOME/IP-SD
* 普通 `Service ID != 0xFFFF` 的，大多是 SOME/IP 业务消息。([AUTOSAR][4])

---

## 12. 你最该建立的“车载以太网直觉”

把整条链路压缩成一句话：

> **二层先靠 ARP 建立“IP → MAC”的可达性，交换机靠源 MAC 学习转发表；
> 三层/四层再跑 IPv4 + UDP/TCP；
> 车载服务层常见再叠一层 SOME/IP-SD 做发现，再叠 SOME/IP 做真正业务。** ([RFC Editor][1])

---

## 13. 一个非常实用的学习顺序

你现在最适合这样学：

1. 先只抓 **ARP**

   * 看 request / reply
   * 看交换机如何从广播过渡到单播

2. 再抓 **SOME/IP-SD**

   * 看 OfferService
   * 看 SubscribeEventgroup / Ack

3. 最后抓 **普通 SOME/IP**

   * 看 Notification
   * 看 Request / Response

这样你会从“线缆上真的发生了什么”一路看到“服务层到底怎么运作”。([RFC Editor][1])
