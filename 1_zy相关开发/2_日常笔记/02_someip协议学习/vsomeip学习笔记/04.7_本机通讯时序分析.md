**先看总图**
这一周的 `hello_world` 用的是本地模式，`service-discovery` 也关了，所以你先把它理解成两条链就够了：

```text
控制面
hello_world_client  <---->  /tmp/vsomeip-0  <---->  routing manager(host 在 hello_world_service 进程里)

数据面
hello_world_client  <---->  /tmp/vsomeip-4444  <---->  hello_world_service
```

也就是说：

- `/tmp/vsomeip-0` 负责“注册、请求服务、下发可用性/routing info”
- `/tmp/vsomeip-4444` 负责“真正的 request/response 数据”
- `tmp_connection` 出现在 `accept()` 之后、正式 `endpoint` 建好之前，专门做握手和认人

**完整时序**
1. client 和 service 的业务类一创建，都会先拿 `runtime`，再创建 `application`，见 [hello_world_client.hpp](/home/jzm/workspace/vsomeip/examples/hello_world/hello_world_client.hpp:29) 和 [hello_world_service.hpp](/home/jzm/workspace/vsomeip/examples/hello_world/hello_world_service.hpp:33)。这里拿到的接口对象，真实实现其实是 `application_impl`。

2. 两边调用 `app_->init()` 后，会进入 [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:85)。这一步主要做配置加载、确定 client id、判断谁是 routing host、创建 routing 相关对象。因为 `helloworld-local.json` 里把 routing 指到了 service 进程，所以 service 这边会同时创建 routing host，而 client 只创建 proxy。

3. 两边再调用 `app_->start()`，真正把线程和 routing 跑起来，见 [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:314)。如果是 routing host，还会启动 routing root `/tmp/vsomeip-0`，见 [routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:75) 和 [endpoint_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_impl.cpp:803)。

4. 每个应用自己也会有一个本地监听口 `/tmp/vsomeip-<clientid>`，创建逻辑在 [endpoint_manager_base.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_base.cpp:227)。所以 service 会有 `/tmp/vsomeip-4444`，client 会有 `/tmp/vsomeip-5555`。

19. service 侧拿到 SOME/IP message 后，最终会进 `host_->on_message(...)`，再进 [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:1704)，最后分发到你的 [hello_world_service.hpp](/home/jzm/workspace/vsomeip/examples/hello_world/hello_world_service.hpp:98) `on_message_cbk()`。

21. service proxy 发送 response 时，会从 SOME/IP 头里读出目标 `client=5555`，然后找到刚才那条已建立的本地连接，把 response 发回去，见 [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:876)。

**你现在先抓住这 4 句话**
- `/tmp/vsomeip-0` 是控制面，负责注册和服务路由信息，不是第一次业务 request 的真正承载通道。
- `/tmp/vsomeip-4444` 才是这个例子里 client 到 service 的本地数据面通道。
- `tmp_connection` 是“正式 endpoint 之前的握手壳”，不是业务处理对象本身。
- 本地 IPC 上传的是 `CONFIG_ID / ASSIGN_CLIENT_ID / SEND_ID` 这类 `vsomeip` 内部命令；`SEND_ID` 里面才包着真正的 SOME/IP 报文。
