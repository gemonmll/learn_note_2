这两个点正好是 `vsomeip` 里最关键的“控制面”和“数据面”。

**1. client 为什么知道去 `/tmp/vsomeip-0` 注册**
先说结论：

**client 不是从 JSON 里读到一个字面量 `/tmp/vsomeip-0`，而是根据一套固定规则“算出来”的。**

这套规则是：

- `routing` 字段只决定“谁当 routing host”，比如这里是 `hello_world_service`，见 [helloworld-local.json](/home/jzm/workspace/vsomeip/examples/hello_world/helloworld-local.json:24)
- routing root 的 `client id` 是固定保留值 `0`，也就是 `VSOMEIP_ROUTING_CLIENT`
- `network` 默认是 `"vsomeip"`，见 [configuration_impl.cpp](/home/jzm/workspace/vsomeip/implementation/configuration/src/configuration_impl.cpp:65)
- base path 默认来自 `VSOMEIP_BASE_PATH`，未设置时走编译期默认值，通常是 `/tmp/`，见 [internal.hpp.in](/home/jzm/workspace/vsomeip/implementation/configuration/include/internal.hpp.in:22) 和 [internal.hpp.in](/home/jzm/workspace/vsomeip/implementation/configuration/include/internal.hpp.in:34)，以及 [utility.cpp](/home/jzm/workspace/vsomeip/implementation/utility/src/utility.cpp:35)
- 路径拼接规则就是 `base_path + network + "-" + client_id`，见 [utility.cpp](/home/jzm/workspace/vsomeip/implementation/utility/src/utility.cpp:273)

所以在这个例子里，routing root 就被算成：

```text
/tmp/ + vsomeip + - + 0
= /tmp/vsomeip-0
```

routing host 创建这个 root 的代码在 [endpoint_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_impl.cpp:806)。

client 侧开始注册时，会明确去创建一个“连到 routing client 0 的本地连接”，见 [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:2395)。  
而真正算出目标路径的地方在 [endpoint_manager_base.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_base.cpp:295) 到 [endpoint_manager_base.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_base.cpp:321)。

所以你可以这样记：

**`routing: hello_world_service` 决定“谁拥有 root”，但 client 注册永远是去连“约定好的 root=0”。**

**2. routing 怎么把 `service -> 4444` 告诉 client，client 又为什么会去连 `/tmp/vsomeip-4444`**
这里也先说结论：

**routing 不一定把字符串 `/tmp/vsomeip-4444` 直接发给 client。**  
在这个本地例子里，它更核心地发的是：

```text
[1111, 2222] 这个 service/instance 由 client 0x4444 提供
```

具体过程是：

- service `offer_service(1111, 2222)` 后，routing host 会记住“这个服务是 4444 提供的”，见 [routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:742)
- client `request_service(1111, 2222)` 后，routing 会给 requester 发 `routing_info`，如果服务早就存在就立刻发，见 [routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:1326)；如果服务是之后才 offer 的，就走 [routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:859)
- 这个 `routing_info_entry` 里最关键的是 `set_client(_hoster)`，也就是把 provider 的 `client id=4444` 告诉 client，见 [routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:871) 到 [routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:879)

client 收到后会更新两张表：

- `available_services_`：把 `(1111, 2222)` 映射到 provider `4444`
- `address_table_`：如果是远端 guest，还会记 IP/port

代码在 [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:1587) 到 [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:1599)。

之后你一旦 `send(request)`，proxy 会先根据 `service/instance` 找 provider client id：

- `find_local_client(1111, 2222) -> 4444`，见 [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:2550)

然后再去创建到 `4444` 的本地连接：

- `find_or_create_local_client(4444)`，见 [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:872) 到 [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:875)

而这个连接的路径又是按同一套规则本地推导出来的：

```text
/tmp/ + vsomeip + - + 4444
= /tmp/vsomeip-4444
```

路径计算代码还是 [endpoint_manager_base.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_base.cpp:295) 到 [endpoint_manager_base.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_base.cpp:321)。

所以这部分最准确的理解是：

- routing 告诉 client 的核心信息不是“路径字符串”
- 而是“这个 service 的 provider client id 是 4444”
- 在本地 UDS 模式下，client 再根据约定公式自己推出 `/tmp/vsomeip-4444`

你可以把整个过程压成这两句：

```text
注册阶段：所有 app 都先去找约定好的 routing root 0
发送阶段：client 先从 routing 学到 service 的 provider 是 4444，再自己推导出 /tmp/vsomeip-4444
```
