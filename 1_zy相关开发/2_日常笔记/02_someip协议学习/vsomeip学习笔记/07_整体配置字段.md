## 1. 先建立整体地图

vSomeIP 配置大概分成这些核心块：

```json
{
  "logging": {},
  "routing": "...",
  "applications": [],
  "unicast": "...",
  "services": [],
  "service-discovery": {},
  "security": {}
}
```

文档目录里也基本按这个顺序组织：Logging、Routing、Applications、Network、Security、Service Discovery、Services、Environment Variables 等。

你学习时不要一开始啃所有字段，先掌握 **5 个核心概念**：

| 概念                    | 作用                                    |
| --------------------- | ------------------------------------- |
| `applications`        | 定义本机有哪些 vSomeIP 应用，以及它们的 client ID    |
| `routing`             | 指定谁是 routing manager，负责本机应用之间的路由      |
| `unicast` / `network` | 定义本机 IP、诊断地址、client ID 生成规则           |
| `services`            | 定义本机提供或访问哪些 SOME/IP service           |
| `service-discovery`   | 定义 OfferService / FindService 的组播发现机制 |

---

## 2. `applications`：先理解“谁在用 vSomeIP”

`applications` 是本机使用这个配置文件的应用列表，每个应用有 `name` 和 `id`。文档里说 `name` 是应用名，`id` 是 client ID，而且通常高字节和 diagnosis address 相关。比如 diagnosis 是 `0x63` 时，合法 client ID 常见范围是 `0x6301` 到 `0x63FF`。

例子：

```json
"applications": [
  {
    "name": "service-sample",
    "id": "0x1277"
  },
  {
    "name": "client-sample",
    "id": "0x1344"
  }
]
```

可以理解为：

```text
service-sample 这个进程启动后，在 vSomeIP 里叫 0x1277
client-sample 这个进程启动后，在 vSomeIP 里叫 0x1344
```

这里的 `name` 很重要，因为启动时可以通过环境变量指定：

```bash
export VSOMEIP_APPLICATION_NAME=client-sample
```

文档说明 `VSOMEIP_APPLICATION_NAME` 用来把运行中的应用映射到配置文件里的 client id，而且它和二进制文件名无关。

---

## 3. `routing`：谁是本机的“总调度员”

vSomeIP 里有一个很关键的角色叫 **routing manager**。它负责本机多个 vSomeIP 应用之间的内部通信、服务路由、endpoint 管理。

最简单配置是：

```json
"routing": "service-sample"
```

意思是：`service-sample` 这个应用承担 routing manager。文档里给了这个 simple routing 配置例子。

如果是跨机器或内部 TCP 通信，就会变复杂：

```json
"routing": {
  "host": {
    "name": "service-sample",
    "unicast": "10.0.3.1",
    "port": "31000"
  },
  "guests": {
    "unicast": "10.0.3.1",
    "ports": [
      {
        "first": "1026",
        "last": "65535"
      }
    ]
  }
}
```

这里可以理解为：

```text
host  = routing manager 所在位置
guests = 其他应用连接 routing manager 时使用的地址/端口范围
```

文档里还特别提到，guest port range 需要给每个 client application 两个端口：一个收消息，一个发消息。

---

## 4. `network` / `unicast` / `diagnosis`

这部分决定本机在 SOME/IP 网络里的身份。

典型配置：

```json
"unicast": "10.10.3.1",
"netmask": "255.255.255.0",
"diagnosis": "0x63",
"diagnosis_mask": "0xF300"
```

文档说 `network` 是网络标识，会影响 `/tmp/vsomeip-$CLIENTID` 这类 Unix domain socket 名字；`diagnosis` 是诊断地址，会用于生成 client identifier 的高字节；`diagnosis_mask` 用来控制允许的 client ID 范围。

你可以先记住：

```text
unicast       = 本机 IP
diagnosis     = 本 ECU / 节点的诊断地址
diagnosis_mask = client ID 分配范围控制
```

---

## 5. `services`：最核心，定义服务

`services` 是服务提供方配置的重点。文档说明它包含 service id、instance id、协议、TCP/UDP 端口、事件、eventgroup 等。

一个典型 service：

```json
"services": [
  {
    "service": "0x1000",
    "instance": "0x0001",
    "unreliable": "30509",
    "reliable": {
      "port": "30510",
      "enable-magic-cookies": "false"
    }
  }
]
```

含义是：

```text
service id   = 0x1000
instance id  = 0x0001
UDP 端口      = 30509，也就是 unreliable
TCP 端口      = 30510，也就是 reliable
```

在 SOME/IP 里：

```text
reliable   = TCP
unreliable = UDP
```

如果是 **服务端**，这个配置表示“我提供这个 service，监听这些端口”。
如果是 **客户端**，配置里经常还要带远端 IP，例如：

```json
{
  "service": "0x1000",
  "instance": "0x0001",
  "unicast": "192.168.1.9",
  "unreliable": "30509"
}
```

文档示例里也强调，client side 访问外部服务时，`unicast` 用来标记远端服务地址。

---

## 6. `service-discovery`：服务发现

SOME/IP-SD 负责：

```text
服务端：周期性 OfferService
客户端：发送 FindService
```

典型配置：

```json
"service-discovery": {
  "enable": "true",
  "multicast": "239.192.255.251",
  "port": "30490",
  "protocol": "udp",
  "initial_delay_min": "10",
  "initial_delay_max": "10",
  "repetitions_base_delay": "30",
  "repetitions_max": "3",
  "cyclic_offer_delay": "1000",
  "ttl": "3"
}
```

文档说明 `service-discovery` 包含 SD 相关配置，比如是否开启、组播地址、端口、协议、初始延迟、重复发送次数、TTL 等；默认端口常见是 `30490`。

你可以这样记：

```text
multicast = SOME/IP-SD 组播地址
port      = SOME/IP-SD 端口，常见 30490
ttl       = offer/find entry 的有效时间
cyclic_offer_delay = 服务端周期 offer 间隔
```

## 7. 事件模型：`events` 和 `eventgroups`

SOME/IP 不只有 request/response，还有 notification。
事件要放到 eventgroup 里，客户端订阅 eventgroup，服务端发送 events。

文档里说明：

```text
events 可以被归到 eventgroup 里，客户端可以订阅 eventgroup，然后接收其中的 events。
```

还提到 `threshold` 控制什么时候用 unicast，什么时候用 multicast：如果 threshold 为 0，全部 unicast；如果 threshold 为 1，则事件会走 multicast。

例子：

```json
"events": [
  {
    "event": "0x8001",
    "is_field": "false"
  },
  {
    "event": "0x8003",
    "is_field": "false",
    "is_reliable": "true"
  }
],
"eventgroups": [
  {
    "eventgroup": "0x06",
    "multicast": {
      "address": "239.192.255.251",
      "port": "30506"
    },
    "threshold": "1",
    "events": [
      "0x800B"
    ]
  }
]
```

注意：SOME/IP 里事件 ID 常见在 `0x8000` 以上。

## 8. nPDU / debounce：降负载但增加延迟

文档里说 nPDU 也叫 **Zugverfahren**，作用是把多个 vSomeIP 消息合并到一个以太网帧里，以降低网络负载，但代价是速度/实时性下降。

两个核心参数：

```text
debounce-time
maximum-retention-time
```

文档解释：

```text
debounce-time = 同一个连接、同一个 method 的消息最小发送间隔
max-retention = 消息最多允许在发送端缓存多久
```

也就是说：

```text
debounce-time 越大，越容易合包，网络负载越低，但延迟越大
maximum-retention-time 是最大等待上限
```

示例：

```json
"debounce-times": {
  "requests": {
    "0x1001": {
      "debounce-time": "20",
      "maximum-retention-time": "200"
    }
  }
}
```

含义：

```text
对 method 0x1001 的 request：
最小发送间隔 20ms
最多缓存 200ms
```

文档还说明，如果方法级配置缺失，默认值是 `2ms debounce time` 和 `5ms max retention time`，也可以通过 `npdu-default-timings` 改全局默认值。

## 9. SOME/IP-TP：UDP 大包分片

普通 SOME/IP over UDP 有大小限制，文档说 SOME/IP-TP 可以传输超过 UDP payload 限制的消息；如果启用，消息会被分成多个 UDP datagram。

典型配置：

```json
"someip-tp": {
  "client-to-service": [
    "0x1", "0x2"
  ],
  "service-to-client": [
    "0x1", "0x2", "0x8001"
  ]
}
```

含义：

```text
client-to-service = 哪些 request 可以从客户端到服务端分片
service-to-client = 哪些 response/event/field 可以从服务端到客户端分片
```

文档示例里还配合了：

```json
"max-payload-size-unreliable": "5000"
```

表示 UDP 方向允许的最大 payload。

## 10. `security`：本地凭证 + 服务访问控制

vSomeIP 的安全机制基于 Linux/Unix credentials。文档说，如果启用，每个本地连接在 connect 时通过标准 UNIX credential passing 机制认证，客户端会把 client identifier 和 UID/GID 传给服务端，然后和配置策略匹配。

核心字段：

```json
"security": {
  "check_credentials": "true",
  "policies": [
    {
      "credentials": {
        "uid": "44",
        "gid": "any"
      },
      "allow": [
        {
          "requests": [
            {
              "service": "0x6731",
              "instance": "0x0001"
            }
          ]
        }
      ]
    }
  ]
}
```

这里含义是：

```text
UID=44 的本地应用，允许请求 service 0x6731 / instance 0x0001
```

文档还说明：

```text
allow = 白名单；空 allow 表示全部禁止
deny  = 黑名单；空 deny 表示全部允许
```

并且 `allow` 和 `deny` 不能在一个 policy 里同时使用。

初学阶段可以先不启用 security，等 request/response 和 event 跑通后再学。

## 11. 启动时最常用的环境变量

你最应该记住两个：

```bash
export VSOMEIP_APPLICATION_NAME=client-sample
export VSOMEIP_CONFIGURATION=./vsomeip.json
```

文档说明：

`VSOMEIP_APPLICATION_NAME` 指定应用名，用于在配置里找到对应 client id；`VSOMEIP_CONFIGURATION` 指定配置文件或配置目录，覆盖默认 `/etc/vsomeip.json`、`/etc/vsomeip`、`./vsomeip.json`、`./vsomeip` 等查找路径。

最小启动脚本类似：

```bash
#!/bin/bash
export VSOMEIP_APPLICATION_NAME=my_vsomeip_client
export VSOMEIP_CONFIGURATION=my_settings.json
./my_vsomeip_application
```

文档里也给了类似例子。

## 12. 推荐你按这个顺序学

### 第一阶段：能跑起来

目标：理解一个 client 和一个 service 怎么通信。

重点看：

```text
applications
routing
unicast
services
service-discovery
VSOMEIP_APPLICATION_NAME
VSOMEIP_CONFIGURATION
```

先不要碰：

```text
security
someip-tp
npdu
e2e
watchdog
partitions
acceptances
```

---

### 第二阶段：理解通信模型

重点搞清楚三种通信：

```text
method request/response
event notification
field get/set/notify
```

对应配置重点：

```text
services.service
services.instance
services.reliable
services.unreliable
events
eventgroups
threshold
```

---

### 第三阶段：调试和抓包

重点看：

```text
logging
tracing
service-discovery
payload-sizes
udp-receive-buffer-size
```

你要能从日志里判断：

```text
应用是否找到配置
routing manager 是否启动
服务是否 offer
客户端是否 find 到服务
request 是否发出
response 是否返回
eventgroup 是否订阅成功
```

---

### 第四阶段：高级特性

再看：

```text
security
someip-tp
npdu-default-timings
debounce-times
e2e
watchdog
```

这些属于工程化、量产化或性能优化阶段。

## 14. 一句话总结

你这份文档的核心不是“API 怎么写”，而是 **vSomeIP 运行时如何通过 JSON 把应用、服务、路由、网络、服务发现、安全策略串起来**。

学习主线应该是：

```text
应用是谁 applications
谁做路由 routing
本机地址是什么 unicast/network
提供/访问什么服务 services
服务怎么被发现 service-discovery
怎么启动环境变量 VSOMEIP_*
```
