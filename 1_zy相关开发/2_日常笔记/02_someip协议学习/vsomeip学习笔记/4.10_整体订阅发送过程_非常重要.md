把它当成三层链路，会最清楚：

```text
1. 本地控制面
client app <-> client proxy <-> 本机 routing host
service app <-> service proxy <-> 本机 routing host
承载：内部 command
例如 CONFIG / ASSIGN_CLIENT / REQUEST_SERVICE / SUBSCRIBE / SEND

2. 发现控制面
client routing host <-> SD UDP <-> service routing host
承载：SOME/IP-SD
例如 FindService / OfferService / SubscribeEventgroup / Ack

3. 业务数据面
client routing host <-> TCP/UDP endpoint <-> service routing host
承载：正常 SOME/IP
例如 request / response / notification
```

最关键的一点是：**app 自己不直接开远端 TCP/UDP socket**。  
远端网络 socket 基本都归 `routing host` 管；app 只通过本机 `proxy` 讲话。

**完整时序**

2. 本机 routing host 那边，`local_server` 早就在等本地 app 连进来，所以先挂的是 `async_accept`，accept 到连接后再对临时连接挂 `async_receive`。[local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:283) [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:300)

所以 `on_state` 的前提是：**本地 proxy <-> 本机 routing host 已经建好链路**。

4. `service app` 一般在 `init()` 里先 `offer_event(...)`，把“我有哪些 event / eventgroup”注册给本机 routing；真正把服务对外开放是在 `offer_service(...)`。[application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:1361)  
`offer_service` 到了 service 侧 routing host 后，会为本地服务创建真正的 server endpoint，也就是对外监听的 TCP/UDP 端口。[routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:1535) [endpoint_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_impl.cpp:296)

5. 这些 service server endpoint 一启动，就会把网络 I/O 挂到 `io` 上。  
TCP 服务端挂 `async_accept`，每个连接再挂 `async_receive/async_write`。[tcp_server_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/tcp_server_endpoint_impl.cpp:115) [tcp_server_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/tcp_server_endpoint_impl.cpp:129) [tcp_server_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/tcp_server_endpoint_impl.cpp:564)

6. 同时，routing host 的 SD 也启动。SD 自己有一个专门的 UDP endpoint，外加一堆 timer：`offer_debounce`、`find_debounce`、`main_phase`、`ttl`、`last_msg_received`。[service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:142)  
所以 SD 这条线上，`io` 挂的是：`async_receive_from` + 多个 `async_wait`。

7. `service app` 一旦 `offer_service`，service 侧 routing host 会把服务放入 SD offer 流程，后续由 SD timer 发 `OfferService`。这一步发出去的是 **SOME/IP-SD**，不是普通 SOME/IP request/response。

8. `client app` 这边通常会先 `request_event(...)` 和 `subscribe(...)`，再在 `on_state(ST_REGISTERED)` 里 `request_service(...)`。  
`request_event` 只是把本地 event 占位信息注册进 routing，不会直接发远端网络报文。[application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:1384)  
`subscribe` 也会先进入 `pending_subscriptions_`；只有服务真的 available 了，proxy 才会把订阅发出去。[routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:613)

9. `client app->request_service(...)` 到 client 侧 routing host 后，如果这是远端服务，就会触发 SD `FindService`。[routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:414)  
这时真正发包的是 client 侧 SD endpoint，不是 app，也不是 proxy。

10. client 侧一旦收到远端 `OfferService`，会进入 `add_routing_info(...)`，把远端 service 的 IP/port 存下来；如果本地已经请求过该 service，就立刻创建真正的 remote client endpoint。[routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:1616) [endpoint_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_impl.cpp:139)

11. 这一步非常关键：  
**SD 只负责告诉 client 侧 routing host“远端服务在哪儿”；真正把这条发现信息变成业务链路的，是 `find_or_create_remote_client(...)` 创建出来的 TCP/UDP client endpoint。**

**订阅时序**

14. client 侧真正发订阅时，第一跳还是本地 control plane。  
`routing_manager_client::send_subscribe(...)` 先构造本地 `SUBSCRIBE_ID`，优先看是不是本地 service；跨 ECU 时通常会发给本机 routing host 的 `sender_`。[routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:633)

**真正的 SOME/IP 报文阶段**

22. 订阅成功后，service app `notify(...)` 并不会走 SD；它走的是**正常 SOME/IP notification**。  
service app -> service proxy -> service routing host，然后 service routing host 按 eventgroup 的订阅目标，把 raw SOME/IP notification 发到远端 TCP/UDP endpoint。[routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:656)

23. client 侧 remote endpoint 收到 notification 后，client routing host 解包，再通过本地 `SEND_ID` 送回 client proxy；proxy 反序列化成 `message`，最后 `application_impl::on_message(...)` 入队，由 dispatcher 调你注册的 message handler。  
request/response 也是同理，只是 request 是 client app 主动 `send(request)`，response 再反向回来。

**这条链上 `io` 具体挂了什么**

- 本地 proxy endpoint：`async_connect`、`async_receive`、`async_send`，外加连接超时、assign timeout、重连 debounce timer。[local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:177)
- 本地 routing root：`async_accept`，accept 后的临时连接 `async_receive`，外加 watchdog/ping timer。[local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:283) [routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:1109)
- SD：UDP `async_receive_from`，以及 `offer/find/main/ttl/last_msg` 这些 `async_wait` timer。[service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:142)
- 远端业务 client endpoint：TCP 的 `async_connect/receive/write`，UDP 的 `async_connect/send/receive_from`。
- 本地 service server endpoint：TCP 的 `async_accept` 和 per-connection `async_receive/write`，UDP 的 `async_receive_from`。
- application 自己：`signals_.async_wait`、watchdog timer、dispatcher 阻塞检测 timer。[application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:287) [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:1906)

最后浓缩成一句：

**本地 proxy-host 链路承载内部 command，SD 链路承载发现和订阅控制，远端 TCP/UDP 链路承载真正的 SOME/IP request/response/notification；`io_context` 负责把这三类链路上的 connect/accept/receive/send/timer 全部跑起来，dispatcher 只负责最后把结果交给你的应用回调。**
