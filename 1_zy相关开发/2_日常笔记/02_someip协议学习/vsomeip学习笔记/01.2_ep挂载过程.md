这三段最好用同一个视角看：

**`start()` 里的同步调用，只是在把异步操作挂到 `io_context` 上；真正的通信，是后面 `io_.run()` 把这些 `async_*` 回调一个个跑起来。**

**1. proxy 连接本机 routing host**
如果当前 app 只是普通 app，`[application_impl::start](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:314)` 里最关键的是 `routing_->start()`；如果它同时又是 routing host app，还会先 `routing_app_->start()` 再 `routing_->start()`。

`routing_->start()` 的同步链是：

```text
application_impl::start
  -> routing_manager_client::start
  -> restart_sender
  -> ep_mgr_->create_local_client(VSOMEIP_ROUTING_CLIENT)
  -> endpoint_manager_base::create_local_client_unlocked
  -> local_endpoint::create_client_ep
  -> sender_->start()
  -> local_endpoint::start()
  -> connect_unlock()
  -> socket_->async_connect(...)   // 这里只是挂到 io
```

对应入口分别在 [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:314)、[routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:124)、[routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:2379)、[endpoint_manager_base.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_base.cpp:293)、[local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:83)、[local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:177)。

这里还有个关键细节：`create_local_client_unlocked()` 在 endpoint 还没真正连上之前，就已经把两条握手命令塞进发送队列了：

- `config_command`
- `assign_client_command`

见 [endpoint_manager_base.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_base.cpp:350)。

所以 `io` 真正跑起来后的异步链是：

```text
io_.run()
  -> async_connect 完成
  -> local_endpoint::connect_cbk()
  -> send_unlock()      // 把预先排队的 CONFIG / ASSIGN 发出去
  -> receive_unlock()   // 挂 async_receive，等 host 回包
```

`receive_unlock()` 和 `send_buffer_unlock()` 分别在 [local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:262) 和 [local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:281)。

所以这条链在 `io` 上最先挂的是：

- `async_connect`
- connect 成功后挂 `async_receive`
- 如果发送队列非空，再挂 `async_send`

**2. host 侧 accept、读握手、把临时连接变成真正 endpoint**
如果这是 routing host 进程，它的同步启动链是：

```text
application_impl::start
  -> routing_app_->start()
  -> routing_application::start()
  -> routing_manager_impl::start()
  -> stub_->start()
  -> uds_root_ / tcp_root_->start()
  -> local_server::start()
  -> start_unlock()
  -> acceptor_->async_accept(...)   // 这里只是挂到 io
```

入口在 [routing_application.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/routing_application.cpp:49)、[routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:201)、[routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:84)、[local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:40)、[local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:283)。

所以 host 这边最早挂到 `io` 的，是 `async_accept`。

一旦 proxy 连进来，异步链是：

```text
io_.run()
  -> local_server::accept_cbk()
  -> 创建 tmp_connection
  -> tmp_connection::async_receive()   // 先读握手命令
```

见 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:120) 和 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:300)。

`tmp_connection` 不是最终的 routing endpoint，它只是“握手阶段的临时壳”。它只接受两类命令：

- `CONFIG_ID`
- `ASSIGN_CLIENT_ID`

见 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:305)。

读到它们之后，链路会继续：

```text
tmp_connection::receive_cbk
  -> read_config_command()     // 记 hostname / expected_id
  -> assign_client()           // 给 client 分配真正 client id
  -> hand_over(client_id)
  -> local_server::add_connection(...)
  -> local_endpoint::create_server_ep(...)
  -> 回发 CONFIG
  -> 回发 ASSIGN_CLIENT_ACK
  -> connection_handler_(ep)   // 把真正 endpoint 交给 routing 管理
```

关键点在 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:355)、[local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:182)、[local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:250)。

所以 host 侧其实是两阶段：

- `tmp_connection` 先收握手
- `add_connection()` 再把 socket 接管成真正的 `local_endpoint`

这也解释了为什么 client 侧 connect 成功后，很快又会在 `async_receive` 里收到 host 回的 `CONFIG` 和 `ASSIGN_CLIENT_ACK`。

**3. SD 启动、UDP endpoint、timer**
SD 不是在 app 一 `start()` 就立刻开始工作的。更准确地说，是 routing host 进入 ready 后才 `discovery_->start()`：

[routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:2996)

之后同步链是：

```text
routing_manager_impl::discovery_->start()
  -> service_discovery_impl::start()
  -> host_->create_service_discovery_endpoint(...)
  -> routing_manager_impl::create_service_discovery_endpoint(...)
  -> ep_mgr_impl_->create_server_endpoint(sd_port, reliable, true)
  -> udp_server_endpoint_impl::start()
  -> receive_unicast_unlocked()
  -> 如有 multicast，再 join group
  -> start_main_phase_timer()
  -> start_offer_debounce_timer()
  -> start_find_debounce_timer()
  -> start_ttl_timer()
  -> start_last_msg_received_timer()
```

入口分别在 [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:142)、[routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:1442)、[endpoint_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_impl.cpp:296)、[udp_server_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/udp_server_endpoint_impl.cpp:196)、[udp_server_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/udp_server_endpoint_impl.cpp:227)、[udp_server_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/udp_server_endpoint_impl.cpp:279)。

所以 SD 这条线上，`io` 上真正挂的是两类东西：

- UDP 收包：`async_receive_from`
- 各种定时器：`offer/find/main/ttl/last_msg` 的 `async_wait`

然后 `request_service` 和 `offer_service` 本身，并不会立刻发网络包，它们先只是改 SD 内部状态：

- `request_service()` 先把请求记到 `requested_` [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:236)
- `offer_service()` 先把 offer 记到 `collected_offers_` [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2415)

真正发 `FindService` / `OfferService`，是在前面那些 debounce timer 到期时。

**一句话把三段串起来**

- `routing_->start()` 在 client/service app 这边挂出“连本机 routing host”的 `async_connect`
- `stub_->start()` 在 host 这边挂出“等本机 app 连进来”的 `async_accept`
- `discovery_->start()` 在 host 这边挂出“收发 SD UDP”和“一组 SD timer”

所以 `start()` 之后你看到的整个系统，本质上是三套异步源一起跑：

- 本地 IPC：`connect / accept / receive / send`
- SD：`udp receive_from + timers`
- 业务 endpoint：`tcp/udp connect / receive / write`
