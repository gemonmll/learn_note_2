关键点先说清楚：

**在“远端 ECU + 必经 SD”的场景里，`offer_service` 和 `request_service` 自己大多不是直接去 `async_connect` 远端。**  
它们更多是在改 routing / SD 的内部状态。真正往 `io` 上挂东西，主要分 4 类：

- 启动时就常驻挂好的：本地 `proxy <-> routing host`、SD endpoint、SD timers
- `offer_service` 时新增的：service 对外监听的 server endpoint
- 收到远端 `OfferService` 后新增的：client 侧 remote client endpoint
- 每次发一条命令/报文时临时触发的：`async_send` / `async_write`

你可以按这条线看。

**1. 启动阶段，哪些东西先挂到 `io` 上**

`application_impl` 持有唯一的 `io_` 和 `work_`，`start()` 时多个 `_ioXX` 线程一起跑 `io_.run()`。[application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:53) [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:314)

然后会先挂这几组“常驻事件”：

- `proxy -> 本机 routing host` 的本地 endpoint：`local_endpoint::start()` 先挂 `async_connect`，连上后挂 `async_receive`，发送时挂 `async_send`。[local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:83) [local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:177) [local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:262) [local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:281)
- 本机 routing host 的本地 root：`local_server` 先挂 `async_accept`，accept 后对临时连接挂 `async_receive`。[local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:283) [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:300)
- SD：`service_discovery_impl::start()` 里创建 SD endpoint，并启动 `offer/find/main/ttl/last_msg` 这些 timer。[service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:142)
- SD endpoint 本身是 routing host 创建的 UDP server endpoint。[routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:1442)

所以在你还没 `offer_service` / `request_service` 之前，`io` 上已经有：

```text
本地 control plane:
  async_connect / async_accept / async_receive / async_send

SD plane:
  async_receive_from
  offer_debounce async_wait
  find_debounce async_wait
  ttl async_wait
  main_phase async_wait
```

**2. `offer_service` 时，`io` 真正新增了什么**

`service app->offer_service()` 先走到 proxy，再发本地 `offer_service_command` 给本机 routing host。[application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:598) [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:298)

这里有个很重要的实现细节：

- proxy 侧只是把服务记到 `provided_services_ / pending_offers_`
- 如果已注册，就通过**已经存在的本地 local endpoint** 发一条控制命令
- 这一步通常**不会新建远端 socket**

真正新增的挂载发生在 host 侧 `routing_manager_impl::offer_service()`：

- 它会 `init_service_info(...)`
- 对本地 service 创建 server endpoint [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:1537)

如果 service 用 TCP：
- `create_server_endpoint(..., true, true)` 最后 `tcp_server_endpoint_impl::start()`
- `start()` 里挂 `async_accept` [endpoint_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_impl.cpp:296) [tcp_server_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/tcp_server_endpoint_impl.cpp:115)

如果 service 用 UDP：
- `udp_server_endpoint_impl::start()` 里 `start_unlocked()`
- `start_unlocked()` 立即调用 `receive_unicast_unlocked()`
- 里面挂 `async_receive_from` [udp_server_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/udp_server_endpoint_impl.cpp:196) [udp_server_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/udp_server_endpoint_impl.cpp:245) [udp_server_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/udp_server_endpoint_impl.cpp:279)

所以 `offer_service` 带来的**新增挂载**是：

```text
service 对外监听 endpoint
  TCP -> async_accept
  UDP -> async_receive_from
```

**3. `offer_service` 对 SD 做了什么，但没直接挂什么**

`routing_manager_impl::offer_service()` 会调用 `discovery_->offer_service(its_info)`。[routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:314)

但 `service_discovery_impl::offer_service()` 本身只是把服务放进 `collected_offers_`，**不当场发包，也不新挂 timer**。[service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2415)

真正发 `OfferService` 的，是**早就挂好的** `offer_debounce_timer_`：

- `start_offer_debounce_timer()` 用 `async_wait` 挂到 `io` 上 [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2507)
- 到时进入 `on_offer_debounce_timer_expired()`，这里才序列化并发送 `OfferService` [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2526)
- 发送后还会再创建一个新的 repetition-phase `steady_timer`，继续 `async_wait` 做重复广播 [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2557)

所以这一步最值得记住的是：

**`offer_service` 改的是 `collected_offers_`；真正把它变成网络报文的是已经挂在 `io` 上的 SD timer 回调。**

**4. `request_service` 时，`io` 真正新增了什么**

client 侧也是一样的套路。

`client app->request_service()`：
- `application_impl` 转给 proxy [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:608)
- `routing_manager_client::request_service()` 先把请求放进 `requests_` 或 debounce 集合
- 若已注册，发本地 `request_service_command` 给本机 routing host [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:428)

这一步也主要是走**已有的本地 local endpoint**，不是新建远端链路。

到 host 侧 `routing_manager_impl::request_service()` 后：
- 记住“这个 client 请求了这个 service”
- 如果是远端服务，调用 `discovery_->request_service(...)` [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:414)

而 `service_discovery_impl::request_service()` 自己只是把请求放进 `requested_`，**也不当场发包**。[service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:236)

真正发 `FindService` 的，是已挂好的 `find_debounce_timer_`：

- `start_find_debounce_timer()` 用 `async_wait` 挂到 `io` [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2434)
- 到时进 `on_find_debounce_timer_expired()`，这里才序列化并发送 `FindService` [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2457)
- 然后同样再建 repetition-phase timer，继续 `async_wait` [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2495)

所以：

**`request_service` 改的是 `requested_`；真正发 `FindService` 的也是早已挂好的 SD timer。**

**5. 什么时候才会新建远端业务 endpoint**

这是整个流程里最关键的“新增网络链路”点：

**不是在 `request_service()` 当场，而是在 client 侧收到远端 `OfferService` 之后。**

当 client 侧 SD endpoint 收到远端 `OfferService` 时，触发的是它**已经挂好的** UDP `async_receive_from` 回调。  
回调里最终会走到 `routing_manager_impl::add_routing_info(...)`。[routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:1616)

这里如果发现：
- 这个服务已经被 request 过
- 现在又拿到了远端 IP/port

就会调用：

- `ep_mgr_impl_->find_or_create_remote_client(...)`

这一步才真正新建 remote client endpoint，并 `start()`。[endpoint_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_impl.cpp:139)

如果是 TCP remote client：
- `start()` 后会挂 `async_connect`
- connect 成功后再挂 `async_receive`
- 发送时挂 `async_write` [tcp_client_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/tcp_client_endpoint_impl.cpp:240) [tcp_client_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/tcp_client_endpoint_impl.cpp:301)

如果是 UDP remote client：
- `start()` 后挂 `async_connect`
- 接收挂 `async_receive_from`
- 发送时挂 `async_send` [udp_client_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/udp_client_endpoint_impl.cpp:155) [udp_client_endpoint_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/udp_client_endpoint_impl.cpp:219)

所以远端 request/response 真正依赖的那条业务链路，是在这里才出现的：

```text
收到远端 OfferService
  -> add_routing_info
  -> find_or_create_remote_client
  -> async_connect
  -> async_receive
```

**6. 真正 `send(request)` 时，`io` 又发生了什么**

到这时，基础链路已经都有了：

- 本地 proxy <-> 本机 host：早就挂好
- SD endpoint + SD timers：早就挂好
- service 侧 server endpoint：offer 时挂好
- client 侧 remote client endpoint：收到 offer 后挂好

所以 `app->send(request)` 时，一般**不会再新增“长期监听型”的挂载**，而是用现有 endpoint 做一次发送：

- proxy 先通过本地 local endpoint 发内部 `SEND_ID`，这会触发一次 `async_send`
- 本机 routing host 再通过 remote TCP/UDP endpoint 发 raw SOME/IP，这会触发一次 `async_write` 或 `async_send`
- 对端 service 侧的 server endpoint 早就挂着 `async_accept/async_receive` 或 `async_receive_from`，因此能立刻收
- 收到后回调里又会**重新挂下一轮 receive**

所以每次 request/response 更像是：

```text
已有 receive 常驻挂着
+ 本次 send 时临时挂一次 write/send
+ 收到后回调里再续挂下一轮 receive
```

**7. 你可以把整个远端 `offer/request` 过程压成一张“挂载变化图”**

启动时常驻挂载：
- 本地 proxy：`async_connect / async_receive / async_send`
- 本地 routing root：`async_accept / async_receive`
- SD：`async_receive_from + 多个 async_wait`

`offer_service` 时新增：
- service server endpoint
- TCP: `async_accept`
- UDP: `async_receive_from`

`request_service` 时新增：
- 基本没有新的远端 socket
- 只是更新 `requested_`，等待已有 `find_debounce_timer` 回调发包

收到远端 `OfferService` 时新增：
- client remote endpoint
- TCP: `async_connect -> async_receive`
- UDP: `async_connect -> async_receive_from`

真正 `send(request)` 时：
- 只是在现有链路上触发一次 `async_send/async_write`
- 收包端继续复挂 `async_receive`

**一句话总结**

在远端 ECU + SD 场景下：

- `offer_service` 主要让 service 侧**新增 server endpoint 的 I/O 挂载**
- `request_service` 主要让 client 侧**修改 SD 请求状态**，并借助**已经挂好的** SD timer 发 `FindService`
- 真正新增 client 侧远端业务 I/O 挂载的时刻，是**收到远端 `OfferService` 后创建 remote client endpoint**
- 所以 `io` 的挂载不是“一次全挂完”，而是：
  **启动先挂本地 control + SD，offer 再挂 service server，收到 offer 再挂 remote client，之后每条 request/response 只是在这些现有挂载上跑。**
