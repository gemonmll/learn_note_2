# 第 5 周：进阶事件机制

这一周我们从“能订阅、能收到通知”继续往前走，重点学 5 个更贴近真实项目的能力：

1. `ET_SELECTIVE_EVENT` 和 `notify_one`
2. `subscription_status_handler`
3. multicast eventgroup 的设备级行为
4. `subscribe_with_debounce`
5. 可选深挖：异步订阅处理和 pending subscription

## 本周目标

学完这一周，你应该能把下面几句话讲清楚：

- selective event 不是 field，也不是普通广播 event
- `notify_one` 不是“本地只发给一个 handler”，而是完整的“按 client 定向分发”路径
- `subscription_handler` 和 `subscription_status_handler` 不是一回事
- multicast eventgroup 的加入/离开，很多时候是 routing manager 代表整台设备处理
- debounce filter 是订阅端的转发过滤，不是服务端的发送节流

## 先记住 5 个接口

### 1. selective 订阅仍然走 `subscribe`

`interface/vsomeip/application.hpp:351-352`

`subscribe(...)` 本身就允许你传一个具体 `_event`。文档里也写得很清楚：

- 如果指定了具体 event，应用只会接收这个 event
- 对外部服务来说，这个具体 event 主要用于接收过滤和 initial event 判断

### 2. 定向通知走 `notify_one`

`interface/vsomeip/application.hpp:448-449`

它和 `notify(...)` 的最大区别不是“多一个 client 参数”这么简单，而是：

- `notify` 面向这个 event 的全部订阅者
- `notify_one` 只面向某个具体 client

### 3. selective event 是单独的事件类型

`interface/vsomeip/enumeration_types.hpp:61`

枚举值里能看到：

- `ET_EVENT`
- `ET_SELECTIVE_EVENT`
- `ET_FIELD`

说明 selective 在 vSomeIP 里不是 field 的变种，而是独立语义。

### 4. 订阅状态监听走 `register_subscription_status_handler`

`interface/vsomeip/application.hpp:692-693`

这个 handler 的职责是：

- 告诉客户端这次订阅最终是 `OK (0x00)` 还是 `REJECTED (0x07)`

### 5. 带过滤订阅走 `subscribe_with_debounce`

`interface/vsomeip/application.hpp:961-962`

它的过滤结构是 `debounce_filter_t`：

`interface/vsomeip/structured_types.hpp:20-47`

里面最关键的几个字段是：

- `on_change_`
- `on_change_resets_interval_`
- `interval_`
- `ignore_`

## 本周第一主线：Selective Event 和 `notify_one`

### 1. 先看仓库里的完整例子

最推荐的入口是：

- `test/network_tests/second_address_tests/second_address_test_service.cpp`
- `test/network_tests/second_address_tests/second_address_test_client.cpp`

服务端在 `125-140` 同时 offer 了两种 event：

- 普通 event: `ET_EVENT`
- selective event: `ET_SELECTIVE_EVENT`

然后在 `143-159`：

- 对普通 event 用 `notify(...)`
- 对 selective event 用 `notify_one(...)`

客户端在 `74-90`：

- 对普通 event 调用 `request_event(..., ET_EVENT)`
- 对 selective event 调用 `request_event(..., ET_SELECTIVE_EVENT)`
- 然后分别订阅对应 eventgroup

这是第 5 周最值得你反复对照的一组代码。

### 2. selective eventgroup 的判断规则

`implementation/routing/src/eventgroupinfo.cpp:410-416`

这里能直接看到：

- selective eventgroup 必须只包含一个 event
- 且这个 event 的类型必须是 `ET_SELECTIVE_EVENT`

这意味着：

- 不是你“想 selective 就 selective”
- routing 层对 selective eventgroup 有明确结构要求

### 3. `notify_one` 真正怎么找目标

从应用层入口开始：

- `implementation/runtime/src/application_impl.cpp:884-889`

应用层只是把 payload 复制后交给 routing。

真正按 client 找目标端点的逻辑在：

- `implementation/routing/src/routing_manager_impl.cpp:930-960`

这里做了几件关键事：

1. 先找到 event
2. 再遍历 event 所属的 eventgroup
3. 再遍历这个 eventgroup 的 remote subscriptions
4. 只有 `has_client(_client)` 的 subscription 才会被选中
5. 最后把 payload 发给该 client 对应的 reliable/unreliable endpoint

一句话总结：

`notify_one` 不是“本地 handler 过滤”，而是 routing 基于订阅关系和目标 endpoint 的定向发送。

### 4. selective subscription 为什么在状态管理上要区分 client

这一点在 routing 里很关键：

- `implementation/routing/src/routing_manager_impl.cpp:3322-3329`

这里能看到：

- 如果 event 类型是 `ET_SELECTIVE_EVENT`
- `handle_subscription_state(...)` 会把 `_client` 本身作为状态 key

而不是像普通 eventgroup 那样只按 `VSOMEIP_ROUTING_CLIENT` 聚合。

对应地，SD 层也有 selective 分支：

- `implementation/service_discovery/src/service_discovery_impl.cpp:317-321`
- `implementation/service_discovery/src/subscription.cpp:40-45`

也就是说：

- 普通 eventgroup 常常按“设备级 / routing manager 级”管理订阅
- selective subscription 则需要保留“具体是哪个 client 订了它”

## 本周第二主线：`subscription_handler` 和 `subscription_status_handler`

这两个名字很像，但职责完全不同。

### 1. `subscription_handler`

第 4 周你已经见过它的基本职责：

- 服务端侧
- 决定接不接受订阅

### 2. `subscription_status_handler`

这是客户端侧的结果通知。

在 `application_impl` 里：

- `1106-1148`
  `on_subscription_status(...)`
- `1151-1313`
  `deliver_subscription_state(...)`
- `1315-1324`
  `register_subscription_status_handler(...)`

这条链的核心含义是：

- routing/SD 收到 ACK/NACK
- 应用内更新订阅状态
- 再把结果投递给你注册的 handler

### 3. `_is_selective` 这个参数什么时候有意义

接口文档里已经提示过：

- `interface/vsomeip/application.hpp:689-690`

如果你把 `_is_selective` 设为 `true`，那么在收到 `SUBSCRIBE_NACK` 时也会回调你的 handler。

而在实现里：

- `implementation/runtime/src/application_impl.cpp:1163-1173`

能直接看到：

- 没错误时一定可以投递
- 有错误时只有注册项标记了 `_is_selective` 才会进 handler

### 4. 一个很好的 selective 状态回调例子

看：

- `test/network_tests/second_address_tests/second_address_test_client.cpp:58-67`
- `test/network_tests/second_address_tests/second_address_test_client.cpp:158-176`

这个测试把：

- 普通 event 的状态回调
- selective event 的状态回调

分开注册、分开校验。

注意：

- 这个测试是通过给 callback 额外绑定 `true/false` 参数来区分“当前是在处理普通 event 还是 selective event”
- 它不是通过 `register_subscription_status_handler(..., _is_selective = true)` 这个 API 参数来开启 NACK 回调

这里非常适合你用来回答一个实际问题：

“为什么 selective 场景下更建议认真处理 subscription status，而不是只盯着 availability？”

### 5. 客户端为什么不会无脑重复 subscribe

`implementation/runtime/src/application_impl.cpp:2253-2294`

这里的 `check_subscription_state(...)` 做了两件很实用的事：

1. 如果当前已经在订阅或已经 ACK，不会重复发 subscribe
2. 如果已经 ACK 过，还会直接重新投递一次订阅状态给 handler

这能避免你在应用层重复触发同一轮订阅动作。

## 本周第三主线：Multicast Eventgroup

### 1. 先看这个测试想验证什么

`test/network_tests/multicast_group_tests/multicast_group_test.md`

这个测试的核心问题是：

- 同一台设备上有两个 client
- 它们都订阅了一个远端 multicast 通知
- 其中一个 client 退订后
- routing manager 不能因为“少了一个订阅者”就让整台设备离开 multicast group

### 2. 服务端怎么发

`test/network_tests/multicast_group_tests/multicast_group_test_service.cpp:24-30`

这里同时 offer 了：

- multicast service/event
- unicast service/event

然后在 `71-73` 同时对二者 `notify(...)`。

### 3. 客户端怎么订

`test/network_tests/multicast_group_tests/multicast_group_test_first_client.cpp:40-55`

客户端会：

- `request_service`
- `request_event`
- 在 availability 回调里对 multicast 和 unicast 都 `subscribe(...)`

### 4. 你应该理解的不是“多播地址”，而是“谁持有组成员关系”

这个测试真正想让你明白的是：

- 远端服务看到的不是“你这台设备上有几个进程”
- 往往是 routing manager 代表整台设备参与订阅和接收

因此：

- 一个本地 client 退订
- 不等于这台设备就该立刻离开 multicast group

只要同设备上还有别的 client 仍然依赖该 eventgroup，routing manager 就必须继续保留组成员关系。

## 本周第四主线：`subscribe_with_debounce`

### 1. 先看接口，不要先看配置

`subscribe_with_debounce(...)` 是 API 级能力，不只是 json 配置。

实现入口在：

- `implementation/runtime/src/application_impl.cpp:2554-2572`

它做的事情非常直接：

1. 检查是否需要发回 cached event
2. 检查订阅状态
3. 把 `debounce_filter_t` 包装成 `debounce_filter_impl_t`
4. 跟普通 `subscribe(...)` 一样交给 routing

### 2. 一个非常好的现成例子

客户端：

- `test/network_tests/debounce_filter_tests/debounce_filter_test_client.cpp:20-35`

这里能直接看到：

- `dBFilter.interval_ = interval`
- 先 `request_event(...)`
- 再 `request_service(...)`
- 再 `subscribe_with_debounce(...)`

服务端：

- `test/network_tests/debounce_filter_tests/debounce_filter_test_service.cpp:66-79`

服务端每 5ms/5ms 交替发两种 payload，也就是高频发送。

### 3. 这不是“服务端减速”，而是“客户端少收 / 少转发”

真正的过滤逻辑在 `event.cpp` 的 subscriber filter 里，最值得看的是：

- `implementation/routing/src/event.cpp:431-535`

这里会按 client 记录 filter，并用这些条件决定是否向该 client 转发：

- 是否发生变化 `on_change_`
- 时间间隔是否达到 `interval_`
- 某些字节或位是否忽略 `ignore_`

所以请一定记住：

- `subscribe_with_debounce` 不会让服务端少 `notify`
- 它改变的是“订阅侧最终被转发出去的频率和条件”

### 4. 一个容易忽略的小点

`interface/vsomeip/structured_types.hpp:47`

`send_current_value_after_` 这个字段在当前实现里是：

- `ignored, does nothing`

并且在 `event.cpp:454-456` 也会打印 unsupported 警告。

所以这个字段你先知道就好，不要把它当成熟能力来用。

## 这一周你必须分清的 5 组概念

### 1. `ET_EVENT` vs `ET_SELECTIVE_EVENT` vs `ET_FIELD`

- `ET_EVENT`
  面向普通通知广播
- `ET_SELECTIVE_EVENT`
  面向特定 client 的选择性通知
- `ET_FIELD`
  面向状态量，带“当前值”语义

### 2. `notify` vs `notify_one`

- `notify`
  发给这个 event 的全部订阅者
- `notify_one`
  只发给某个 client

### 3. `subscription_handler` vs `subscription_status_handler`

- `subscription_handler`
  服务端做“接不接受”
- `subscription_status_handler`
  客户端拿“最终结果”

### 4. multicast subscription vs 本地 client 数量

- 你看到的是多个本地 client
- 远端网络层很多时候看到的是一台设备上的 routing manager

### 5. debounce filter vs 服务端发送周期

- debounce filter 改的是接收侧转发策略
- 不等于服务端少发

## 一个很容易忽略但很重要的结论

### selective event 没有 field 的“当前值”语义

`implementation/routing/src/eventgroupinfo.cpp:425-453`

这里在发 initial events 时只会处理：

- `ET_FIELD`

不会处理 `ET_SELECTIVE_EVENT`。

所以：

- selective event 不是“只发给一个 client 的 field”
- 它没有 field 那种“新订阅后立即补当前值”的默认语义

这个区别非常值得你单独记下来。
