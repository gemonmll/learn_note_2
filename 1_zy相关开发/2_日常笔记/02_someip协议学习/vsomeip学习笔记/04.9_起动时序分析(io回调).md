我用一个最清楚的模型来梳理：

- `ECU-A` 上有一个 `service app`
- `ECU-B` 上有一个 `client app`
- 两边各自都有一个 local `routing manager`
- 这样一定会真实走 `SD`
- 我也会顺手说明：**如果 client 和 service 在同一个 routing manager 下，很多步骤会简化，不一定真的发 SD**

下面我分成 4 部分讲：

1. client / service 启动后各自在 `io` 上挂了什么  
2. `service` 怎么被 `SD` 广播出去  
3. `client` 怎么通过 `SD` 发现它  
4. 发现后真正的 SOME/IP request/response 怎么走

---

**1. client 和 service 各自在 `io` 上挂了什么**

先说一个总原则：

- `register_message_handler`
- `register_availability_handler`
- `register_state_handler`

这些**不是直接挂到 `io` 上的 socket 事件**，它们只是注册到 `application_impl` 内部表里，等事件来了再由 dispatcher 调。

比如：
- message handler 注册到 `members_` [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:1350)
- availability handler 注册到 `availability_` [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:913)

真正挂到 `io_context` 上的是这些异步源：

- `async_connect`：连 routing manager / 远端 endpoint
- `async_receive`：收本地或远端 socket 数据
- `async_accept`：接入站连接
- `async_wait`：各种 timer，像 keepalive、SD debounce、TTL

这些入口你已经见过了：

- `local_endpoint::async_connect` [local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:209)
- `local_endpoint::async_receive` [local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:262)
- `local_server::async_accept` [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:283)
- `keepalive_timer_.async_wait(...)` [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:232)
- SD 的 `find/offer/main/ttl` timers [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:189)

所以对一个普通 app 来说，`start()` 后它的 `io` 里至少会有：

- proxy 到 routing manager 的连接和收发
- keepalive timer
- 如果它是 routing host，还会有 routing stub 的 accept/receive
- 如果它启用了 SD，还会有 SD endpoint 的 receive 和若干 timer

**2. service 端时序：怎么“被发现”**

我用 `response-sample` 这个例子 [response-sample.cpp](/home/jzm/workspace/vsomeip/examples/response-sample.cpp:27)：

### `service app` 初始化阶段
它做了两件业务相关的事：

- 注册 `state_handler` [response-sample.cpp](/home/jzm/workspace/vsomeip/examples/response-sample.cpp:34)
- 注册 `message_handler(SERVICE, INSTANCE, METHOD)` [response-sample.cpp](/home/jzm/workspace/vsomeip/examples/response-sample.cpp:35)

这时还**没有**真正 offer service。

### `service app->start()`
`start()` 之后：
- app 连上本地 routing manager
- 注册成功后触发 `ST_REGISTERED`
- 用户 `on_state()` 被 dispatcher 调起来 [response-sample.cpp](/home/jzm/workspace/vsomeip/examples/response-sample.cpp:70)

然后示例里不是直接 `offer_service`，而是唤醒一个线程，那个线程再调用：

- `app_->offer_service(SAMPLE_SERVICE_ID, SAMPLE_INSTANCE_ID)` [response-sample.cpp](/home/jzm/workspace/vsomeip/examples/response-sample.cpp:60)

### `offer_service` 进入 routing manager
这会走到 `routing_manager_impl::offer_service(...)` [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:308)

核心动作有 3 个：

1. 在本地 routing 表里建立这个 service 的信息
2. `discovery_->offer_service(its_info)`，把它交给 SD [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:346)
3. `stub_->on_offer_service(...)`，通知本地请求者 [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:353)

### SD 端怎么处理 offer
`service_discovery_impl::offer_service()` 本身先只是把 offer 收集进 `collected_offers_` [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2415)

真正发出去靠 SD timer：
- `start_offer_debounce_timer(true)` 已经在 SD start 时挂上 [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:190)
- timer 到期后会把 offer entries 序列化并发送
- `insert_offer_service(...)` 构造 `OFFER_SERVICE` entry [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:745)
- 最终 `send(...)` 走 `host_->send(VSOMEIP_SD_CLIENT, m, true)` 发到 SD endpoint [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2250)

所以 service 端“被发现”的本质是：

```text
offer_service
  -> routing_manager_impl 记本地服务
  -> discovery_->offer_service 收集
  -> offer debounce timer 到期
  -> 发送 SD OfferService 报文
```

**3. client 端时序：怎么通过 SD 发现 service**

看 `request-sample` [request-sample.cpp](/home/jzm/workspace/vsomeip/examples/request-sample.cpp:28)

### `client app` 初始化阶段
它做了 3 件业务相关的事：

- 注册 `state_handler` [request-sample.cpp](/home/jzm/workspace/vsomeip/examples/request-sample.cpp:37)
- 注册 `message_handler`，用于收 response [request-sample.cpp](/home/jzm/workspace/vsomeip/examples/request-sample.cpp:39)
- 注册 `availability_handler`，用于知道服务何时可用 [request-sample.cpp](/home/jzm/workspace/vsomeip/examples/request-sample.cpp:53)

### `client app->start()`
和 service 一样，先连本地 routing manager，注册成功后用户 `on_state(ST_REGISTERED)` 被触发 [request-sample.cpp](/home/jzm/workspace/vsomeip/examples/request-sample.cpp:81)

然后示例里调用：

- `app_->request_service(SAMPLE_SERVICE_ID, SAMPLE_INSTANCE_ID)` [request-sample.cpp](/home/jzm/workspace/vsomeip/examples/request-sample.cpp:83)

### `request_service` 进入 routing manager
这会走到 `routing_manager_impl::request_service(...)` [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:414)

这里有个非常关键的分叉：

- **如果 service 是本地的**：不会触发 SD find [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:426)
- **如果 service 是远端的**：`discovery_->request_service(...)`，开始发 `FindService` [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:425)

所以如果你真要理解“SD 发现”，要假设它们在两个 ECU 上。

### SD 端怎么发 FindService
`service_discovery_impl::request_service()` 会先把请求记到 `requested_` [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:236)

真正发包靠 `find debounce timer`：
- `start_find_debounce_timer(true)` 在 SD start 时已经挂上 [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:191)
- timer 到期后 `insert_find_entries(...)` 构造 `FIND_SERVICE` entries [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:704)
- 然后 `send(its_messages)` 发送 [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:2484)

### service 端 SD 收到 FindService
service 侧 SD endpoint 收到 SD 报文后，会走：

- `service_discovery_impl::on_message(...)` [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:1029)

里面会解析 entries：
- 遇到 `FIND_SERVICE` / `OFFER_SERVICE` 都会进对应处理逻辑 [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:1321)

如果是 `OFFER_SERVICE`，最终会 `host_->add_routing_info(...)` [service_discovery_impl.cpp](/home/jzm/workspace/vsomeip/implementation/service_discovery/src/service_discovery_impl.cpp:1482)

### client 端 routing manager 收到远端 OfferService
客户端本地 routing manager 在收到远端 `OfferService` 后，会进：

- `routing_manager_impl::add_routing_info(...)` [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:1616)

这里会做几件关键事：

1. 建立远端 service info
2. 建立远端 endpoint 信息
3. 如果这个服务已经被本地 app 请求过，就创建到远端 service 的连接
4. 连接建立后，把服务标记为 available

关键代码点：
- `ep_mgr_impl_->find_or_create_remote_client(...)` [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:1678)
- 然后 `on_availability(... AS_AVAILABLE ...)` [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:1697)

### availability 怎么回到 client app
这时 routing manager 会通过 stub/proxy 把 routing info / availability 送到 client app 那边，最后到 `application_impl::invoke_availability_handler(...)` [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:920)

它会把 availability handler 封成任务，丢进 dispatcher 队列 [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:944)

于是用户的：

- `on_availability(service, instance, true)` 被真正执行 [request-sample.cpp](/home/jzm/workspace/vsomeip/examples/request-sample.cpp:87)

示例里在这里调用 `send()`，然后异步线程里真正 `app_->send(request_)` [request-sample.cpp](/home/jzm/workspace/vsomeip/examples/request-sample.cpp:117)

**4. 发现后真正的 SOME/IP request / response 怎么走**

这一步已经不是 SD 了，而是正常的 SOME/IP 数据面。

### client 发 request
`app_->send(request_)` [request-sample.cpp](/home/jzm/workspace/vsomeip/examples/request-sample.cpp:124)

走到：
- `application_impl::send(...)`
- `routing_manager_client::send(std::shared_ptr<message>)` [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:2599)

proxy 会：
1. 把 message 序列化成 SOME/IP 字节流
2. 再包装成内部 `SEND_ID / send_command`
3. 找目标 endpoint
4. 发出去

如果目标是远端 service，就走远端 endpoint / routing path；  
如果是本地 service，有时会直接走本地 endpoint。

### service 收到 request
service 侧 proxy / routing 解包后，最终会：

- `host_->on_message(std::move(its_message))` [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:1132)

这里的 `host_` 就是 service app 的 `application_impl`。

然后 `application_impl::on_message(...)`：
- 找到你注册的 `message_handler`
- 把它封装进 dispatcher 队列
- dispatcher 真正执行 service 的 `on_message(_request)` [response-sample.cpp](/home/jzm/workspace/vsomeip/examples/response-sample.cpp:85)

### service 发 response
service 的 `on_message` 里创建 response，然后：

- `app_->send(its_response)` [response-sample.cpp](/home/jzm/workspace/vsomeip/examples/response-sample.cpp:98)

之后路径和 request 对称，只是这次 message type 是 response。

### client 收到 response
client 侧 `application_impl::on_message(...)` 再把用户 response handler 入队，最后执行：

- `request-sample` 里的 `on_message(_response)` [request-sample.cpp](/home/jzm/workspace/vsomeip/examples/request-sample.cpp:101)

**把整个时序压成一张图**

假设跨 ECU，真实走 SD：

```text
Service App (ECU-A)
  start
  -> ST_REGISTERED
  -> offer_service
  -> local routing manager 记本地服务
  -> SD 发送 OfferService

Client App (ECU-B)
  start
  -> ST_REGISTERED
  -> request_service
  -> local routing manager 记 requested service
  -> SD 发送 FindService

Client routing manager / SD
  <- 收到远端 OfferService
  -> add_routing_info
  -> 建立到远端 service endpoint
  -> AS_AVAILABLE
  -> client availability handler 被 dispatcher 执行

Client App
  on_availability(true)
  -> send(request)

Service App
  on_message(request)
  -> send(response)

Client App
  on_message(response)
```

---

**特别提醒：同机时序不一样**

如果 client 和 service 都在**同一个 routing manager** 下：

- `service offer_service` 后，`stub_->on_offer_service(...)` 会直接 `inform_requesters(...)` [routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:742)
- client 可能通过本地 `routing_info_command` 就知道服务可用了
- `request_service` 也可能根本不触发 `SD find`

也就是说：

**同机主要靠本地 routing info；跨机才主要靠 SD。**
