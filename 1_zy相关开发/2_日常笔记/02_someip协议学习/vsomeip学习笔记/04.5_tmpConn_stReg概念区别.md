不是“server 作为业务角色多一个 `tmp_connection`”，而是：

**谁在“被动 accept 连接”，谁就会先有一个 `tmp_connection`。**

这里要把两件事彻底分开：

- `ST_REGISTERED`：应用层/中间件层的“注册到 routing manager”
- `tmp_connection`：传输层/本地 socket 层的“临时握手连接对象”

它们不是一回事。

**`tmp_connection` 是干什么的**
源码注释其实写得很直白：

- [local_server.hpp](/home/jzm/workspace/vsomeip/implementation/endpoints/include/local_server.hpp:123)  
  `Creates a tmp_connection to perform handshake before creating local_endpoint.`
- [local_server.hpp](/home/jzm/workspace/vsomeip/implementation/endpoints/include/local_server.hpp:147)  
  它是一个 temporary wrapper，用来先做握手，再把连接转交给真正的 `local_endpoint`

也就是说：

**刚 `accept()` 到一个 socket 时，vsomeip 还不知道对方到底是谁，所以不能立刻把它当成正式连接。**

于是先放进 `tmp_connection` 里，做这几件事：

- 读第一条内部握手消息
- 确认对方的 `client id`
- 读 `hostname`
- 检查 `expected_id`
- 检查 IPC 协议版本
- 如果是 routing manager，还要给对方分配/确认 client id

握手完成后，再调用 `hand_over(...)`，把它升级成正式 `local_endpoint`，见 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:404)。

**为什么 accept 侧需要它，而主动 connect 的 client 侧不需要**
因为主动发起连接的一侧，已经知道自己要连谁了。

比如 `hello_world_client` 发请求时，日志是：

- [client.txt](/home/jzm/workspace/vsomeip/examples/hello_world/client.txt:27)
- [client.txt](/home/jzm/workspace/vsomeip/examples/hello_world/client.txt:28)

它已经明确知道：

- 目标 client 是 `0x4444`
- 目标路径是 `/tmp/vsomeip-4444`

所以主动连接方可以直接创建“正式的 client endpoint”。

但服务端那边在 `accept()` 的瞬间，只拿到一个“裸 socket”，它还不知道：

- 这是哪个 client 连进来的
- 对方是不是说自己是 `0x5555`
- 这条连接是不是发给我这个 endpoint 的
- 协议版本对不对

所以它必须先有个 `tmp_connection` 做一次入场检查。

这点在代码里也很清楚：

- `accept_cbk()` 里先创建 `tmp_connection`，见 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:130)
- `tmp_connection::receive_cbk()` 先看第一条命令是什么，见 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:319)

**握手时到底读了什么**
有两种主要情况：

1. 如果接受方是 routing manager
   它期待先收到 `ASSIGN_CLIENT_ID`
   见 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:320)

   然后它会：
   - 反序列化 `assign_client_command`
   - 申请/确认 client id
   - 记录对方地址和端口  
   见 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:352)

2. 如果接受方是普通应用 endpoint
   它期待先收到 `CONFIG_ID`
   见 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:324)

   然后它会：
   - 检查 IPC 版本
   - 读取 `expected_id`
   - 读取 `hostname`  
   见 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:372)

**所以你看到的“server 多一个 tmp_connection”，本质原因是**
不是因为它是 request/response 里的 server，  
而是因为它是这条 socket 的 **accept 侧**。

**在 `hello_world` 里怎么体现**
客户端这边主动连服务端：

- 客户端发起连接到 `/tmp/vsomeip-4444`  
  见 [client.txt](/home/jzm/workspace/vsomeip/examples/hello_world/client.txt:27) 到 [client.txt](/home/jzm/workspace/vsomeip/examples/hello_world/client.txt:30)

服务端这边被动接收，握手完成后变成正式 endpoint：

- [service.txt](/home/jzm/workspace/vsomeip/examples/hello_world/service.txt:106)

同样，routing manager 接收 client 注册时，也是在 accept 侧先走这套逻辑：

- [service.txt](/home/jzm/workspace/vsomeip/examples/hello_world/service.txt:102)
- [service.txt](/home/jzm/workspace/vsomeip/examples/hello_world/service.txt:103)
- [service.txt](/home/jzm/workspace/vsomeip/examples/hello_world/service.txt:104)

**还有一个容易忽略的点**
`hello_world_client` 自己其实也创建了一个本地 server endpoint：

- `/tmp/vsomeip-5555`

这说明从传输层看，client 进程也不是“纯客户端”。  
谁来连它，它也会变成 accept 侧，也一样会走 `tmp_connection` 这套逻辑。

所以更准确的说法是：

- 业务语义上：它是 client / server
- 传输语义上：它可能同时既有 outgoing client endpoint，也有 incoming server endpoint

**一句话总结**
`tmp_connection` 不是“server 特权”，而是 **accept 侧的临时握手壳**。  
它的任务是：**先认人、验身份、读配置，再把裸 socket 交给正式 endpoint。**
