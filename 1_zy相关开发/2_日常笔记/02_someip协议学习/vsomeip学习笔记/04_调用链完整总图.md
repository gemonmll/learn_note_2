**总图**
先记这两个事实：

- **每个 app 进程里都有一个 routing proxy**
- **整台 ECU 上通常还有一个中央 routing host**

所以完整关系不是“只有一个 router”，而是这样：

```text
ECU 内

[client app 进程]
app
-> routing proxy
-> 本地 client endpoint / socket
                         \
                          \  本机直连数据通道
                           \
[target app 进程]           [中央 routing host 进程]
local_server                routing host
-> acceptor                 -> routing_manager_stub / impl
-> server-side socket       -> root local_server
-> local_endpoint           -> acceptor
-> routing proxy            -> socket
-> app
```

**这几个角色分别是什么**
- `app`
  你的业务代码，比如 `hello_world_client`、`hello_world_service`。
- `routing proxy`
  每个 app 进程里都有，负责和 routing 交互，也负责把收到的消息往上交给 app。应用里统一通过 `routing_` 调用，见 [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:598)。
- `routing host`
  ECU 内的中央中枢，维护“哪个 service 在哪个 client 上”，决定本地还是远端。host/proxy 的实例化在 [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:245) 和 [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:253)。
- `local_server`
  某个进程对外提供“本地接入入口”的对象。
- `acceptor`
  只负责监听和 `accept` 新连接。
- `socket`
  代表“一条已建立连接”，后续收发都靠它。
- `local_endpoint`
  比 socket 更上一层，负责消息边界、缓冲区、把完整消息交给 routing。

**控制面：一定经过中央 routing host**
这个路径是：

```text
client app
-> client 进程里的 routing proxy
-> sender_ 连接中央 routing host
-> host 的 root local_server
-> host 的 acceptor / socket
-> routing_manager_stub / routing_manager_impl
```

它负责的事情是：

- app 注册
- `request_service`
- `offer_service`
- 同步“本机哪个 client 提供了哪个 service”
- SD 相关控制

关键代码：
- app 启动时创建 routing proxy [application_impl.cpp](/home/jzm/workspace/vsomeip/implementation/runtime/src/application_impl.cpp:253)
- proxy 连中央 routing host：`sender_ = create_local_client(VSOMEIP_ROUTING_CLIENT)` [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:2395)
- host 侧 root 初始化 [routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:719)
- root 启动 [routing_manager_stub.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_stub.cpp:84)

**数据面：本地服务时，常常不再经过中央 routing host**
这就是你前面一直困惑的地方。

当中央 routing host 已经知道：

- `service + instance` 是本地服务
- 它属于哪个本地 client

发送端 proxy 就会直接连目标 app，而不是每条 payload 都继续绕中央 host。

路径变成：

```text
client app
-> client 进程里的 routing proxy
-> 直连 target app 的 local_server
-> target app 的 acceptor accept
-> server-side socket
-> target app 的 local_endpoint
-> target app 进程里的 routing proxy
-> target app
```

关键代码：
- request 时先查本地目标 client [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:869)
- 找到了就直连 [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:874)
- 如果找不到直连目标，再退回发给中央 routing host [routing_manager_client.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_client.cpp:900)

**为什么你前面会觉得“这里面没有 router”**
因为你看到的那条：

```text
client local_endpoint
-> server local_endpoint
-> mock_routing_host
```

其实只画了“接收端半边栈”。

它对应的是：

```text
socket 收到字节
-> local_endpoint 组装完整消息
-> 往上交给 routing
```

在真实系统里，完整路径应该补上发送端 proxy 和中央 routing host 的前置控制流程。  
只是进入“本地数据直连”阶段后，中央 routing host 不再站在 payload 正中间搬数据了。

**acceptor 和 router 的关系**
这两个一定要分开：

- `acceptor` 是连接接入工具
- `router` 是消息分发中枢

关系是：

```text
router 可能拥有 local_server
local_server 内部有 acceptor
acceptor accept 出 socket
socket 交给 endpoint
endpoint 再把完整消息交给 router
```

比如本地 UDS 接入这段：
- 创建本地 server [endpoint_manager_base.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/endpoint_manager_base.cpp:227)
- `acceptor` 开始 `async_accept` [local_acceptor_uds_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_acceptor_uds_impl.cpp:96)
- accept 成功后包装成 `local_socket_uds_impl` [local_acceptor_uds_impl.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_acceptor_uds_impl.cpp:118)
- `local_server` 接住它 [local_server.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_server.cpp:111)
- `local_endpoint` 最终往上交给 routing [local_endpoint.cpp](/home/jzm/workspace/vsomeip/implementation/endpoints/src/local_endpoint.cpp:410)

**一句话定型**
- 中央 `routing host` 负责“知道该找谁”
- 两边的 `routing proxy` 负责“替 app 说话/收话”
- `acceptor` 负责“把连接接进来”
- `socket` 负责“这条连接后续怎么传数据”

所以最准确的脑图是：

```text
app
-> routing proxy
-> socket
-> acceptor / local_server
-> local_endpoint
-> routing proxy
-> app

中央 routing host 不一定一直在数据中间，
但一定在控制和路由信息中间。
```

## 远端ecu通信

变化很大。最核心的一句是：

**跨 ECU 时，没有“应用直连应用”的本地 IPC 了，数据一定要经过两边 ECU 的 routing 和网络 endpoint。**

**本地和远端的对比**
本地服务时，常见路径是：

```text
client app
-> client 进程里的 routing proxy
-> 直连 target app 的 local endpoint/socket
-> target 进程里的 routing proxy
-> target app
```

远端服务时，路径会变成：

```text
client app
-> client 进程里的 routing proxy
-> 本机 routing host
-> TCP/UDP 网络 endpoint
-> 网络
-> 对端 ECU 的 TCP/UDP endpoint
-> 对端 routing host
-> 对端本地 local endpoint
-> target app
```

所以区别不是“多了一点网络”，而是：

- 本地：数据面可以绕开中央 routing host，走 app-to-app 本地直连
- 远端：数据面回到“routing host + 网络 + 对端 routing host”这条主路径

**具体变化点**
- 本地直连没了。不能跨 ECU 用 `local_server / UDS / local_socket` 这种本机机制。
- 两边 routing host 重新回到数据主链路中。
- 发送前如果还不知道远端服务在哪，本机 routing 会先触发 SD `FindService` [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:414)。
- 真正发请求时，如果没找到本地目标，就走远端分支：`find_or_create_remote_client(...)` 再 `send(...)` [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:673)。
- 对端 ECU 收到网络报文后，不是直接交给 app，而是先进入对端 routing 的网络接收入口 `on_message(...)` [routing_manager_impl.cpp](/home/jzm/workspace/vsomeip/implementation/routing/src/routing_manager_impl.cpp:1059)，再由它转给本地 app。

**你可以这样记**
- **同一 ECU**：routing 负责“告诉你该找谁”，之后常常本地直连。
- **不同 ECU**：routing 不只是“告诉你找谁”，还要真正负责“把消息发到对端 ECU，再在对端分发给正确 app”。

**再补一个你现在特别关心的点：`acceptor` 还在不在**
要看协议：

- **远端 TCP**
  有服务端监听/接入的概念，等价于网络侧也有“accept 新连接”这层。
- **远端 UDP**
  没有 `accept`，只有 `bind + receive`。

所以“远端一定还有 acceptor”这句话不完全对，得分 TCP/UDP。  
但不管 TCP 还是 UDP，**远端 app 本身都不是直接面对网络 socket 的，先面对网络的是对端 routing host 的 endpoint。**

**一句话总结**
跨 ECU 后，链路从“本地 app 间直连”变成了“本机 routing -> 网络 -> 对端 routing -> 对端 app”。  
这也是 why `SOME/IP-SD`、远端 endpoint、网络收发检查这些逻辑，都会重新变得重要。
