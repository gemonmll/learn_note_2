在 **SOME/IP Field** 里，`set / get / notify` 可以理解成“带缓存值的属性访问模型”。
它和普通 method/event 的区别是：**Field 同时具备读、写、订阅通知三种语义**。

你问的是 **server 端和 client 端如何收发**，可以直接按下面理解。

---

## 1. Field 由哪几部分组成

一个 Field 通常包含 3 个操作语义：

* **getter**：客户端读当前值
* **setter**：客户端写新值
* **notifier**：服务端在值变化时主动通知订阅者

在协议映射上通常表现为：

* `get` → 一个 **request/response**
* `set` → 一个 **request/response**
* `notify` → 一个 **event**，通常走发布/订阅

所以 Field 本质上像：

> “一个可读可写、还能订阅变化的变量”

---

## 2. get 是怎么收发的

### client 端

client 想读取 field 当前值时，会发一个 **Get 请求** 给 server。

流程：

1. client 发送 `GET(field)` request
2. server 收到后读取当前 field 值
3. server 返回 response，payload 里带当前值
4. client 收到 response，完成读取

### server 端

server 需要做的事：

* 注册这个 field 的 getter 处理函数
* 收到 get request 后，把当前缓存值或实时值返回

### 通信方向

* **client → server**：get request
* **server → client**：get response

---

## 3. set 是怎么收发的

### client 端

client 想修改 field 值时，会发一个 **Set 请求**。

流程：

1. client 发送 `SET(field=newValue)` request
2. server 收到后校验这个值是否合法
3. server 更新内部 field 值
4. server 返回 response，表示成功/失败，有时也会返回更新后的值
5. 如果值真的变化了，server 后续还可能发 notify 给订阅者

### server 端

server 需要做的事：

* 注册 setter 处理函数
* 收到 set request 后：

  * 校验权限、范围、状态
  * 更新值
  * 返回结果

很多实现里 server 会在 set 成功后：

* 更新自身缓存
* 触发 field 的 notify/event

### 通信方向

* **client → server**：set request
* **server → client**：set response
* **server → 所有订阅者**：notify（如果值变化）

---

## 4. notify 是怎么收发的

`notify` 不是 client 主动拉，而是 server 主动推。

但前提通常是：**client 先订阅这个 field 的通知**。

### client 端

流程一般是：

1. client 通过 service discovery 找到服务
2. client 对 field 的 notifier 对应 event group 发起订阅
3. 订阅成功后，等待 server 推送 notify
4. 收到 notify 后更新本地缓存值/触发回调

### server 端

流程一般是：

1. server 提供该 field 的 notifier/event
2. 接收 client 的订阅
3. 当 field 值变化时，向已订阅的 client 发送 notify
4. 有些实现支持首次订阅后立即发当前值，也有些只在变化时发

### 通信方向

* **client → server / SD**：subscribe
* **server → client**：subscribe ack
* **server → client**：notify event

---

## 5. 三者合起来的完整交互

最典型的场景是这样：

### 场景 A：client 先读

1. client 发 `get`
2. server 回当前值 `100`

### 场景 B：client 订阅变化

1. client subscribe notifier
2. server 接受订阅

### 场景 C：client 改值

1. client 发 `set(120)`
2. server 更新成功，回 response
3. server 给所有订阅者发 `notify(120)`

于是：

* 发起 set 的 client 可能收到：

  * set response
  * 之后再收到 notify
* 其他订阅了的 client 只会收到：

  * notify

---

## 6. server 和 client 各自职责

## Server 端

server 通常维护：

* field 当前值
* getter 回调
* setter 回调
* notifier/event
* 订阅者列表

server 负责：

* 响应 get
* 处理 set
* 在值变化时发 notify

---

## Client 端

client 通常维护：

* 发 get/set 的代理接口
* 对 notify 的回调
* 本地缓存值（可选）
* 订阅状态

client 负责：

* 想读时发 get
* 想改时发 set
* 想监听变化时订阅 notify

---

## 7. 很多人容易混淆的点

### ① get/set 是 method 语义

虽然叫 field，但 `get` 和 `set` 在链路上通常还是 **request/response**。

### ② notify 是 event 语义

notify 不需要 client 每次轮询，它是 server 主动推。

### ③ set 成功不一定自动 notify

这取决于服务实现，但常见做法是：

* 值发生变化 → notify
* 值没变化 → 不发 notify

### ④ client 不订阅就收不到 notify

除非实现特殊处理，否则 notify 一般只发给订阅者。

---

## 8. 一个很直观的类比

假设有个 Field：`CabinTemperature`

* `get CabinTemperature`
  客户端问：“当前温度多少？”
* `set CabinTemperature = 24`
  客户端说：“把温度设成 24”
* `notify CabinTemperature = 24`
  服务端广播：“温度现在变成 24 了”

## 9. 简化时序图

```text
Client                     Server
  |                          |
  |---- GET(field) --------->|
  |<--- RESPONSE(value) -----|
  |                          |
  |---- SUBSCRIBE ---------->|
  |<--- SUBSCRIBE ACK -------|
  |                          |
  |---- SET(newValue) ------>|
  |<--- RESPONSE(ok) --------|
  |                          |
  |<--- NOTIFY(newValue) ----|
```

如果有多个 client：

```text
ClientA        ClientB         Server
  |              |               |
  |--subscribe-->|               |
  |              |--subscribe--->|
  |              |               |
  |--set(10)-------------------->|
  |<---------response------------|
  |<---------notify(10)----------|
  |              |<--notify(10)--|
```

## 10. 在代码框架里一般怎么映射

不同中间件实现名字不一样，但概念类似：

### Server

* 提供 service instance
* 注册 field getter
* 注册 field setter
* 提供 notifier/event
* 在值变化时调用 notify/offer/update

### Client

* 查找并连接 service
* 调用 get()
* 调用 set(value)
* subscribe notifier
* 注册 onNotify 回调

## 11. 一句话总结

**SOME/IP Field = getter + setter + notifier 的组合：**

* `get`：client 发请求，server 回当前值
* `set`：client 发请求，server 更新并回结果
* `notify`：server 对已订阅 client 主动推送变化值

## 疑问： set get在报文上就是method方法吗
对，**从报文语义上看，Field 的 `get` 和 `set` 本质上就是 method 调用**。

更准确地说：

* `get`：走 **Request / Response**
* `set`：走 **Request / Response**
* `notify`：走 **Event / Notification**

所以在 SOME/IP 报文层面可以理解成：

* `get/set` 像普通 method
* `notify` 像普通 event

---

## 为什么说它们是 method

SOME/IP 底层主要就是两大类交互：

1. **方法调用**

   * Request
   * Response
   * Error

2. **事件通知**

   * Notification

Field 虽然在接口定义上叫“字段/属性”，但它会被拆成：

* 一个 getter method
* 一个 setter method
* 一个 notifier event

所以到了报文层，不会有一种独立于 method 的“field-get 专用报文类型”。

而是：

* `get` 用 method 的报文模式发
* `set` 用 method 的报文模式发
* `notify` 用 event 的报文模式发

---

## 你的问题如果说得更细一点

### `get`

client 发一个请求让 server 返回当前 field 值。
这和普通远程方法调用非常像：

```text
Client --request(get)--> Server
Client <--response------ Server
```

### `set`

client 发一个请求让 server 修改 field 值。
这也还是普通 method 风格：

```text
Client --request(set=value)--> Server
Client <--response----------- Server
```

所以你可以把它理解成：

* `getX()`
* `setX(value)`

只是接口建模上它被归类为 **Field**，而不是普通 method。

---

## 但要注意一个点

**“语义上像 method” 不等于 “IDL 里它就是普通 method”**。

在接口定义层：

* 它还是 Field
* 工具链会自动为这个 Field 生成 get/set/notifier 相关内容

在传输层/报文层：

* `get/set` 确实按 method 方式传输

所以常见说法是：

> **Field 是一种高层抽象；落到 SOME/IP 线上的时候，getter/setter 会映射成 method call，notifier 会映射成 event。**

---

## 可以这样记

### 接口模型层

* Field = 属性

### 报文传输层

* Get = method request/response
* Set = method request/response
* Notify = event notification

---

## 一个容易混的点

有的人会问：

**“那 get/set 有没有 method ID？”**

通常是有对应标识的。因为既然在线上按 method 交互，就需要能区分：

* 这是哪个 getter
* 这是哪个 setter

而 notify 作为 event，也会有自己的 event 标识。

只是这些 ID 很多时候是由接口定义和代码生成工具帮你管理，不一定手写。

---

## 一句话结论

**是的，SOME/IP Field 的 `get` 和 `set` 在报文层就是按 method 来收发的；只有 `notify` 是按 event 来发的。**

你要的话，我下一条可以继续给你讲清楚：

**Field 的 method id / event id 在协议里通常怎么分配，和普通 method/event 到底有什么区别。**
