好，我们直接按 **Wireshark 抓包视角** 来看。
目标不是只知道“这是什么包”，而是要学会：

* 先看哪一层
* 每层哪些字段最关键
* 怎么从 ARP 一路跟到 SOME/IP-SD，再到真正的 SOME/IP 业务

我按 4 类报文带你看：

1. **ARP Request**
2. **ARP Reply**
3. **SOME/IP-SD Offer / Subscribe**
4. **SOME/IP Notification / Request-Response**

---

# 1. Wireshark 里看包的总方法

你抓到一个包后，通常会看到三大块：

```text id="b0uvso"
Frame
Ethernet II
Internet Protocol Version 4
User Datagram Protocol / Transmission Control Protocol
Application Layer (ARP / SOME/IP-SD / SOME/IP)
```

你读包时建议永远按这个顺序：

### 第一步：先看 Ethernet II

判断：

* 是广播、单播还是多播
* 源 MAC / 目的 MAC 是谁
* EtherType 是什么

### 第二步：再看 IPv4

判断：

* 源 IP / 目的 IP
* 是不是同网段
* TTL、协议号之类先不急，先看 IP 对不对

### 第三步：再看 UDP / TCP

判断：

* 源端口 / 目的端口
* 是 SD 还是普通业务

### 第四步：最后看应用层

判断：

* 是 ARP
* 是 SOME/IP-SD
* 还是普通 SOME/IP

这个顺序对车载控制器抓包特别重要，因为你会经常碰到“应用层看似有问题，实际上是前面 ARP 或交换机学习没完成”。

---

# 2. 第一类：ARP Request 在 Wireshark 里怎么看

这是控制器刚上电、还不知道对方 MAC 时最常见的第一跳。ARP 的协议格式由 RFC 826 定义。

---

## 2.1 Display Filter

你在 Wireshark 顶部可以直接输：

```text id="8q3emp"
arp
```

这样先只看 ARP 包。

---

## 2.2 展开 Ethernet II 看什么

你点开 **Ethernet II**，先看这三个字段：

```text id="jlwmn8"
Destination: ff:ff:ff:ff:ff:ff
Source:      02:11:22:33:44:55
Type:        ARP (0x0806)
```

### 这里怎么理解

* `Destination = ff:ff:ff:ff:ff:ff`
  说明这是**广播帧**
* `Source = 02:11:22:33:44:55`
  说明是这个 ECU 发出来的
* `Type = 0x0806`
  说明上层不是 IPv4，而是 **ARP**

EtherType `0x0806` 对应 ARP，`0x0800` 对应 IPv4。

---

## 2.3 展开 ARP 层看什么

你会看到类似：

```text id="8tmut9"
Hardware type: Ethernet (1)
Protocol type: IPv4 (0x0800)
Hardware size: 6
Protocol size: 4
Opcode: request (1)

Sender MAC address: 02:11:22:33:44:55
Sender IP address: 192.168.10.10
Target MAC address: 00:00:00:00:00:00
Target IP address: 192.168.10.20
```

### 你要重点盯住这 4 个

* `Sender MAC`
* `Sender IP`
* `Target MAC`
* `Target IP`

### 这条包的人话

> `192.168.10.10` 在问：
> “谁是 `192.168.10.20`？请把你的 MAC 告诉我。”

ARP request 里目标 MAC 未知，因此常显示为全 0，这正符合 RFC 826。

---

# 3. 第二类：ARP Reply 在 Wireshark 里怎么看

ARP Reply 就是对方回答：

> “我是这个 IP，我的 MAC 是这个。”

RFC 5227 也明确规定了主机在目标 IP 命中自己时应如何回复 ARP。

---

## 3.1 继续用 display filter

还是：

```text id="3di8hk"
arp
```

然后找 `Opcode: reply`

---

## 3.2 Ethernet II 层看什么

这次通常会变成：

```text id="jzd4bw"
Destination: 02:11:22:33:44:55
Source:      02:aa:bb:cc:dd:ee
Type:        ARP (0x0806)
```

### 这里的关键变化

* `Destination` 不再是广播，而是**单播给请求方**
* `Source` 是被查询 IP 对应的那个 ECU 的 MAC

---

## 3.3 ARP 层看什么

你会看到类似：

```text id="pmtn7x"
Opcode: reply (2)

Sender MAC address: 02:aa:bb:cc:dd:ee
Sender IP address: 192.168.10.20
Target MAC address: 02:11:22:33:44:55
Target IP address: 192.168.10.10
```

### 你要学会一句话翻译

> `192.168.10.20` 回答：
> “我就是 `192.168.10.20`，我的 MAC 是 `02:aa:bb:cc:dd:ee`。”

---

## 3.4 到这里你在脑子里要补出什么

ARP 完成后，发送方本地 ARP 表就会有一条类似：

```text id="5hq3aq"
192.168.10.20  ->  02:aa:bb:cc:dd:ee
```

此后再发给 `192.168.10.20` 的 IPv4 包，二层目的 MAC 就可以直接写成 `02:aa:bb:cc:dd:ee`，不再需要广播问路。ARP 的作用就是做协议地址到以太网地址的映射。

---

# 4. 第三类：IPv4 / UDP 包怎么看

ARP 之后，你会开始看到大量：

```text id="9jlwmw"
Ethernet II
Internet Protocol Version 4
User Datagram Protocol
...
```

这时先别急着看 SOME/IP，先看外壳是不是合理。

---

## 4.1 Ethernet II 层先看

比如：

```text id="p3jlv8"
Destination: 02:aa:bb:cc:dd:ee
Source:      02:11:22:33:44:55
Type:        IPv4 (0x0800)
```

这说明：

* 现在已经不是 ARP 了
* 是正常的 IPv4 业务帧
* 二层已经能单播

RFC 894 定义了 IPv4 在以太网上的标准封装。

---

## 4.2 IPv4 层看什么

展开 **Internet Protocol Version 4**，先看：

```text id="3akjzh"
Source Address:      192.168.10.10
Destination Address: 192.168.10.20
Protocol: UDP (17)
```

### 你主要要确认

* 源 IP 对不对
* 目的 IP 对不对
* 协议是 UDP 还是 TCP

---

## 4.3 UDP 层看什么

展开 **User Datagram Protocol**：

```text id="fnnmpe"
Source Port:      40000
Destination Port: 30490
```

### 这时你要开始联想

* 如果目的端口是 `30490`
* 且上层内容像 SD

那它大概率是 **SOME/IP-SD**

someipy 文档里常用的 SD 端口就是 `30490`。

---

# 5. 第四类：SOME/IP-SD 在 Wireshark 里怎么看

这是车载以太网上特别关键的一类包。AUTOSAR 定义里，SOME/IP-SD 负责服务发现和订阅，而且只跑 UDP。

---

## 5.1 Display Filter

你可以直接输：

```text id="t6mgoq"
someipsd
```

Wireshark 有专门的 SOME/IP-SD 解析字段。

---

## 5.2 先看 Ethernet / IPv4 / UDP

### 如果是 OfferService

常见会是：

* `Dst IP = 224.224.224.245`
* `Dst UDP Port = 30490`

也就是发往 SD 多播组和 SD 端口。someipy 文档中就是这么配置和说明的。

### 如果是 SubscribeEventgroup

常见会是：

* 单播到服务端 IP
* 目的 UDP 端口仍然是 SD 端口

---

## 5.3 SOME/IP Header 看什么

展开 SOME/IP-SD 上层时，先找：

```text id="9slv90"
Message ID
Protocol Version
Interface Version
Message Type
Return Code
```

对于 SD，固定特征通常是：

```text id="u3l76d"
Service ID = 0xFFFF
Method ID  = 0x8100
Message Type = 0x02
```

这就是 AUTOSAR SOME/IP-SD 的固定外层标识。

---

## 5.4 SD Header 看什么

找：

```text id="et6cpt"
Flags
Length of Entries Array
Length of Options Array
```

你现在应该已经知道：

* `Entries Array` 放动作主体
* `Options Array` 放地址端口等附加信息

---

## 5.5 Entries Array 里看什么

Wireshark 里通常会把 entry 解析开，你重点盯：

```text id="fqvjlwm"
Type
Service ID
Instance ID
Major Version
TTL
Minor Version / Eventgroup ID
```

### 如果是 OfferService

你会看到类似：

```text id="j36hbm"
Type: OfferService
Service ID: 0x1234
Instance ID: 0x0001
Major Version: 0x01
TTL: 3
Minor Version: 0x00000001
```

这表示：

> 服务端在宣告：
> 我提供 `0x1234.0x0001`

### 如果是 SubscribeEventgroup

你会看到类似：

```text id="pn7ob7"
Type: SubscribeEventgroup
Service ID: 0x1234
Instance ID: 0x0001
Major Version: 0x01
TTL: 5
Eventgroup ID: 0x0001
```

这表示：

> 客户端在申请订阅这个 eventgroup

SOME/IP-SD 里 Offer、Find、Subscribe、Ack 都通过 Entries Array 表达。

---

## 5.6 Options Array 里看什么

重点盯：

```text id="v8jmjlwm"
Option Type
IPv4 Address
L4 Protocol
Port
```

### 最常见：IPv4 Endpoint Option

你会看到类似：

```text id="akbjlwm"
Type: IPv4 Endpoint
IPv4 Address: 192.168.10.20
L4-Proto: UDP
Port: 30509
```

这表示：

> 这个 entry 对应的 endpoint 在 `192.168.10.20:30509/UDP`

AUTOSAR 明确定义了 IPv4 Endpoint Option 的类型和字段。

---

# 6. 第五类：普通 SOME/IP Notification 在 Wireshark 里怎么看

这已经是**真正业务数据**了，不再是 SD。

---

## 6.1 Display Filter

可以试：

```text id="ozxhs1"
someip
```

或者配合 IP / 端口过滤。

---

## 6.2 先看 Ethernet / IPv4 / UDP

例如：

```text id="knup5d"
Ethernet II
  Dst MAC = 02:11:22:33:44:55
  Src MAC = 02:aa:bb:cc:dd:ee
  Type    = IPv4 (0x0800)

IPv4
  Src IP  = 192.168.10.20
  Dst IP  = 192.168.10.10

UDP
  Src Port = 30509
  Dst Port = 40000
```

这时你要立刻联想：

* 源端口 30509 是服务端业务端口
* 目的端口 40000 是客户端订阅时告诉服务端的接收端口

---

## 6.3 SOME/IP Header 看什么

你要重点看：

```text id="rsjlwm"
Service ID
Method ID / Event ID
Message Type
Interface Version
Return Code
Length
```

### 如果是事件通知

你通常会看到：

```text id="cz76l7"
Service ID      = 0x1234
Event ID        = 0x8001
Message Type    = 0x02   # Notification
Return Code     = 0x00
```

AUTOSAR SOME/IP 协议里 `NOTIFICATION` 的消息类型就是 `0x02`。

### 这条包的人话

> 服务 `0x1234` 正在向订阅者发送事件 `0x8001`

---

## 6.4 Payload 怎么看

Payload 就要看你自己的接口定义了。
比如你约定：

* 用 32-bit 大端整数表示“温度 × 10”

那如果 Wireshark 里 payload 是：

```text id="ljlwm8"
00 00 01 09
```

那就是：

```text id="45e0jg"
0x00000109 = 265
265 / 10 = 26.5°C
```

---

# 7. 第六类：普通 SOME/IP Request / Response 在 Wireshark 里怎么看

如果不是事件，而是方法调用，就看 `REQUEST / RESPONSE`。

AUTOSAR SOME/IP 定义的典型消息类型包括：

* `0x00 REQUEST`
* `0x01 REQUEST_NO_RETURN`
* `0x02 NOTIFICATION`
* `0x80 RESPONSE`
* `0x81 ERROR`。

---

## 7.1 Request 看什么

假设客户端调用某个方法：

```text id="mjlwmf"
Service ID   = 0x1234
Method ID    = 0x0421
Message Type = 0x00
```

你在 Wireshark 里要重点看：

* `Service ID`
* `Method ID`
* `Client ID`
* `Session ID`

因为 Request / Response 是靠 `Request ID = Client ID + Session ID` 来匹配的。AUTOSAR SOME/IP 规范对此有定义。

---

## 7.2 Response 看什么

服务端回包时：

```text id="vqzcdx"
Service ID   = 0x1234
Method ID    = 0x0421
Message Type = 0x80
Return Code  = 0x00
```

然后你要比对：

* Method ID 是否对应
* Client ID / Session ID 是否和 request 一致

如果一致，说明这条 response 就是那次 request 的回复。

---

# 8. 车载控制器抓包时最常见的排查路线

这部分很实战。

---

## 8.1 看不到 SOME/IP，先不要急着怀疑应用

先看：

* 有没有 ARP request / reply
* 是否有 IPv4 单播
* MAC 是否学起来了

如果连 ARP 都没通，那 SOME/IP 不可能正常。

---

## 8.2 有 ARP，但没有 SD

看：

* 是否发到了对的多播地址
* 是否用了 UDP 30490
* Wireshark 里 `someipsd` 能否解析

---

## 8.3 有 Offer，但没有 Subscribe

说明问题可能在客户端：

* 没处理 Offer
* 没发 SubscribeEventgroup
* eventgroup ID 不匹配
* 版本不匹配

---

## 8.4 有 Subscribe，但没有 Ack

说明问题可能在服务端：

* 不接受该 eventgroup
* endpoint option 不合法
* 服务端策略拒绝订阅

---

## 8.5 有 Ack，但没有 Notification

说明：

* 订阅表可能建立了，但业务没触发
* event 没发送
* 发送端口 / 目标端口不对
* payload 路径有问题

---


### 看 UDP



# 10. 给你一个最推荐的学习顺序

你现在最适合这样练：

### 第一轮

只抓：

* `arp`

练到你能一眼分清：

* request
* reply
* 谁在问谁
* 谁在回答谁

### 第二轮

抓：

* `udp.port == 30490`

练到你能分清：

* OfferService
* SubscribeEventgroup
* SubscribeAck

### 第三轮

抓：

* `someip`

练到你能分清：

* Notification
* Request
* Response

# 11. 最后送你一句抓包口诀

**先看 MAC，后看 IP，再看端口，最后看 SOME/IP。**
**没 ARP，就别谈业务。**
**有 SD 没 Notification，问题多半在订阅或业务触发。**
**有 Notification 但数据不对，再去看 payload 编码。**
