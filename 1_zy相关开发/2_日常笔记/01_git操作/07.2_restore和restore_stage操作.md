## 把握一个文件存在三个区

可以。`git restore` 这块最容易绕的地方就在于：

**Git 里同一个文件，可能同时存在 3 份状态：**

1. **HEAD**：上一次提交里的版本
2. **暂存区（index / stage）**：你已经 `git add` 进去、准备下次提交的版本
3. **工作区（working tree）**：你当前正在编辑的文件内容

`restore` 的本质就是：

> **把某个地方的内容，恢复到另一个地方去。**

所以你看到有时是“恢复工作区”，有时是“恢复暂存区”，其实恢复的目标不一样。

---

# 1. 先建立最核心的图

假设你改了一个文件 `a.txt`。

Git 里可以想成这样：

```text
HEAD      : 上次提交版本
暂存区     : git add 之后准备提交的版本
工作区     : 你编辑器里当前版本
```

通常开发时这三者关系会变化。

---

## 初始状态

刚提交完：

```text
HEAD   = v1
暂存区  = v1
工作区  = v1
```

三者一样。

---

## 你修改了文件，但还没 add

你把 `a.txt` 改成 `v2`：

```text
HEAD   = v1
暂存区  = v1
工作区  = v2
```

这时 Git 会显示：

* `Changes not staged for commit`

意思是：

* 工作区变了
* 暂存区还没跟上

---

## 你再执行 git add

```bash
git add a.txt
```

就变成：

```text
HEAD   = v1
暂存区  = v2
工作区  = v2
```

这时 Git 会显示：

* `Changes to be committed`

意思是：

* 暂存区和 HEAD 不一样了
* 工作区和暂存区一致

---

## 你 add 之后又继续改

比如你又把文件改成 `v3`：

```text
HEAD   = v1
暂存区  = v2
工作区  = v3
```

这时一个文件会同时出现在：

* staged changes
* changes not staged

因为：

* 暂存区相对 HEAD：有变化（`v1 -> v2`）
* 工作区相对暂存区：也有变化（`v2 -> v3`）

这个就是很多人最晕的场景。

---

# 2. `git restore` 到底恢复什么

默认：

```bash
git restore a.txt
```

恢复的是：

**工作区**

更准确地说：

> 把“暂存区里的版本”复制到工作区，覆盖工作区当前内容。

也就是：

```text
暂存区  ->  工作区
```

所以它恢复的是你“还没暂存的改动”。

---

# 3. 为什么还要有 `--staged`

因为有时你不是想恢复工作区，而是想：

> “我 add 错了，我想把暂存区恢复一下。”

这时用：

```bash
git restore --staged a.txt
```

它恢复的是：

**暂存区**

更准确地说：

> 把 `HEAD` 里的版本复制到暂存区。

也就是：

```text
HEAD  ->  暂存区
```

所以它取消的是“已经暂存的改动”。

这就是你说的 `restore_stage`，严格命令名其实是：

```bash
git restore --staged
```

不是单独一个 `restore_stage` 命令。

---

# 4. 两者到底各自恢复了什么

我用一个最清楚的表给你看。

## `git restore 文件`

恢复目标：**工作区**

来源：默认从 **暂存区** 拿内容

效果：

* 丢弃工作区未暂存修改
* 不影响暂存区
* 不影响 HEAD

---

## `git restore --staged 文件`

恢复目标：**暂存区**

来源：默认从 **HEAD** 拿内容

效果：

* 取消 `git add`
* 不丢工作区内容
* 不影响 HEAD

---

# 5. 举一个完整例子

假设 `a.txt` 最开始内容是：

```text
hello
```

并且已经提交过了，所以：

```text
HEAD   = hello
暂存区  = hello
工作区  = hello
```

---

## 第一步：你修改文件

把 `a.txt` 改成：

```text
hello world
```

现在：

```text
HEAD   = hello
暂存区  = hello
工作区  = hello world
```

这时你执行：

```bash
git status
```

会看到它在 `Changes not staged for commit`。

---

## 第二步：你执行 `git restore a.txt`

```bash
git restore a.txt
```

Git 会把暂存区的版本覆盖工作区。

所以变回：

```text
HEAD   = hello
暂存区  = hello
工作区  = hello
```

也就是说：

**你刚才在编辑器里改的内容没了。**

所以 `git restore a.txt` 是：

> 放弃工作区修改

---

# 6. 再看 `--staged` 例子

同样一开始：

```text
HEAD   = hello
暂存区  = hello
工作区  = hello
```

你把工作区改成：

```text
hello world
```

然后执行：

```bash
git add a.txt
```

现在：

```text
HEAD   = hello
暂存区  = hello world
工作区  = hello world
```

这时你发现：

> “代码我还想留着，但我现在不想提交它了。”

于是执行：

```bash
git restore --staged a.txt
```

Git 会把 `HEAD` 的版本恢复到暂存区，于是：

```text
HEAD   = hello
暂存区  = hello
工作区  = hello world
```

结果是：

* 暂存区回去了
* 工作区内容还在

所以这个操作相当于：

> **撤销 git add**

也就是老命令：

```bash
git reset HEAD a.txt
```

和它接近。

---

# 7. 为什么要区分这两个操作

因为这两个需求完全不同。

## 需求 A：我写错了，不想要本地修改了

这时你要恢复的是 **工作区**

用：

```bash
git restore a.txt
```

---

## 需求 B：代码我还想保留，但先别提交

这时你要恢复的是 **暂存区**

用：

```bash
git restore --staged a.txt
```

---

# 8. 最容易混淆的场景

这个场景你一定要会。

假设：

1. 原文件是 `v1`
2. 你改成 `v2`，然后 `git add`
3. 接着你又改成 `v3`

这时状态是：

```text
HEAD   = v1
暂存区  = v2
工作区  = v3
```

那如果你执行：

## `git restore a.txt`

工作区会被恢复成暂存区版本：

```text
HEAD   = v1
暂存区  = v2
工作区  = v2
```

结果：

* `v3` 没了
* `v2` 还在
* staged 内容没变

---

## `git restore --staged a.txt`

暂存区会被恢复成 HEAD 版本：

```text
HEAD   = v1
暂存区  = v1
工作区  = v3
```

结果：

* `v2` 这个暂存版本没了
* 工作区 `v3` 还在
* 文件变成“未暂存修改”

---

## 两个一起用

```bash
git restore --staged a.txt
git restore a.txt
```

最终：

```text
HEAD   = v1
暂存区  = v1
工作区  = v1
```

也就是：

**彻底回到上次提交状态。**

---

# 9. 还能指定来源

前面说的是默认来源。

其实 `restore` 还可以加 `--source` 指定“从哪恢复”。

比如：

```bash
git restore --source=HEAD~1 a.txt
```

意思是：

* 不从暂存区拿
* 而是从 `HEAD~1` 那次提交拿内容
* 恢复到工作区

同理：

```bash
git restore --source=HEAD~1 --staged a.txt
```

意思是：

* 从 `HEAD~1` 拿内容
* 恢复到暂存区

所以 `restore` 本质上就是“拷贝某个版本到某个位置”。

---

# 10. 为什么 Git 要设计 `restore` 和 `--staged`

因为早期 Git 里很多“恢复”操作都混在 `checkout`、`reset` 里，新手很难分清。

例如以前常见：

```bash
git checkout -- a.txt
git reset HEAD a.txt
```

一个恢复工作区，一个恢复暂存区，但命令看起来不直观。

后来 Git 就引入了更语义化的命令：

* `git restore`：专门管“恢复内容”
* `git switch`：专门管“切分支”

这样更清楚。

---

# 11. 你可以这样记

## `git restore 文件`

含义：

> 把工作区恢复掉，丢弃没 add 的修改

---

## `git restore --staged 文件`

含义：

> 把暂存区恢复掉，撤销 git add，但保留工作区内容

---

# 12. 一句话区别

**不带 `--staged`，恢复工作区。**
**带 `--staged`，恢复暂存区。**

或者再形象一点：

* `restore`：撤销“你手上正在改的东西”
* `restore --staged`：撤销“你已经放进暂存区的东西”

---

# 13. 最后给你一个最实用的小抄

## 放弃工作区修改

```bash
git restore file
```

## 撤销 git add

```bash
git restore --staged file
```

## 既撤销 git add，又丢弃工作区修改

```bash
git restore --staged file
git restore file
```

## 或者一步到位恢复工作区和暂存区

```bash
git restore --staged --worktree file
```
