不是主要靠“信号量”通知。更准确地说：

```text
跨模块通知：靠 AUTOSAR 回调函数，也就是 Indication / Confirmation API
内部同步：靠状态机、临界区、少量 RAM 标志/TxSemaphore
DCM 内部调度：靠 Dcm_TskSetEvent* 这种事件位
```

**接收侧**

接收侧的“通知 DCM”不是信号量，而是函数回调一路调用上去：

```text
CanTp 收完整请求
-> CanTp_RxInit(... NotifyCode = OK)
-> CanTp_PduRRxIndication(...)
-> PduR_CanTpRxIndication(...)
-> PduR_LoTpRxIndication(...)
-> Dcm_TpRxIndication(...)
```

CanTp 里真正通知 PduR 的位置在 [CanTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/CanTp/CanTp.c:4256)：

```c
CanTp_PduRRxIndication(CanTp_RxCfg_GetPduRRxSduId(lRxHdl), NotifyCode);
```

这个宏最终就是：

```c
PduR_CanTpRxIndication(id, result);
```

PduR 再查路由，调用 DCM 的 `Dcm_TpRxIndication`，位置在 [PduR.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/PduR/PduR.c:923)。

DCM 收到 `Dcm_TpRxIndication(E_OK)` 后，不是靠信号量唤醒，而是把请求放进自己的诊断处理队列/事件里。这里能看到它设置 DCM 内部事件：

[Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:32512)

```c
Dcm_TskSetEventByThread(
    DCM_TSK_ID_DIAG_RX,
    DCM_TSK_EV_DIAG_RX_NEW_REQ,
    ...
);
```

所以接收侧是：

```text
CanTp/PduR/DCM 之间：函数回调通知
DCM 内部：TaskEvents 事件位驱动后续 MainFunction 处理
```

这里的 DCM 事件位设置函数在 [Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:38500)，它本质是设置 `TaskEvents` bit，并把 scheduler 标记为 active。

**接收侧哪里用到了 CanTp TxSemaphore**

接收多帧请求时，ECU 需要给 tester 发 FlowControl，比如 `FC.CTS / FC.WT / FC.OVFL`。这些 FC 帧也是 CanTp 通过 CanIf 发出去的，所以它也需要等 `CanIf_TxConfirmation`。

这个等待 FC 发送确认的归属关系，会用 `CanTp_TxSemaphores`。表在 [CanTp_Lcfg.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Gen/GenData/CanTp_Lcfg.c:198)：

```c
CanTp_TxSemaphores[1]
```

但注意：这不是通知 DCM 的信号量，而是 CanTp 内部记录：

```text
这个 TxPduId 当前被哪个 CanTp Rx/Tx 状态机占着
```

接收侧它主要用于“我发出去的 FlowControl 收到 TxConfirmation 了吗”。

**响应侧**

响应侧的最终确认链路也是回调，不是信号量通知 DCM：

```text
CanIf_TxConfirmation()
-> CanTp_TxConfirmation()
-> CanTp_TxInit(... NotifyCode = OK)
-> CanTp_PduRTxConfirmation()
-> PduR_CanTpTxConfirmation()
-> Dcm_TpTxConfirmation()
```

CanTp 对上层发 TxConfirmation 的位置在 [CanTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/CanTp/CanTp.c:4320)：

```c
CanTp_PduRTxConfirmation(CanTp_TxCfg_GetPduRTxSduId(lTxHdl), NotifyCode);
```

PduR 收到后，在 [PduR.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/PduR/PduR.c:1019) 进入 `PduR_LoTpTxConfirmation`，然后在 [PduR_RmTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/PduR/PduR_RmTp.c:845) 里先释放 PduR 的发送占用状态，再调用 DCM 的 Tx confirmation。

DCM 最终入口在 [Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:36149)：

```c
Dcm_TpTxConfirmation(txPduId, result);
```

然后进入：

[Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:30489)

```c
Dcm_NetUsdtTxConfirmation(...)
```

最后 DCM 内部也会设置任务事件，例如最终响应发送结束：

[Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:32583)

```c
Dcm_TskSetEventByThread(
    DCM_TSK_ID_DIAG_WORK,
    DCM_TSK_EV_DIAG_WORK_TX_END_FINAL,
    ...
);
```

**响应侧 TxSemaphore 真正干什么**

响应侧 CanTp 每发一个 SF/FF/CF 到 CanIf 前，会先检查这个 Tx confirmation PDU 是否空闲。位置在 [CanTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/CanTp/CanTp.c:4126)：

```c
if (CanTp_Data_TxSemaphore(TxConfPduId).Handle == CANTP_INVALID_HDL)
{
    CanIf_Transmit(...);

    CanTp_Data_TxSemaphore(TxConfPduId).Direction = CANTP_DIR_TX;
    CanTp_Data_TxSemaphore(TxConfPduId).Handle    = SduHdl;
}
```

当 CanIf 回调 `CanTp_TxConfirmation(TxPduId)` 时，CanTp 用 `TxPduId` 作为索引读这个 semaphore，找到之前是谁发的，然后释放：

[CanTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/CanTp/CanTp.c:6021)

```c
lSduHdl = CanTp_Data_TxSemaphore(TxPduId).Handle;
lSduDir = CanTp_Data_TxSemaphore(TxPduId).Direction;

CanTp_Data_TxSemaphore(TxPduId).Handle = CANTP_INVALID_HDL;
CanTp_Data_TxSemaphore(TxPduId).Direction = CANTP_DIR_NONE;
```

所以它的作用是：

```text
CanIf 只告诉 CanTp：TxPduId 发完了
CanTp 通过 TxSemaphore 反查：这是哪个 CanTp Tx/Rx 状态机发的
```

一句话总结：**通知机制靠回调；CanTp 的 TxSemaphore 只是内部“占用 + 反查 TxConfirmation 归属”的标记；DCM 内部继续处理靠 `Dcm_TskSetEvent*` 事件位，不是 OS 信号量。**
