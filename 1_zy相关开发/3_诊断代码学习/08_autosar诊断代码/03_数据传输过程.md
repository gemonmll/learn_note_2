这些模块数据是怎么传的，比如pdur收到数据后如何传给dcm，如何通知dcm取数据的

---

这里要抓住一个关键词：**诊断走 TP，所以数据传递不是“一次把完整请求从 PduR 塞给 DCM”，而是 CanTp 分段接收，PduR 转发 DCM 的 TP 回调。**

接收方向可以这样看：

```text
CanTp 收到一段诊断数据
-> PduR_CanTpStartOfReception()
-> Dcm_StartOfReception()

CanTp 后续每收到一段 payload
-> PduR_CanTpCopyRxData()
-> Dcm_CopyRxData()

CanTp 判断完整 UDS 请求收完
-> PduR_CanTpRxIndication(E_OK)
-> Dcm_TpRxIndication(E_OK)
```

**1. StartOfReception：先问 DCM 能不能收**

CanTp 收到单帧/首帧后，会知道整个 TP SDU 长度，然后调用 PduR 的 `StartOfReception`。CanTp 的宏在 [CanTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/CanTp/CanTp.c:179)：

```c
CanTp_PduRStartOfReception(...)
-> PduR_CanTpStartOfReception(...)
```

PduR 通过路由表找到 DCM 回调。DCM 回调函数指针配置在 [PduR_Lcfg.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Gen/GenData/PduR_Lcfg.c:151)：

```c
Dcm_CopyRxData
Dcm_CopyTxData
Dcm_StartOfReception
Dcm_TpRxIndication
Dcm_TpTxConfirmation
```

PduR 真正调用 DCM 的地方在 [PduR.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/PduR/PduR.c:605)：

```c
retVal = Dcm_StartOfReception(
    DcmRxPduId,
    info,
    TpSduLength,
    bufferSizePtr
);
```

DCM 在 `Dcm_NetStartOfReception` 里做几件事，见 [Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:30754)：

```text
根据 rxPduId 找 connection
分配/获取 transport object
检查 DCM buffer 是否够大
设置 RxLength、BuffPos、RxData
返回 BUFREQ_OK 或 BUFREQ_E_OVFL / BUFREQ_E_NOT_OK
```

这里的 `bufferSizePtr` 很关键：DCM 通过它告诉 CanTp：“我当前还有多少接收 buffer”。

**2. CopyRxData：CanTp 把数据片段推给 DCM**

这一步不是 DCM 主动“取数据”，而是 CanTp 每收到一段有效 payload 后，调用：

```text
CanTp_PduRCopyRxData
-> PduR_CanTpCopyRxData
-> PduR_LoTpCopyRxData
-> Dcm_CopyRxData
```

PduR 转发位置在 [PduR.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/PduR/PduR.c:741)：

```c
retVal = Dcm_CopyRxData(
    DcmRxPduId,
    info,
    bufferSizePtr
);
```

`info` 是 `PduInfoType`：

```c
info->SduDataPtr   // 当前这一段数据的指针
info->SduLength    // 当前这一段数据长度
```

DCM 的入口在 [Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:35949)，里面转到 `Dcm_NetCopyRxData`。真正拷贝动作在 [Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:16814)：

```c
Dcm_UtiMemCopySafe(
    info->SduDataPtr,
    lBufferInfo->BufferPtr,
    pTranspObj->BuffPos,
    lBufferInfo->Size,
    info->SduLength
);

pTranspObj->BuffPos += info->SduLength;
*bufferSizePtr = pTranspObj->RxLength - pTranspObj->BuffPos;
```

所以接收数据流是：

```text
CanTp 临时 payload buffer
-> PduR 原样转发 PduInfoType
-> DCM 自己的诊断 buffer
```

PduR 在这条链路里通常不保存诊断数据，因为路由类型是 `PDUR_TP_UNBUFFERED_RX_API_FWD`。

**3. TpRxIndication：通知 DCM 完整请求已经收完**

当 CanTp 收完整个 UDS 请求后，会调用：

```text
PduR_CanTpRxIndication(id, E_OK)
-> Dcm_TpRxIndication(rxPduId, E_OK)
```

PduR 调 DCM 的位置在 [PduR.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/PduR/PduR.c:923)。DCM 入口在 [Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:36028)，最终进 [Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:16920)：

```c
if(result != OK)
{
    Dcm_NetTpRxIndicationNotOk(...)
}
else
{
    Dcm_NetTpRxIndicationOk(...)
}
```

到这里 DCM 才认为：**请求完整收到了，可以开始解析 SID、子功能、调用服务处理器。**

**4. 响应方向反过来：CanTp 向 DCM 要数据**

响应不是 DCM 一段段主动塞给 CanTp。DCM 服务处理完成后，先准备好响应 buffer，然后调用 [Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:16002)：

```c

```

PduR 根据路由调用 `CanTp_Transmit`。之后 CanTp 需要发送每一帧时，会调用：

```text
CanTp_PduRCopyTxData
-> PduR_CanTpCopyTxData
-> Dcm_CopyTxData
```

CanTp 请求数据的位置在 [CanTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/CanTp/CanTp.c:4565)。PduR 再转给 DCM，见 [PduR_RmTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/PduR/PduR_RmTp.c:923)。DCM 真正把响应数据拷给 CanTp 的地方在 [Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:16958)：

```c
Dcm_UtiMemCopySafe(
    &pTranspObj->BuffInfo.SduDataPtr[pTranspObj->BuffPos],
    info->SduDataPtr,
    0u,
    info->SduLength,
    info->SduLength
);

pTranspObj->BuffPos += info->SduLength;
*availableDataPtr = totalLength - BuffPos;
```

所以响应方向是：

```text
DCM 响应 buffer
-> Dcm_CopyTxData
-> PduR 原样转发
-> CanTp 组 CAN TP 帧
-> CanIf/Can 发送
```

一句话总结：**接收时 CanTp 通过 `CopyRxData` 把数据推给 DCM；发送时 CanTp 通过 `CopyTxData` 向 DCM 拉数据。PduR 主要按 Lcfg 路由表查到 DCM 的函数指针，然后做 API 转发。**

## 响应侧链路
对，响应侧刚好反过来，可以理解成两段：

```text
第一段：DCM 请求发送响应
Dcm
-> PduR_DcmTransmit()
-> PduR_UpTransmit()
-> CanTp_Transmit()

第二段：CanTp 分段发送时向 DCM 拉取响应数据
CanTp
-> PduR_CanTpCopyTxData()
-> Dcm_CopyTxData()

发送完成后：
CanIf_TxConfirmation()
-> CanTp_TxConfirmation()
-> PduR_CanTpTxConfirmation()
-> Dcm_TpTxConfirmation()
```

**1. DCM 先发起响应发送**

DCM 服务处理完后，响应数据已经在 DCM 的 response buffer 里了。然后 DCM 调这里：

[Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:16002)

Dcm：我要发一个诊断响应，这个响应长度是 N。
PduR：我查路由，发现这个响应要交给 CanTp。
PduR -> CanTp：请启动 TP 发送。

**2. PduR_DcmTransmit 如何路由到 CanTp**

生成 wrapper 在 [PduR_Lcfg.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Gen/GenData/PduR_Lcfg.c:460)：

然后最终会走到 [PduR_RmTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/PduR/PduR_RmTp.c:812)：

所以响应数据流是：

```text
DCM 通过 PduR 请求 CanTp 开始发送；
CanTp 发送过程中通过 PduR 反向向 DCM 拉取数据；
发送完成后 CanTp 再通过 PduR 通知 DCM 发送结果。
```

## 问题4 响应侧需要做资源检查吗
对，响应侧也会检查“有没有资源/位置”，只是**不是用 `StartOfReception` 这种问法**。

接收侧是外部数据要进 DCM，所以要：

```text
CanTp -> PduR -> Dcm_StartOfReception()
```

语义是：

```text
我要给你塞一个长度 N 的请求，你 DCM 有接收 buffer 吗？
```

但响应侧是 DCM 自己已经把响应数据准备好了，所以没有必要问 DCM “你有没有地方收”。响应侧要检查的是另外几种“位置”。

**1. PduR 先检查这条响应路由是否正在被占用**

DCM 调：

```text
PduR_DcmTransmit()
```

PduR 进入 TP 路由发送逻辑，在 [PduR_RmTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/PduR/PduR_RmTp.c:795) 可以看到：

```c
if(!PduR_IsTransmissionActiveOfRmGDestNto1InfoRam(...))
{
    PduR_SetTransmissionActiveOfRmGDestNto1InfoRam(..., TRUE);
    retVal = CanTp_Transmit(...);
}
```

这一步相当于 PduR 问自己：

```text
这条 DCM -> CanTp 的响应路由现在空不空？
```

如果已经 active，就不会再启动新的发送。

**2. CanTp_Transmit 检查 CanTp 自己有没有 Tx channel**

PduR 最终调用：

```text
CanTp_Transmit()
```

CanTp 入口在 [CanTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/CanTp/CanTp.c:5716)。它会检查初始化、PDU ID、长度、半双工状态，然后进入 `CanTp_TxStartConnection`。

关键在 [CanTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/CanTp/CanTp.c:2684)：

```c
pTpTxState = CanTp_TxGetFreeChannelPtr(TxHdl, CanTpTxInfoPtr);

if (pTpTxState != NULL_PTR)
{
    lReturnValue = E_OK;
    ...
}
```

这里就是 CanTp 问自己：

```text
这个 Tx NSdu 对应的发送通道现在有没有空？
```

如果没有空闲 channel，`CanTp_Transmit()` 返回 `E_NOT_OK`，然后一路返回给：

```text
CanTp_Transmit
-> PduR_DcmTransmit
-> Dcm_NetTransmitUsdtResponse
```

**3. CanTp 发送每一帧前，再向 DCM 拉数据**

启动发送成功后，CanTp 还没有拿走所有响应数据。它每次要组 SF/FF/CF 时，通过：

```text
CanTp -> PduR_CanTpCopyTxData -> Dcm_CopyTxData
```

向 DCM 要一段数据。

这个 `CopyTxData` 的返回值也可以失败：

```text
BUFREQ_OK
BUFREQ_E_NOT_OK
BUFREQ_E_BUSY
```

DCM 拷贝响应数据的位置在 [Dcm.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/Dcm/Dcm.c:16958)：

```c
Dcm_UtiMemCopySafe(...);
pTranspObj->BuffPos += info->SduLength;
*availableDataPtr = totalLength - BuffPos;
```

所以这里不是问“DCM 有没有接收位置”，而是问：

```text
DCM 现在能不能给我下一段响应数据？
还有多少响应数据没给完？
```

**4. CanTp 再问 CanIf/CAN 底层能不能发这一帧**

CanTp 组好一帧后，会调用 CanIf：

[CanTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/CanTp/CanTp.c:4093)

```c
lCanIfResult = CanTp_CanIfTransmit(...);
```

`CanTp_CanIfTransmit` 在配置里映射到 `CanIf_Transmit`，见 [CanTp_Cfg.h](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Gen/GenData/CanTp_Cfg.h:162)。

如果底层忙，CanTp 会根据配置排队或稍后重试；在 [CanTp.c](d:/workspace/autosar_learn/Proj_Code/_01_BSW/Src/CanTp/CanTp.c:4810) 后面可以看到：

```c
if (lTxResult == CANTP_TX_OK)
{
    等 TxConfirmation
}
else
{
    if (lTxResult == CANTP_TX_NOT_OK)
        abort;
    else /* CANTP_TX_BUSY */
        保持状态，后续 task 再试
}
```

**5. 如果是多帧响应，还要等对端 FlowControl**

这点也很重要。响应如果超过单帧，CanTp 会先发 First Frame，然后等待 tester 回 Flow Control：

```text
Tester -> ECU: FC CTS / WAIT / OVFL
```

这个是 ISO-TP 层面问**对端接收 buffer/节奏**：

```text
对端能不能继续收？
一次能收几帧？
STmin 是多少？
```

所以多帧响应里，“有没有位置”还包括 tester 通过 FC 告诉 ECU。

总结一下：

```text
响应侧没有 StartOfReception，因为 DCM 不是接收方，而是数据源。

但会检查：
1. PduR 路由是否空闲
2. CanTp Tx channel 是否空闲
3. DCM CopyTxData 能不能提供数据
4. CanIf/CAN mailbox 能不能发送
5. 多帧时 tester 是否通过 FlowControl 允许继续发
```

所以响应侧不是“不问位置”，而是**问的位置不同、问的接口不同**。
