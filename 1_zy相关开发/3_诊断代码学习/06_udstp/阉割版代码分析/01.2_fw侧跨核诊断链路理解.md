FW 上有跨核诊断，而且从源码看，**最明确的一条是：App DoIP 收到请求后，把 FW-owned 服务跨核转给 FW 处理**。

FW 侧链路可以这样看：

```text
App DoIP UDS
  -> App DCM
  -> App Dcm_Mcu_Proxy
  -> DCMS: /sys/cross_domain_req/uds
  -> FW cross_diag_request_cb
  -> FW DCM
  -> FW PduR_UpTransmit_doip_cross_diag
  -> DCMS: /sys/cross_domain_resq/uds
  -> App DCM
  -> DoIP 回复诊断仪
```

**FW 入口**

FW 在 [DiagApp.c](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/oem_feature_fw/diagnosis/DcmApp/DiagApp.c:396) 注册跨核诊断请求：

```c
dcms_mcu_topic_setup_callback(
    DCMS_TOPIC_DIAG_CROS_UDS_REQ,
    cross_diag_request_cb,
    NULL,
    0);
```

对应 DCMS topic 在 [fawhq_e001_10_dcms_config.h](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/fawhq_e001_10_fw/config/app_config/fawhq_e001_10_dcms_config.h:493)：

```text
DCMS_TOPIC_DIAG_CROS_UDS_REQ  -> /sys/cross_domain_req/uds
DCMS_TOPIC_DIAG_CROS_UDS_RESP -> /sys/cross_domain_resq/uds
```

从配置看，FW 是：

```text
/sys/cross_domain_req/uds  的 server
/sys/cross_domain_resq/uds 的 client / response sender
```

**FW 收到跨核 UDS 后做什么**

核心函数是 [Diag_CrossDiag.c](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/oem_feature_fw/diagnosis/DcmApp/Diag_CrossDiag.c:249)：

```c
cross_diag_request_cb(const uint8_t *data, uint32_t len)
```

这里 `data` 就是完整 UDS 报文：

```text
data[0] = SID
data[1...] = UDS payload
```

比如：

```text
19 xx xx
14 xx xx xx
28 xx xx
85 xx
```

FW 先做少量特殊处理：

```text
0x19：去掉 suppress positive response bit
0x28：走 cross_diag_request_0x28()
0x85：走 cross_diag_request_0x85()
0x22 / 0x2E：代码里有预留，但当前是注释掉的
```

其他通用服务，会被重新灌进 FW DCM：

```c
Dcm_StartOfReception(...)
Dcm_CopyRxData(...)
Dcm_TpRxIndication(...)
```

这一步很关键：**跨核来的 UDS 报文不是绕过 DCM，而是被伪装成一条 TP 接收完成的请求，重新进入 FW DCM 正常服务流程。**

**FW 响应怎么回 App**

FW DCM 处理完后，响应会进入 [PduR_UpTransmit_doip_cross_diag](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/oem_feature_fw/diagnosis/DcmApp/Diag_CrossDiag.c:390)：

```c
PduR_UpTransmit_doip_cross_diag(...)
```

这里把 DCM 输出的响应复制到：

```c
doip_cross_diag_resp
```

然后调用：

```c
Dcm_TpTxConfirmation(...)
```

释放 DCM 发送通道。

随后 `cross_diag_request_cb()` 会检查 `doip_cross_diag_resp`，再通过：

```c
dcms_mcu_topic_send(DCMS_TOPIC_DIAG_CROS_UDS_RESP, ...)
```

把完整 UDS 响应发回 App。

响应格式还是标准 UDS：

```text
正响应：SID + 0x40
负响应：7F SID NRC
等待中：7F SID 78
```

`dcms_mcu_service_server_set_ackdata(...)` 只是 DCMS 服务层 ack，不是 UDS 响应。

**doip_cross_diag_flag 的作用**

`doip_cross_diag_flag` 是 FW 跨核诊断的状态锁，在 [Diag_CrossDiag.c](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/oem_feature_fw/diagnosis/DcmApp/Diag_CrossDiag.c:11)。

它主要做两件事：

```text
1. 防止同一时刻多个跨核诊断请求重入 FW DCM。
2. 让 FW DCM 区分“普通 CAN 请求”和“App 跨核请求”。
```

这个区分在 [DiagApp.c](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/oem_feature_fw/diagnosis/DcmApp/DiagApp.c:255) 的 supplier notification 里能看到：

```text
0x14：doCAN 直接请求时拒绝
0x19：doCAN 直接请求时拒绝
0x31 / 0x34-0x38 / 0x28 / 0x85：FW 本地直接处理会拒绝
```

这说明 FW 对某些服务有明确边界：

```text
0x14 / 0x19 这类服务允许 App DoIP 跨核过来处理。
0x31 标定类服务不归 FW 本地 DCM 处理，更像 App-owned。
0x28 / 0x85 在跨核入口里有特殊处理逻辑。
```

**FW 上的两种跨核角色**

FW 实际有两种跨核角色：

```text
角色 1：App -> FW
App DoIP 收到 FW-owned 服务，跨核转 FW。
源码里最清楚，就是 Dcm_Mcu_Proxy.c -> FW cross_diag_request_cb 这条。
```

```text
角色 2：FW -> App
FW CAN UDS 收到 App-owned 服务，比如标定 0x31 / RID 0x510x，
需要转给 App 的 DCM / DiagCalib 处理。
这条在当前源码里底层转发实现不如 App->FW 明显，更多应该封在平台库或生成代码里。
```

结合你之前的理解，标定方向大概是：

```text
CAN 诊断仪
  -> FW CAN UDS
  -> FW 判断是 App-owned 标定服务
  -> 跨核给 App
  -> App DCM / dji_rte / DiagCalib
  -> App 返回 UDS 响应
  -> FW 通过 CAN 回诊断仪
```

但当前能直接从 FW 源码里完整看到的是：

```text
App DoIP -> FW 跨核请求 -> FW DCM -> FW 跨核响应 -> App
```

**一句话总结**

FW 上的跨核诊断核心就是 [Diag_CrossDiag.c](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/oem_feature_fw/diagnosis/DcmApp/Diag_CrossDiag.c:249)：它把 App 跨核来的 UDS 请求重新注入 FW DCM，再把 DCM 响应抓出来通过 DCMS 回给 App。FW 不是只做 CAN 入口，它同时也是 App DoIP 诊断访问 MCU/FW 服务的跨核服务端。
