你这条链本质是：**App 收到 DoIP UDS，但 0x14/0x19/0x28/0x85 这些服务被配置成 proxy，所以 App 不自己处理，而是把完整 UDS 请求跨核交给 FW，FW 处理后再把完整 UDS 响应回给 App。**

**总链路**

```text
DoIP 诊断仪
  -> App DoIP
  -> App PduR
  -> App DCM
  -> App DCM service proxy
  -> Dcm_Mcu_Proxy.c
  -> call_uds_service_async_cross_domain()
  -> DCMS / cross_domain_req
  -> FW cross_diag_request_cb()
  -> FW DCM 或 FW 侧特殊处理
  -> FW 生成 UDS response
  -> DCMS / cross_domain_resq
  -> App get_resp_data_cross_domain()
  -> App DCM 组响应
  -> App PduR
  -> App DoIP
  -> 诊断仪
```

```c
cross_diag_request_cb(const uint8_t *data, uint32_t len)
```

这里的 `data` 就是完整 UDS：

```text
data[0] = SID
data[1...] = payload
```

FW 分两种处理。

**0x14 / 0x19：注入 FW DCM**

`0x14`、`0x19` 不走特殊分支，所以会进入：

```c
Dcm_StartOfReception(PHY_PDUR_Rx_ID, ...)
Dcm_CopyRxData(PHY_PDUR_Rx_ID, ...)
Dcm_TpRxIndication(PHY_PDUR_Rx_ID, E_OK)
```

这等于告诉 FW DCM：

```text
我收到了一条完整的 TP UDS 请求，请按正常 DCM 服务处理。
```

其中 `0x19` 还会先清掉 suppress-positive-response bit：

```c
req[1] = req[1] & 0x7f;
```

FW 的 `DcmDslServiceRequestSupplierNotification_Indication` 里也能看出这个设计：[DiagApp.c](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/oem_feature_fw/diagnosis/DcmApp/DiagApp.c:271)

```text
0x14 / 0x19 如果是普通 DoCAN 直接请求，会拒绝；
但 doip_cross_diag_flag == true，也就是跨核请求时，允许继续。
```

**0x28 / 0x85：FW 侧特殊处理**

`0x28` 和 `0x85` 在 `cross_diag_request_cb()` 里不注入 DCM，而是直接走：

```c
cross_diag_request_0x28(req, len, TRUE);
cross_diag_request_0x85(req, len, TRUE);
```

`0x28` 会控制 FW 侧 COM / CanTSyn，例如 `Com_IpduGroupStop/Start`、`CanTSyn_SetTransmissionMode`。  
`0x85` 会控制 DTC setting，例如 `Diag_SvcEnableDtcSetting` 和 `DemFunc`。

处理后它们直接填 `doip_cross_diag_resp`：

```text
0x28 正响应：68 xx xx
0x85 正响应：C5 xx
负响应：7F SID NRC
pending：7F SID 78
```

**FW 响应链路**

如果是 `0x14/0x19` 这种注入 FW DCM 的服务，FW DCM 处理完后，响应会进入 [PduR_UpTransmit_doip_cross_diag](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/oem_feature_fw/diagnosis/DcmApp/Diag_CrossDiag.c:390)：

```c
PduR_UpTransmit_doip_cross_diag(...)
```

它把 DCM 响应复制到：

```c
doip_cross_diag_resp
```

然后调用：

```c
Dcm_TpTxConfirmation(...)
```

释放 FW DCM 的发送通道。

接着 `cross_diag_request_cb()` 检查 `doip_cross_diag_resp`：

```text
如果还是 7F xx 78：通过 DCMS 回 pending
如果是最终正响应/负响应：通过 DCMS 回最终响应
```

发送函数是：

```c
dcms_mcu_topic_send(DCMS_TOPIC_DIAG_CROS_UDS_RESP, ...)
```

注意：

```c
dcms_mcu_service_server_set_ackdata(...)
```

只是 DCMS 服务层 ack，不是 UDS 响应。真正 UDS 响应是 `DCMS_TOPIC_DIAG_CROS_UDS_RESP` 发回去的那包数据。

**App 收到 FW 响应后**

App 侧 `cross_uds_req_msg_context()` 在 `DCM_PENDING` 阶段调用：

```c
get_resp_data_cross_domain(resp_data, ...)
```

然后分三种：

```text
resp_data[0] == SID + 0x40
  -> 正响应
  -> Dcm_UtiProvideResDataAsUN() 填到 pMsgContext
  -> return E_OK
  -> App DCM 组 DoIP 正响应

resp_data[0] == 0x7F && resp_data[2] == 0x78
  -> FW 还在 pending
  -> return DCM_E_PENDING
  -> 后续继续轮询/重发查询

resp_data[0] == 0x7F
  -> 负响应
  -> *ErrorCode = resp_data[2]
  -> return E_NOT_OK
  -> App DCM 组 7F SID NRC
```

最后 App DCM 的响应走：

```text
App DCM
  -> PduR_DcmTransmit
  -> DoIP_TpTransmit
  -> DoIP 诊断仪
```

**一句话抓核心**

这条链路的关键点是：

```text
App DCM 不直接处理 0x14/0x19/0x28/0x85，
而是通过 Dcm_Mcu_Proxy 把完整 UDS 请求跨核发给 FW。
FW 收到后，0x14/0x19 重新注入 FW DCM，0x28/0x85 在 FW cross diag 里特殊处理。
FW 最终把完整 UDS response 通过 DCMS 回给 App，
App DCM 再把它包装成 DoIP 响应发给诊断仪。
```
