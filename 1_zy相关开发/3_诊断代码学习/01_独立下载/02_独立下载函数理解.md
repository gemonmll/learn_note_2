# 二、基础工具函数

这类函数不直接体现业务，而是给上层流程提供基本能力。

---

## 1. `Diag_GetCarModelValue(void)`

这个函数大概率是**获取当前车型/项目类型**的工具函数。

从你前面的长截图看，后面的 `Integrity_Verify()` 和多个 `HashComputer_xxx()` 都会根据：

* 车型
* 项目
* 分支
* 模式

做不同处理。

所以这个函数很可能用于：

* 从系统属性取车型值
* 或从配置中读取车型枚举
* 给后续流程决定走哪条校验链

### 你可以这样理解

```c id="qiksc3"
int Diag_GetCarModelValue(void)
{
    // 从 property / 配置 / 某字段里读出当前车型
    return car_model;
}
```

### 学习重点

这个函数一般不复杂，但很重要，因为它会决定：

* 调哪个 `HashComputer_xxx`
* 走哪个完整性校验分支

## 2. `Binary_To_Hex_String(const unsigned char *, size_t, char *)`

这是**二进制转十六进制字符串**函数。

你前面长截图里已经能看出它大概就是：

* 把输入字节数组
* 转成 `"A1B2C3..."` 这种 hex 字符串

### 用途

通常用于：

* 打日志
* 打印 hash
* 做 hash 比对前的字符串化
* 调试校验值

### 伪代码

```c id="t5tbb8"
void Binary_To_Hex_String(const unsigned char *src, size_t len, char *dst)
{
    for (i = 0; i < len; i++) {
        dst[2*i]   = high_nibble_to_hex(src[i]);
        dst[2*i+1] = low_nibble_to_hex(src[i]);
    }
    dst[2*len] = '\0';
}
```

这是**读文件工具函数**。

* `const char *`：文件路径
* `const uint32_t`：期望读取长度或者 buffer 上限
* `uint8_t *`：输出 buffer

### 用途

* 读取下载包

### 骨架

```c id="jpsuzn"
int read_data_from_file(path, max_len, buf, out_len)

    return 0;
}
```

# 三、完整性与签名校验函数

这组函数是这份代码最核心的“安全检查链”。

文档第 3.2.2 节明确说，下载完成后需要做：

* 解密
* 大包 HASH 校验
* 解压
* 子文件 HASH 校验
* 验签。

---

## 5. `Integrity_Verify(void)`

这是**完整性校验函数**。

结合你前面那张长截图，它应该是一个大函数，里面有很多分支，按车型/项目/配置选择校验规则。

### 它做什么

大概率包括：

* 读取系统属性、车型、项目
* 从包或 xml 中提取字段
* 比较：

  * 车型是否匹配
  * 项目是否匹配
  * 区域/方向/模式是否匹配
* 决定当前包是否“适配当前车”

### 它不是单纯 hash

它更像：

> **业务完整性校验**
> 不仅看文件没坏，还看“这个包该不该装在这台车上”。

### 伪代码

```c id="rhe72r"
int Integrity_Verify(void)
{
    获取车型/项目/环境;
    解析当前包中的目标信息;
    if (包信息和当前环境不匹配)
        return FAIL;
    return OK;
}
```

---

## 6. `Signature_Verify(void)`

这是**签名校验函数**。

文档第 3.2.2 节明确提到，解压缩成功后要继续调用接口做子文件的 HASH 对比校验与验签。

### 它做什么

* 对解压后的包内容做签名校验
* 可能调用 cyber / 安全库
* 确认包未被篡改

### 与 `Integrity_Verify` 的区别

* `Integrity_Verify`：更偏业务合法性/完整性匹配
* `Signature_Verify`：更偏密码学签名验证

### 伪代码

```c id="1h7e49"
int Signature_Verify(void)
{
    ret = cyber_signature_verify(unzip_path or manifest);
    return ret;
}
```

---

## 7. `enc_to_dec(void)`

这是**加密包解密函数**。

文档里下载后处理的第一步就是从加密包变成解密包。

### 它做什么

* 找到加密下载包
* 使用密钥/算法进行解密
* 输出到 `deczip` 一类路径

### 伪代码

```c id="qfpa9g"
int enc_to_dec(void)
{
    decrypt(enc_path, deczip_path, key, method);
    return OK or FAIL;
}
```

---

## 8. `enc_to_signature_verify(void)`

这是**后处理总控函数**。

文档第 3.2.2.1 节直接说了，它主要是：

* 解密大包
* 算大包 HASH 并做 HASH 校验
* 解压到 unzip 路径
* 对解压后的文件做验签与子文件 hash 校验。

### 它相当于把多个步骤串起来

它大概率会调用：

* `enc_to_dec()`
* `Integrity_Verify()`
* `dec_to_file()`
* `Signature_Verify()`

### 伪代码

```c id="j22x62"
int enc_to_signature_verify(void)
{
    if (enc_to_dec() != OK) return FAIL;
    if (Integrity_Verify() != OK) return FAIL;
    if (dec_to_file() != OK) return FAIL;
    if (Signature_Verify() != OK) return FAIL;
    return OK;
}
```

这是**把上次的大包移到当前工作 deczip 路径**的函数。

## 11. `unzip_deczip_to_file(void)`

这是**解压 deczip 到 unzip 目录**的函数。

### 用途

* 把解密后的大包解压
* 后续 xml、签名文件、子文件都从这里取

### 伪代码

```c id="zfa9v6"
int unzip_deczip_to_file(void)
{
    unzip(deczip_path, unzip_path);

```

---

## 12. `dec_to_file(void)`

这是**当前 deczip 包的后处理函数**。

文档第 3.2.2.3 节说得很清楚，它会：

* 先进行 `deczip` 路径下文件的 hash 校验
* 然后将 `deczip` 路径下的大包解压缩到 `unzip`
* 解压后进行验签与子文件 hash 校验
* 将 `unzip` 下的 xml 文件加到 `xml_path`。

### 它可以理解成

> **正式处理当前下载好的包**

### 伪代码

```c id="nbpvg5"
int dec_to_file(void)
{
    verify_deczip_hash();
    unzip_deczip_to_file();
    Signature_Verify();
    verify_subfile_hash();
    collect_xml_to_xml_path();
}
```

---

## 13. `last_to_file(void)`

这是**历史包复用后的后处理函数**。

文档第 3.2.2.2 节说：

* 先将 `last_time_zip_path` 大包挪到 `deczip`
* 再解压到 `unzip`
* 再将 `unzip` 下 xml 加到 `xml_path`
* 最后将 `deczip` move 到 `FAW_LAST_TIME_ZIP_PATH`。

### 它和 `dec_to_file` 的区别

* `last_to_file`：偏“旧包复用”
* `dec_to_file`：偏“当前包正式处理”

### 伪代码

```c id="wv9r8r"
int last_to_file(void)
{
    mv_lastzip_to_deczip();
    unzip_deczip_to_file();
    collect_xml_to_xml_path();
    move_deczip_back_to_last_path();
}
```

# 五、Hash 复用判断函数

这组函数是“断点续传 / 历史包复用”的关键。

文档第 3.2.2.4 节说，`HashComputer_xxx` 会先算 `FAW_LAST_TIME_ZIP_PATH` 下大包 hash，如果与待下载大包 hash 一致，则走 `last_to_file()`；否则重新下载。

---

## 14. `HashComputer_E009(void)`

## 15. `HashComputer_P567(void)`

## 16. `HashComputer_P301(void)`

## 17. `HashComputer_E001(void)`

这四个函数本质一样，只是**面向不同项目/车型平台**。

### 它们做什么

* 计算历史包 hash
* 与目标包 hash 比较
* 决定是否复用旧包
* 命中则调用 `last_to_file()`
* 否则走重新下载 / 当前包处理链

### 为什么要分成多个函数

因为不同项目可能有：

* 路径不同
* hash 来源不同
* 车型状态码映射不同
* 成功后设置的状态不同

### 统一伪代码

```c id="k7v5cv"
int HashComputer_xxx(void)
{
    old_hash = calc_hash(last_zip_path);
    if (old_hash == target_hash) {
        return last_to_file();
    }
    return NEED_REDOWNLOAD;
}
```

### 学习重点

这几个函数的共性比差异更重要。
你可以把它们先统一理解成：

> **“旧包能不能直接拿来用”的判断器**

# 六、协议回调函数

这些函数名里的 `cb` 就是 callback。
它们负责接收 SomeIP / dcos / Gateway 的消息并做处理。

文档第 3.2.3 节列出了独立下载相关协议：
包括 Gateway 下载接口、取消接口、查询状态接口，以及 SomeIP 的下载命令、状态请求、脚本请求、脚本通知 ack 等。

---

## 18. `DiagProxy_SelfDownloadCmd_cb(const PS_SelfDLCmd_strt_P567 *)`

这是**独立下载命令入口回调**。

### 处理什么

收到上层发来的：

* 开始下载
* 或取消下载

请求字段里通常有：

* ECU type
* 下载命令
* 文件名
* SHA256
* URL
* 大小
* 密钥
* CampaignID 等。

### 它做什么

* 解析请求结构
* 保存到全局上下文
* 根据命令分发到“开始下载”或“取消下载”

---

## 19. `vehicle_ota_RRSelfDownloadCmd_cb(const uint8_t *, uint32_t)`

这是**原始下载命令消息的协议层回调**。

### 它做什么

* 校验 payload 长度
* 把字节流转成请求结构
* 进入内部命令处理逻辑

### 你可以把它看成

> **协议层解包入口**

---

## 20. `vehicle_ota_Post_Download_cb(const uint8_t *, uint32_t)`

这是**发起 Gateway 下载**的回调/处理函数。

### 它做什么

* 组织 Gateway 下载请求
* 调 `/sys/gateway/post_file_download/v1`
* 获取/保存 `download_id`
* 状态改为下载中

### 它是

> **真正把下载任务提交给 Gateway 的地方**

---

## 21. `vehicle_ota_Cancel_Download_cb(const uint8_t *, uint32_t)`

这是**取消下载回调**。

### 它做什么

* 读取或使用当前 `download_id`
* 调 Gateway 取消下载接口
* 更新状态为取消中/已取消

这是**下载状态查询业务回调**。

### 它做什么

  * 当前阶段

这是**状态请求 / 状态输出的协议层回调**。

这是**内部下载状态同步回调**。

### 它更像

## 29. `ota_deal_file_cb(const void *)`

这是**周期文件处理函数 / 后台调度函数**。

文档第 3.2.1 节明确说它是一个周期运行的异步处理大包文件的函数，主要用于：

### 它做什么

* 周期检查当前任务状态
* 如果下载没完成，就继续等
* 如果下载完成，就做后处理
* 后处理成功则改状态为成功
* 失败则设置错误码

### 它是整套代码的“后台推进器”

# 九、如果从“读源码顺序”来说，建议你这样看

最适合的顺序不是从上到下，而是按流程读：

### 第一步

先看 `vehicle_ota_dcms_init`
搞清有哪些回调被注册。

### 第二步

看 `DiagProxy_SelfDownloadCmd_cb`
搞清“下载请求是怎么进来的”。

# 十、一句话帮你记住这份代码

这份代码的本质是：

> **接收独立下载命令 → 让 Gateway 下载大包 → 周期处理解密/校验/解压/验签 → 对外上报状态 → 提供刷写脚本。**

所以这些函数并不是平铺并列的，它们是在一条完整链路上各自负责一个环节。

你下一步最适合做的是，把其中一个函数完整截出来。我建议先从 `enc_to_signature_verify()` 或 `vehicle_ota_RRSelfDownloadCmd_cb()` 开始，这两个一个代表“入口”，一个代表“后处理总控”。
