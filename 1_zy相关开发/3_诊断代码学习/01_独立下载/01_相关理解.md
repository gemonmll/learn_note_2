## 这份源码是干什么的

这是一个**车端 OTA 下载与校验模块**，核心职责大概是：

1. 接收 OTA 下载命令
2. 创建/准备本地目录
3. 发起下载或断点续传
4. 对下载包做完整性校验
5. 解压 OTA 包
6. 做签名校验和 hash 校验
7. 根据车型更新不同的状态结构体
8. 支持取消下载

从 include 和全局变量也能看出来，它同时跟这几类组件交互：

* **诊断/代理层**：`DiagProxy_If.h`
* **下载通道**：`dcms_mcu_api.h`
* **OTA 业务层**：`someip_ota.h`、`vehicle_ota_If.h`
* **密码学能力**：`cryptomodule_api.h`
* **文件系统/目录操作**：`dirent.h`、`sys/stat.h`
* **base64/hash**：`base64.h`、`crypto_get_file_hash`

### 2. 文件工具函数

这几个是基础工具：

* `Binary_To_Hex_String`
* `read_data_from_file`
* `write_data_to_file`
* `vehicle_ota_mkdir` 

这类函数不复杂，但很关键，因为 OTA 整个流程离不开：

* 读签名文件
* 读历史 hash
* 写本次 hash
* 建目录

#### `Binary_To_Hex_String`

作用很直接：
把二进制 hash 转成十六进制字符串。

比如 32 字节 SHA256：

* 输入：32 字节原始 digest
* 输出：64 字符 hex string

这一步后面是为了再去做 `base64_encode`。

#### `read_data_from_file`

标准文件读取封装：

* 打开文件
* `fread`
* 返回实际长度到 `length`
* 失败打印日志

注意它这里用 `"rb"`，说明是按**二进制**读。这个很合理，因为 `.p7`、证书、hash 数据都可能不是纯文本。

#### `write_data_to_file`

就是把 buffer 写到文件里，后面大概率用于：

* 保存当前下载包 hash
* 保存续传状态
* 保存一些中间标识

#### `vehicle_ota_mkdir`

如果目录不存在就 `mkdir(path, 0755)`。
OTA 流程通常要准备这些目录：

* 加密包目录
* 解密包目录
* 解压目录
* 上一次包目录
* 当前工作目录

虽然这些宏定义没都展开，但从函数调用注释里能看出来。

## 3. 完整性校验：`Integrity_Verify`

这个函数是在做**下载后的 SHA256 校验**。

主逻辑是：

1. 检查 `FAW_DECRYPTION_ZIP_PATH` 是否存在
2. 调 `crypto_get_file_hash(... SHA256 ...)`
3. 把 32 字节 hash 转成 64 字符 hex
4. 再做 `base64_encode`
5. 和命令里携带的 `PS_Sha256_arr` 比较
6. 一致则通过，不一致则失败并删除包

这些关键步骤都能从代码直接看出来。

### 这里有个很值得你注意的点

代码不是直接把二进制 SHA256 拿去比较，而是：

**二进制 hash → 十六进制字符串 → base64**

这说明协议对端传过来的 `PS_Sha256_arr`，大概率不是“原始 32 字节 digest”，而是某种**编码后的字符串形式**。
这一点是你后面读协议时要重点确认的。

### 这个函数的定位

它更像：

* **校验整个下载压缩包是不是目标包**
* 不是校验解压后内部 tar/xml 的签名完整性

内部文件的进一步验证，是下一个函数做的。

## 4. 签名与内部 hash 校验：`Signature_Verify`

这是整份代码里最核心的学习点。

这个函数实际上做了两层验证：

### 第一层：PKCS#7 签名校验

它先在解压目录里找：

* `.txt`
* `.p7`
* `tar.gz`
* `.xml` 

然后读取：

* `sign_p7_path`
* `sign_txt_path`
* `CA_FILE_PATH`
* `CA_TEST_FILE_PATH`

再调用：

* `securty_cerify_pkcs7_sign(...)`
  先用正式 CA 校验
* 如果失败，再用 test CA 再校验一次 

这个设计说明作者考虑了：

* 生产证书
* 测试证书

也就是说，包签名可能来自正式环境，也可能来自测试环境。

### 第二层：对 `tar.gz` 和 `xml` 再做 SHA512 比较

通过签名后，它又做：

* `crypto_get_file_hash(tar_gz_path, SHA512, ...)`
* `crypto_get_file_hash(xml_path, SHA512, ...)`
* 然后 base64 编码
* 再从 `sign_txt_path` 里读出对应 hash
* 对比 `tar_hash_base64` 和 `xml_hash_base64` 

这个设计很经典：

* **PKCS#7** 保证签名文件可信
* **内部 SHA512** 保证解压出来的关键文件内容没被篡改

也就是说它不是只信签名，而是**签名 + 内容摘要双保险**。

### 这个函数为什么重要

如果你想理解 OTA 安全链路，这个函数就是主角。

你可以把它总结成：

```c
下载包
  -> 解压
  -> 找到 p7/txt/tar.gz/xml
  -> 用 CA 验 p7 签名
  -> 再算 tar.gz/xml 的 SHA512
  -> 和 txt 里的记录比对
  -> 成功才算包可信
```

## 5. 解压与文件搬运

### `mv_lastzip_to_deczip`

本质就是执行：

```c
system(mv_file_command)
```

也就是把“上一轮包”挪到“当前解密包位置”。

这反映出一个业务意图：

**支持复用上一次已下载好的 zip 包**，避免重复下载。

### `unzip_deczip_to_file`

这个函数做两件事：

1. 调系统 `unzip`
2. 最多重试 3 次

失败时会：

* 删除解密 zip
* 清空 `Last_Sha256_arr`
* 重置 `sGatewayPostDownloadFlag`
* 重置 `last_zip_exit_flag` 

这个逻辑说明作者认为：

* 解压失败通常意味着包损坏或流程中断
* 这种情况下要把“可继续状态”清掉，避免后续误续传

## 6. 下载主流程入口

这里有两个入口最重要：

### `DiagProxy_SelfDownloadCmd_cb`

这个是 **P567** 路径的入口。
它做了这些明显的事情：

* 检查命令指针为空不为空
* `memcpy` 到全局 `RRSelfDownloadCmd_P567`
* 为下载请求、取消请求、响应结构体做初始化
* 注释里明确说了后面还有：

  * DownloadCmd == 1 / 0 分支
  * 目录创建
  * 历史 hash 比较
  * 续传/重传
  * topic send
  * 状态机更新 

### `vehicle_ota_RRSelfDownloadCmd_cb`

这个是 **E009 / E001 / P301** 的共用入口。
注释已经把流程骨架写出来了：

* `memcpy` 到对应命令结构体
* 打印日志
* `DownloadCmd == 1` 时创建目录、处理历史包/hash、决定 `kTrunc` / `kAppend`
* `DownloadCmd == 0` 时发送 `Cancel_Download`
* 更新 `SelfDLStatus_Out_*` 

这个注释非常值钱，因为虽然 OCR 丢了一些代码，但**设计意图已经保住了**。

# 这套 OTA 代码的核心业务流程

你可以把整套逻辑简化成下面这条线：

```text
上层下发下载命令
    ->
记录命令参数到全局结构体
    ->
准备目录
    ->
检查本地是否已有历史包 / 历史 hash
    ->
决定重新下载(kTrunc)还是续传(kAppend)
    ->
通过 dcms topic 发起网关下载
    ->
下载完成后做 SHA256 包级校验
    ->
解压 zip
    ->
做 PKCS#7 签名校验
    ->
做 tar.gz/xml 的 SHA512 校验
    ->
更新下载状态 / 错误码
```

# 你学习这份源码时最值得盯住的点

## 1. 它不是“单函数逻辑”，而是“状态机逻辑”

不要试图逐行死抠。
更应该先看：

* 哪些是命令输入
* 哪些是状态输出
* 哪些 flag 控制流程推进

比如这些 flag：

* `sHashComputerFlag`
* `sGatewayPostDownloadFlag`
* `last_zip_exit_flag`
* `dec_flag` 

这类变量说明很多动作可能是在**定时任务 / 线程 / 回调**里继续跑，而不是在当前函数里一次做完。

---

## 2. 它的安全校验是分层的

不是只有一个“验 hash”。

它分成：

* **包级 SHA256**
* **签名校验 PKCS#7**
* **内部文件 SHA512**

这说明设计目标不是“文件没传坏就行”，而是“确认来源可信且内容没被篡改”。

---

## 3. 它支持断点续传/复用历史包

虽然 OCR 丢了一部分代码，但从这些痕迹能看出来：

* 比较 `Last_Sha256_arr`
* 比较 `LC_Sha256_arr`
* 比较历史文件 hash
* 决定 `kTrunc` 或 `kAppend`
* `FAW_LAST_TIME_ZIP_PATH` 的存在检查 

这套逻辑明显是在做：

* 同包继续下载
* 不同包重新下载
* 已有完整包则直接走后续校验/解压

---

## 4. 它是“多车型适配”写法

这份代码不是一个车型一份实现，而是同一份逻辑里区分：

* `E009`
* `E001`
* `P301`
* `P567`

不同车型共用大框架，但：

* 命令结构体不同
* 状态结构体不同
* 枚举名不同
* 错误码不同 

所以你后面读这类代码，要先分清：

**哪些是共性流程，哪些只是车型适配层的外壳差异。**
