我前面错在把 App 侧当成“外部诊断入口”来讲了。更准确地说：App 侧是标定诊断业务的执行端，FW 侧是 CAN UDS 入口、跨核转发端、车辆状态/前置条件提供端。

**1. FW 端做什么**
FW 是 MCU 核，主要贴近 CAN、车辆信号和基础诊断：

- CAN UDS 入口在 FW DCM。
- FW 负责把需要 App 处理的 UDS 请求通过 DCOS/DCMS 转给 App 诊断。
- FW 提供车辆状态，例如门、盖、档位、车速、轮速、胎压、悬架等。
- FW 侧 `DiagCalib` 注册前置条件服务，App 请求前置条件时，FW 返回 bitmask。
- FW 最终把 App 返回的 UDS 响应通过 CAN 回给诊断仪。

**2. App 端做什么**
App 是 SoC 核，主要承载真正的诊断业务逻辑和标定编排：

- App 有自己的 DCM/UDS 配置和 callout。
- App 的 `DiagCalib` 处理 `0x5100/0x5101/0x5102/0x5103` Routine。
- App 向 FW 请求前置条件。
- App 向 DMM 请求进入静态/动态标定模式。
- App 向标定 engine 发起真正标定请求。
- App 接收 engine 的状态、结果、delta，并转换成诊断 DID/RID 响应格式。

**3. 一次静态标定的调用链**
以 `0x5100 TopView 静态标定` 为例：

```text
诊断仪发 31 01 51 00
  -> FW CAN DCM 收到
  -> FW 通过 DCOS/DCMS 转给 App DCM
  -> App DCM 调 DiagRid_0x5100...Start
  -> DiagSwc_Callback.c
  -> dji_rte.c 函数指针
  -> eol_topViewCalibEol_requestStart_routine()
  -> App 请求 FW 前置条件
  -> App 请求 DMM 进入静态标定模式
  -> App 发送 /sys/start_calib_rtos 给标定 engine
  -> engine 周期发 /sys/eol_result_rtos
  -> App 解析结果
  -> App DCM 形成 UDS 响应
  -> DCOS/DCMS 回 FW
  -> FW CAN DCM 回诊断仪
```

**4. dji_rte 的作用**
`dji_rte.c` 是 App 侧 DCM callout 到业务函数的“函数指针路由表”。

DCM 生成代码不会直接调用 `eol_topViewCalibEol_requestStart_routine()`，而是先进入 `DiagSwc_Callback.c` 里的标准 callout，例如：

- `DiagRid_0x5100_AVP_camera_static_calibration_...`
- `DiagDid_0x2000_topview_Calibration_results_...`

这些函数再查 `rte_config`，而 `rte_config` 在 `dji_rte.c` 里绑定到真实业务函数：

- `0x5100 Start -> eol_topViewCalibEol_requestStart_routine`
- `0x5101 Start -> eol_stereoFrontBackArround_requestStart_routine`
- `0x5102 Start -> all_afterSales_dynCalib_requestStart`
- `0x2000 Read -> eol_topview_result_readData`

位置在 [dji_rte.c](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/oem_feature_app/autosar_adapter/microsar_config_uds_v4/dji_rte.c:91)。

所以 `dji_rte` 的价值是：把 Vector DCM 生成层和 OEM 业务层解耦。

**5. DiagProxy_Cfg.h 的作用**
`DiagProxy_Cfg.h` 有两类作用，很关键：

第一类是 DCM 配置补丁/宏配置。比如它定义了 RID/DID 操作表：

- `0x5100/0x5101/0x5102/0x5103` 的 Routine 配置：[DiagProxy_Cfg.h](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/fawhq_e001_10_app/DiagProxy/DiagProxy_Cfg.h:67)
- `0x2000/0x2001/0x2003/0x2004/0x2008...` 的 DID 配置：[DiagProxy_Cfg.h](C:/Users/c-ziming.jiang/Desktop/bak_code/code_bak/fawhq_e001_10_app/DiagProxy/DiagProxy_Cfg.h:95)

第二类是标定结果映射表。engine 返回的是按内部 camera type 和数组 offset 组织的结果，FAW 诊断要的是 DID/RID 定义好的字段。这里做映射：

- `RESULT_ERRORCODE_CONFIG`：静态标定错误码从 engine 数组哪里取。
- `RESULT_DELTA_MAP_CONFIG`：delta 的 X/Y/Z/Roll/Yaw/Pitch 从哪里取。
- `DYN_RESULT_ERRORCODE_CONFIG`：动态标定错误码映射。
- `SENSOR_CONFIG_NUM = 12`：本项目按 12 路传感器结果解析。

这些映射最终被 `DiagCalib_Cfg.c` 实例化，然后 `DiagCalib_Static.c/Dynamic.c` 解析结果时使用。

**6. App/FW 诊断模块的关系**
可以简单记成：

```text
FW Diag = 诊断入口、CAN 响应、车辆状态、跨核通道
App Diag = 诊断业务、标定编排、engine 交互、结果转换
```

两者通过 DCOS/DCMS 关联：

- UDS 请求/响应跨核转发。
- App 请求 FW 前置条件。
- FW 把车辆 CAN 信号抽象成前置条件结果。
- App 把 engine 结果转换成诊断格式，再返回 FW。
